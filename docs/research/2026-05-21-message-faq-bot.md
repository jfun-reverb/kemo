# 응모건 메시지 — 자동응답(FAQ 가이드형) 기획 리서치

> **작성일:** 2026-05-21
> **목적:** 인플루언서 문의를 선택지 트리(+선택적 AI 의도 파악)로 자동 안내해 관리자 응대 부담을 줄이고, 번역 의존도를 낮춤
> **상태:** 리서치/기획 초안 단계 (문의 유형 도출 단계)
> **연관:** [2026-05-21-message-translation-research.md](./2026-05-21-message-translation-research.md) (번역 리서치)

---

## 1. 방식 개요

**1단계 — 선택지 트리 (규칙 기반, AI 불필요)**
인플루언서가 문의하면 카테고리 선택지가 뜨고, 선택에 따라 하위 질문 또는 미리 작성한 정답 안내가 나옴. 답변은 한국어·일본어 양쪽으로 미리 작성 → 번역 불필요.

**2단계 — AI 의도 파악 (선택적, 나중)**
자유 텍스트를 AI가 읽고 알맞은 카테고리로 자동 연결. 트리가 자리잡은 뒤 얹는 게 안전.

**필수 — 사람 연결 탈출구**
트리로 안 풀리면 반드시 관리자에게 넘어가는 경로 유지.

---

## 2. 문의 유형 카탈로그 (기능 구조 기반 도출)

> 코드의 차단·검증 로직(`dev/js/auth.js`, `application.js`, `mypage.js`)에서 인플루언서가 막힐 수 있는 지점을 단계별로 도출. **실제 빈도는 LINE 문의·운영 데이터로 보강 필요.**

### A. 로그인·계정 (auth.js)
- 비밀번호를 잊어버렸어요 → 비밀번호 재설정 안내
- 로그인이 안돼요 (이메일/비밀번호 오류)
- (운영) 가입 확인 메일이 안 와요 / 메일 확인을 못 했어요
- 회원 탈퇴하고 싶어요

### B. 캠페인 신청(응모)이 안돼요 — **가장 막힘 多 예상** (application.js)
코드상 신청을 차단하는 조건이 그대로 문의 유형이 됨:
- **이메일 인증이 필요하다고 나와요** → 인증 안내
- **필수 정보를 입력하라고 나와요** (SNS 계정 / 배송지: 우편번호·도도부현·시·전화 / PayPal 이메일) → 마이페이지 해당 항목 안내
- **팔로워 수가 부족하다고 나와요** → 최소 팔로워 정책 + 팔로워 수 최신화 안내
- **이미 신청했다고 나와요** → 중복 신청 안내
- **모집이 마감됐다고 나와요** (리뷰어 정원 초과 / 마감일 경과) → 마감 안내
- **주의사항 동의 체크가 안돼요** → 체크박스 안내

### C. 승인·심사 (mypage.js 응모이력)
- 언제 승인되나요? / 심사 중이 계속 떠요
- 왜 비승인됐나요?

### D. 배송
- 제품이 언제 오나요? / 배송 조회는 어디서?
- 배송지를 잘못 입력했어요

### E. 결과물 제출 (활동관리, application.js)
- **영수증 제출이 안돼요** (주문번호·구매일·구매금액 필수 누락)
- **게시물 URL을 넣었는데 채널 인식이 안돼요** → 수동 채널 선택 안내
- **제출 기한이 지났어요** (submission_end 경과)
- **반려됐는데 어떻게 재제출하나요?** → 재제출 플로우 안내
- 어떤 형식으로 올려야 하나요? (#PR 태그 등)

### F. 정산·리워드
- 리워드는 언제·어떻게 받나요?
- PayPal 정보를 바꾸고 싶어요

### G. 응모 취소 (mypage.js)
- 신청을 취소하고 싶어요 → phase별 취소 안내
- 취소했는데 다시 신청할 수 있나요?

### H. 프로필·설정
- SNS 계정/팔로워 수 수정은 어디서?
- 대표 SNS는 어떻게 정하나요?
- 마케팅 메일을 그만 받고 싶어요

## 2-1. 실제 LINE 대화 분석 (2026-05-21, 7명 표본)

> 실제 공식 LINE 대화 7건(2025-05 ~ 2026-05)을 분석한 **실측 빈도**. 개인정보(실명·SNS 핸들·주소)는 제외하고 유형만 추출.

### 실측 빈도 순위

| 순위 | 문의/메시지 유형 | 표본 내 빈도 | 비고 |
|---|---|---|---|
| 1 | **결과물(투고) URL 제출 + "확인 부탁드립니다"** | 거의 전원 (반복) | 이미 앱의 「결과물 제출」 기능에 있음 — LINE으로 하고 있을 뿐 |
| 2 | **SNS 계정 URL 공유** (가입 직후) | 7명 전원 | 이미 앱 프로필 SNS 등록에 있음 |
| 3 | **승인·심사 결과 언제 나오나요** | 2~3명 | "ずっと審査中、いつ結果わかりますか" — 앱 응모이력 상태로 흡수 가능 |
| 4 | **당선·인증 확인** ("이게 당선 메시지 맞나요?") | 다수 | 팝업 이벤트 특화 (앱 밖 영역) |
| 5 | **배송 문제** (지연·상품 누락·미도착) | 3명 | ⚠️ 사람 개입 필수 — 자동응답으론 접수만 |
| 6 | **응모 취소가 에러로 안 됨** | 1명 | ⚠️ 시스템 버그성 — FAQ로 안 풀림, 실제 수정 필요 |
| 7 | 구매·쿠폰·환불(정산) 문의 | 1명 | "쿠폰 써도 되나요, 환불 언제" |
| 8 | 제출 기한·마감 규칙 문의 | 1~2명 | "마감인데 추가 투고 가능?" "기한 못 맞춰요" |
| 9 | 이벤트 일정 변경·불참 연락 | 1~2명 | 팝업 이벤트 특화 |

### 핵심 통찰 (고문 관점)

1. **문의 대부분이 "이미 앱에 있는 기능"을 LINE으로 하고 있는 것** — 1·2·3위(결과물 제출·SNS 등록·상태 확인)는 전부 앱에 기능이 있음. **번역·FAQ보다 먼저, "왜 인플루언서가 앱 대신 LINE을 쓰는가"를 풀면 문의량 자체가 급감**할 가능성. 응모건 메시지 기능을 인플루언서가 실제로 쓰게 유도하는 게 1순위 효과
2. **진짜 사람이 필요한 건 소수** — 배송 문제(지연·누락), 취소 오류, 일정 조정. 이건 자동응답으로 "접수"만 하고 관리자에게 넘기는 게 맞음
3. **취소 오류는 FAQ가 아니라 버그 수정 대상** — 표본에서 "취소 신청이 에러로 안 된다"는 실제 시스템 문제 발견. 자동응답 이전에 별도 점검 필요
4. **번역 수요는 비정형 대화에 집중** — 정형(제출·등록·상태)을 앱/FAQ로 흡수하면, 번역이 정말 필요한 건 배송·정산·상담 같은 자유 대화뿐 → 번역 사용량·비용 대폭 절감

### 자동응답 트리 우선순위 제안 (실측 기반)

- **A. 즉답형 (FAQ로 100% 흡수 가능)**: 결과물 제출 방법 / SNS 등록 위치 / 심사 결과 확인 위치 / 제출 기한 규칙
- **B. 접수 후 사람 연결**: 배송 문제 / 정산·환불 / 응모 취소 오류 / 일정 변경
- **C. 앱 기능 유도로 문의 자체를 줄이기**: 결과물 제출·상태 확인을 앱 안에서 끝내도록 안내

## 3. 다음 단계 (미결)

1. **실제 빈도 확정** — 위 카탈로그 중 실제로 자주 오는 톱 유형 선별 (LINE 문의·운영 데이터·실무 경험)
2. **답변 원문 작성** — 자주 묻는 항목부터 한국어·일본어 정답 작성
3. **트리 깊이 결정** — 2~3단계 이내 권장 (너무 깊으면 이탈)
4. **관리자 등록 방식** — 기존 기준 데이터(`/admin#lookups`, `lookup_values` name_ko/name_ja 패턴) 재사용 가능성 검토
5. **사람 연결 트리거** — "기타" 선택·트리 막다른 길·N회 미해결 시 관리자 알림
6. **번역과의 통합 그림** — 정형 문의는 FAQ, 예외 자유 대화만 번역 보조 (2단 구조)

## 4. 비고

- **이 프로젝트와 궁합** — 기준 데이터 관리(`lookup_values`)와 알림(`notifications`) 인프라가 이미 있어, FAQ 트리·답변을 관리자가 직접 등록·수정하는 구조로 확장 가능
- **번역 우회 효과** — 정형 문의(다수 추정)를 미리 작성한 양국어 답변으로 처리하면, 번역 사용량·비용·개인정보 우려가 크게 감소
