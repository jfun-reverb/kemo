# 관리자 「오픈 예정 기능」 보드 (D-day) — 기획 사양서

**작성일:** 2026-06-15
**작성 세션:** 기획/설계
**배경:** 시행일이 걸린 기능(연령 정책 등)이 늘면서, 관리자가 "곧 어떤 기능이 며칠 뒤 오픈되는지"를 한눈에 보는 화면 요구. 사용자 명시: "관리자 페이지에서 오픈될 기능·상세설명·D-day를 보는 화면. 단, 개발이 완료된 것만 노출."

## 현재 상태 (작성일 기준)

### 관련 코드·DB·UI 진입점
- **시행일 코드 상수 선례 존재**: `dev/lib/shared.js`의 `POLICY_NOTICE`(라인 524~) — `effectiveDate`(시행일) + `noticeUntil`(시행일+14일 = 노출종료일). 인플루언서 앱 약관 개정 안내를 시행일 기준 자동 노출/종료. 시행일 운영 규칙(`.claude/rules/release-timing.md`)이 권장하는 "UI 노출은 클라 상수" 방식의 실제 사례.
- **약관 통지 로그** `policy_notice_log`(마이그레이션 153 = `policy_notice_runs`/`policy_notice_sent`): 통지 **메일 발송 기록**일 뿐, "예정 기능 목록" 데이터 아님.
- **관리자 공지** `admin_notices`: `category` 4종 중 `release`(출시) 존재. 단 **시행일·D-day 컬럼 없음**. 공지는 시행 **후** 완료형 알림 성격(`status` draft/published).
- 관리자 사이드바 메뉴 구조: `dev/admin/index.html` + 페인 라우팅(`data-pane`), `dev/js/admin-core.js` 페인 분기.

### 이 제안과 충돌 가능성 있는 기존 동작
- **충돌 없음 — 3개 영역 확인**. ① DB 변경 없음(코드 상수만) ② `admin_notices` 출시 공지와 역할이 다름(공지=시행 후 완료형 / 보드=시행 전 예고형 D-day) ③ `POLICY_NOTICE` 패턴을 확장하므로 기존 인플 약관 안내와 데이터 분리.

### 미해결 백로그·관련 작업
- 연령 정책 사양서(`docs/specs/2026-05-27-age-minor-policy.md`)가 이 보드의 첫 등록 대상이 될 가능성.
- 시행일 운영 규칙(`.claude/rules/release-timing.md`) — 이 보드는 그 규칙의 "예고" 측면. 단 **실제 집행(기능 ON/OFF)은 보드가 하지 않음**.

## 의심·경우의 수 (반대론자 모드)

1. **(데이터) 개발이 등록 안 하면 보드가 빈다 — "개발 완료된 것만"의 이면**: 예정 기능 데이터는 시스템이 자동으로 알 수 없어, 개발 세션이 배포 시 코드 상수에 직접 등록해야 한다. 등록이 누락되면 보드가 무용 → **개발 세션 운영 약속(규칙)으로 보강 필요**(아래 §6).
2. **(개념) 이 보드는 예고판이지 기능 스위치가 아니다 — 가장 중요한 구분**: 화면에 D-day가 떠도 실제 기능이 켜지는 건 아니다. 실제 활성(연령 차단 ON 등)은 별도(시행일 `dev→main` 머지 또는 서버 시각 판정, `release-timing.md` §3·§4)가 담당. 보드의 D-day와 실제 시행이 어긋나면 혼란 → **시행일 상수 하나를 단일 소스로 공유**해 동기화.
3. **(시각) D-day 시각 판정**: 이 보드는 **표시용**이라 클라 시각 기반 계산 허용(`release-timing.md` §3 — UI 노출은 클라 상수 OK). 단 자정 경계 혼동 방지 위해 비교에 **`+09:00`(일본/한국 표준시) 명시**, 날짜 단위 비교.
4. **(UX) 빈 상태**: 예정 기능 0건일 때 화면이 텅 비지 않게 안내 문구(「현재 예정된 기능이 없습니다」). 평상시 기본 상태가 빈 목록일 수 있음.
5. **(UX/권한) 노출 대상**: 읽기 전용 정보성이라 관리자 전원(super_admin/campaign_admin/campaign_manager) 노출 무방. 권한 분기 불필요. campaign_manager가 봐도 문제 없음(CUD 버튼 없음).
6. **(중복) admin_notices 출시 공지와의 경계**: 둘 다 두면 관리자가 헷갈릴 수 있음 → 역할 명문화. **보드 = 시행 전 「곧 옵니다」 D-day 예고 / 공지 = 시행 후 「나왔습니다」 완료 알림**. 같은 기능이 보드(예고) → 시행 → 공지(완료)로 흐름.

### 의도 모호점 (확정 완료)
- "개발 완료된 것만 노출" = 개발 세션이 코드에 등록한 것만 뜨므로 **자동 보장**(사용자/시스템이 완료 여부 판단 불필요). 확정.
- "별도 화면" = 전용 관리자 페인(사이드바 메뉴 1개 추가). 대시보드 카드가 아닌 독립 화면.

## 제안 / 설계

### 1. 개념
개발 세션이 시행일 걸린 기능을 배포할 때 **코드 상수에 항목 1줄 등록** → 관리자 전용 화면이 그 상수를 읽어 **D-day 자동 계산** 표시. 개발이 등록한 것만 뜨므로 "개발 완료된 것만 노출"이 구조적으로 보장. **DB 변경 없음.**

### 2. 데이터 모델 (DB 변경 없음 — 코드 상수)
`dev/lib/shared.js`에 `UPCOMING_FEATURES` 배열 상수 신설(기존 `POLICY_NOTICE` 인근):

```js
// 오픈 예정 기능 보드 데이터. 개발 세션이 시행일 걸린 기능 배포 시 1줄 추가.
// effectiveDate 는 그 기능의 실제 시행일 상수와 동일해야 함(단일 소스 공유).
const UPCOMING_FEATURE_RETENTION_DAYS = 14; // 시행 후 며칠까지 "시행 완료"로 노출
const UPCOMING_FEATURES = [
  {
    key: 'age-policy-2026',                       // 식별자(중복 금지)
    title: '年齢ポリシー（満18歳以上）',            // 기능명(※화면은 관리자=한국어, 항목 라벨 언어 정책은 §4 참조)
    desc: '加入・応募時に生年月日・性別を入力。18歳未満は応募不可。', // 상세 설명
    effectiveDate: '2026-07-15',                  // 시행일(KST). 해당 기능 시행일 상수와 동일값
  },
  // ...
];
```

상태 판정 헬퍼(`shared.js`):
```js
// 오늘(KST) 기준 항목 상태: 'upcoming'(D-n) | 'done'(시행 완료, 유지기간 내) | 'expired'(목록 제외)
function upcomingFeatureStatus(item, nowKst) { /* effectiveDate, +RETENTION_DAYS 로 분기 */ }
function upcomingFeatureDday(item, nowKst)   { /* 시행일까지 남은 일수(양수=예정) */ }
```
- 시각 비교는 `new Date(effectiveDate + 'T00:00:00+09:00')` 기준(POLICY_NOTICE 선례와 동일).

### 3. 화면 (관리자 PC, 한국어)
- **새 사이드바 메뉴 1개** + 전용 페인. 위치 권고: 공지사항/대시보드 인근(개발 세션이 사이드바 순서 확정).
- **항목 카드**(시행일 가까운 순 정렬):
  - 기능명(제목) + 상세 설명
  - 시행일(`ja-JP` 또는 `ko` 포맷)
  - **D-day 배지**: 예정 = 주황 「D-15」 / 당일 = 「D-DAY」 / 시행 완료(유지기간 내) = 초록 「시행 완료」
- **빈 상태**: 「현재 예정된 기능이 없습니다」 안내(아이콘 + 회색 문구).
- **권한**: 관리자 전원 노출(읽기 전용, 권한 분기 없음).
- 깜빡임 방지 cloak·페인 라우팅은 기존 관리자 패턴 상속.

### 4. 항목 라벨 언어
- 관리자 페이지 UI는 한국어 규칙이나, 위 데이터 상수 예시는 일본어로 적힘 → **결정 필요한 디테일**: 항목 제목/설명을 (a) 한국어 단일 (관리자 화면이니 한국어) 또는 (b) 한·일 2종. 관리자 전용 보드이므로 **한국어 단일 권장**(개발 세션이 등록 시 한국어로). 사양 확정값: **한국어 단일**.

### 5. 빌드·영향 파일
- `dev/lib/shared.js` — `UPCOMING_FEATURES` 상수 + 상태/D-day 헬퍼
- `dev/admin/index.html` — 사이드바 메뉴 항목 + 페인 컨테이너
- 렌더 JS — `dev/js/admin-dashboard.js` 확장 또는 신규 `admin-roadmap.js`(신규 파일이면 `build.sh` `ADMIN_JS_FILES` 등록 + 순서 주의)
- `dev/css/admin.css` — 카드·D-day 배지 스타일
- **마이그레이션 없음**

### 6. 개발 세션 운영 약속 (규칙 보강 권고)
이 보드가 의미 있으려면, **시행일 걸린 기능을 운영 배포할 때 `UPCOMING_FEATURES`에 항목을 등록**하는 것을 워크플로에 포함해야 한다. `release-timing.md`에 한 줄 추가 권고:
> 시행일 걸린 기능을 운영 배포(또는 시행일 확정) 시 `dev/lib/shared.js`의 `UPCOMING_FEATURES`에 항목 등록. `effectiveDate`는 해당 기능의 실제 시행일 상수와 동일값(단일 소스).

### 7. 시행일 단일 소스 (중요)
보드의 `effectiveDate`와 실제 기능 시행 판정(클라 노출 상수·서버 시각 집행)이 **같은 값**을 봐야 한다. 연령 정책의 경우 그 기능의 시행일 상수를 만들고 보드가 그걸 참조(또는 동일 리터럴). 두 곳에 다른 날짜가 박히면 "보드는 D-3인데 기능은 이미 켜짐" 같은 불일치 발생.

### 초안 대비 변경 사항
- **렌더 파일은 신규 `dev/js/admin-roadmap.js`로 분리**(초안의 "admin-dashboard.js 확장 또는 신규" 중 신규 선택). `build.sh` `ADMIN_JS_FILES`에 `admin-dashboard.js` 뒤로 등록.
- **사이드바 위치는 「공지」 섹션**(공지사항 바로 아래). 보드=예고 / 공지=완료 흐름이 같은 섹션에 묶이도록(초안 §6 역할 경계와 일관). 아이콘 `upcoming`.
- **D-day 배지는 상태(status)가 아니라 D-day 숫자 기준으로 결정**. 초안 §3 표기(D-15 / D-DAY / 시행 완료)를 그대로 따르되, 시행일 당일은 `upcomingFeatureStatus`가 이미 `done`이라 「시행 완료」로 떠 「D-DAY」가 안 나오던 모순을 발견 → 카드 배지는 `upcomingFeatureDday()` 값으로 분기(>0=D-n, 0=D-DAY, <0=시행 완료). `status`는 목록 노출/제외(expired 필터) 판정에만 사용.
- **상태/D-day 헬퍼를 `shared.js`에 4종 + 내부 2종 추가**: `upcomingFeatureStatus`/`upcomingFeatureDday`/`upcomingFeatureDateLabel`/`visibleUpcomingFeatures` (+ `_upcomingEffectiveTs`/`_todayKstTs`).
- **`UPCOMING_FEATURES` 초기값은 빈 배열**(예시는 주석으로). 등록 대상(연령 정책 등)이 아직 시행일 확정 전이라 빈 상태가 첫 노출 — 「현재 예정된 기능이 없습니다」 안내가 뜸.

### 구현 중 기술 결정 사항
- **마이그레이션 없음**(코드 상수만, 초안대로).
- **D-day 날짜 단위 비교**: `_todayKstTs()`가 현재 시각을 KST(+09:00)로 환산해 그날 0시 타임스탬프를 계산, 시행일 0시와 일(day) 단위로 비교(자정 경계 혼동 방지). `POLICY_NOTICE` 선례와 동일하게 `T00:00:00+09:00` 기준.
- **`esc()` 의존**: 카드 텍스트는 코드 상수(개발자 입력)지만 일관성·XSS 방어 차원에서 `esc()` 적용. 빌드 순서상 `ui.js`(esc 정의)가 `admin-roadmap.js`보다 앞이라 안전.
- **항목 라벨 언어 = 한국어 단일**(초안 §4 확정값). 상수 예시 주석도 한국어로 작성.
- **운영 약속(초안 §6)**: 시행일 걸린 기능 배포 시 `UPCOMING_FEATURES`에 등록하는 워크플로는 별도 규칙 보강 대상(고문/기획 몫 — 이번 개발 범위에서 제외). 코드 상수 상단 주석에 사용법 명시.
