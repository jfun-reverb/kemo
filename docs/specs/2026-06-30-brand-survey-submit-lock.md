## 0. 한 줄 요약

공개 홍보를 중단했는데도 옛 URL이 살아 있어 **기존 브랜드가 브랜드 서베이 폼으로 계속 제출**하고 있다. 영업 2단계 폼 용도는 **유지**하되(이전 결정), **공개 무분별 제출만 즉시 차단**한다. 영업이 보낸 링크로만 받는 토큰 발급 화면은 영업이 실제 필요해질 때 2단계로(지금은 차단만).

## 1. 현재 상태 (planning.md 규칙 A — 검증 완료)

### 관련 코드·DB·UI 진입점
- **제출 폼**: `sales/reviewer.html`(라인 1664)·`sales/seeding.html`(라인 1528)이 비로그인으로 `submit_brand_application` 원격 호출 함수를 직접 호출. `sales/index.html`은 두 폼으로 가는 선택(랜딩) 페이지. 도메인 `sales.globalreverb.com`(운영)·`sales-dev`(개발).
- **제출 함수**: `submit_brand_application(p_form_type, p_brand_name, p_contact_name, p_phone, p_email, p_products, p_billing_email, p_business_license_path, p_request_note)` RETURNS text, **SECURITY DEFINER + anon GRANT**(마이그레이션 087). 함수 시작부에 `p_form_type` 검증 등 가드가 이미 있어 **차단 가드 삽입 지점 명확**.
- **이전 결정(메모리 `project_brand_survey_future`, 2026-06-24)**: 폼을 **닫지 않고 영업 2단계 상세 폼으로 유지**. "토큰 잠금(아무나 제출 차단)은 랜딩페이지 신설 때 함께"로 **보류**. → 본 작업은 그 토큰 잠금을 **앞당기되 1단계(차단)만** 먼저.
- **별개 보호 대상**: `sales/orient.html`(오리엔시트 작성)은 **토큰 기반이라 이미 보호**됨(`submit_orient_sheet`, 토큰 없으면 접근 불가). 브랜드 서베이 차단과 **무관 — 영향 주면 안 됨**.

### 이 제안과 충돌 가능성 있는 기존 동작
- ⚠️ **이전 「유지」 결정과의 정합**: 완전히 닫으면 영업 2단계 용도와 충돌. → **공개 제출만 차단**(완전 차단 아님)으로 정합 유지(사용자 확정 2026-06-30).
- **관리자 파이프라인 무영향**: 「광고주 신청」 페인·회사/브랜드 마스터·오리엔 연결·캠페인 비용카드·기존 신청 데이터는 **그대로 유지**. 막는 건 신규 제출 입구만.
- 그 외 충돌 없음(확인 완료).

### 미해결 백로그·관련 작업
- 2단계(영업 초대 링크 발급 화면) — 오리엔시트 토큰 발급 패턴(`create_orient_sheet`) 재사용 가능. 본 사양 범위 밖(영업 실제 필요 시).
- 랜딩페이지 + 1단계 이용문의 폼 신설(메모리 `project_brand_survey_future` 깔때기) — 별도.

## 2. 의심·경우의 수 (planning.md 규칙 B — 반대론자 모드)

### 깨질 수 있는 경우의 수
1. **(서버 방어선 필수)** 폼 화면만 가리면 우회 제출 가능(원격 호출 함수 직접 호출). **함수 자체가 거부**해야 진짜 막힌다 → 차단의 1차 방어선은 RPC.
2. **(빈 화면·에러 UX)** 옛 URL로 들어온 기존 브랜드에게 빈 페이지·원시 에러를 보이면 안 됨 → "현재 신규 접수하지 않습니다 + 문의 안내" 한국어 안내(브랜드사 대상 UI는 한국어).
3. **(오리엔시트 오차단)** 같은 sales 도메인의 `orient.html`은 토큰 보호된 별개 도구 → 차단이 오리엔 작성·발급에 영향 0이어야(함수·경로 분리 확인).
4. **(2단계 전환 매끄러움)** 지금 전면 차단 후 영업 발급 화면을 붙일 때, 차단 로직이 토큰 검증으로 자연스럽게 전환돼야 함 → 구현 방식 선택(§5)에 영향.
5. **(개발/운영 동기화)** 차단은 RPC(데이터베이스)+폼(코드) 양쪽. 운영 반영 시 운영 데이터베이스에 함수 갱신 + 폼 배포 둘 다.
6. **(관리자 대리 입력 여지)** 영업이 당장 못 보내는 공백 동안, 꼭 받아야 할 브랜드가 생기면? → 관리자가 「광고주 신청」 페인에서 직접 입력하는 경로가 이미 있는지 확인(없으면 2단계 발급으로). 본 사양은 공개 입구 차단까지.

### 현재 구현과 어긋나는 지점
- 이전 「유지」 결정과의 정합 1건 — 공개 제출만 차단으로 해소(§1). 그 외 충돌 없음(확인 완료).

### 의도 모호점
- "진행하지 않는다" = **무분별 공개 제출 차단**, 영업 2단계 용도는 유지(사용자 확정).
- "막는다" = 1단계로 **공개 제출 차단**, 발급 화면(영업 링크)은 2단계로 분리(사용자 확정).

## 5. 차단 방식 — 2안 비교 (개발/supabase-expert 결정 인계)

| | A. 전역 차단 플래그 | B. 토큰 인자 미리 받기 |
|---|---|---|
| 구현 | 설정값(단일행 설정 또는 RPC 가드) 확인 → 차단 | `p_invite_token` 인자 추가 + 검증 테이블, 유효 토큰 없으면 거부 |
| 지금 효과 | 전면 차단 | 발급 0개라 사실상 전면 차단 |
| 2단계 전환 | 플래그→토큰 검증으로 **교체** 필요 | 발급 화면만 추가하면 **자동으로 토큰 제출 허용** |
| 비용 | 최소(테이블 없이 가능) | 검증 테이블 신설(빈 테이블 시작) |
| 운영 유연성 | 관리자 토글로 임시 재개 가능(설정 테이블이면) | 토큰 단위 제어 |
| 권고 | — | **B 권고**(2단계 전환 매끄러움). 단 지금 비용 최소가 우선이면 A로 시작 후 2단계에서 토큰화 |

→ 최종은 supabase-expert 착수 시 확정. 사양은 **B 권고, A 허용**.

## 9. 구현 결과

### PR 1 — 공개 제출 차단

**구현일:** 2026-06-30
**마이그레이션:** `supabase/migrations/206_brand_survey_submit_lock.sql` (단일)
**브랜치:** `feature/brand-survey-submit-lock` → dev

**차단 방식 결정:** §5 **A안(설정 테이블)** 채택. B(토큰 인자)는 함수 시그니처 변경 → 폼 호출부도 바꿔야 해 비용↑. A는 시그니처 유지(폼은 에러 분기만), super_admin 토글로 임시 재개, 2단계 전환 시 함수 내부만 교체.
- `brand_survey_settings` 싱글톤(id=1, `submissions_open boolean DEFAULT false`=차단) + RLS(SELECT `is_admin()`/UPDATE `is_super_admin()`).
- `submit_brand_application` CREATE OR REPLACE — 0단계 차단 가드 추가(차단 시 `RAISE 'submissions_closed'` P0001). 087 본문·시그니처·anon GRANT·092/098 트리거 보존.
- `is_brand_survey_open()` anon 읽기 함수 추가(bool만 노출) — sales 폼이 로드 시 차단 여부 판단(설정 테이블은 anon SELECT 불가라 함수 경유).
- **관리자 override**: `admin_create_brand_application`(091) 기존 함수로 영업 공백 동안 관리자 페인 직접 입력(별도 경로 설계 불필요 — 이미 존재 확인).

**sales 폼(`dev/sales/` → 빌드 `sales/`):**
- `reviewer.html`·`seeding.html`: `</body>` 앞 `#surveyClosedOverlay` + 로드 시 `is_brand_survey_open()` false면 오버레이로 폼 덮음(서버 가드가 1차 방어). 동적 — 재개 시 즉시 반영.
- `index.html`: 선택 카드(`choice-list`) `display:none` + 차단 안내(한·일 토글 호환). **정적 — 임시 재개 시 함께 되돌려야 함**(reviewer 경고②).
- `orient.html`: **무변경**(토큰 보호 별개, §3④).
