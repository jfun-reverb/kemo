# 시행일 운영 규칙 — 배포 타이밍 거버넌스 (코드는 미리, 효력은 시행일)

**작성일:** 2026-06-02
**상태:** 기획 초안 (개발 미진입 — 규칙 파일화는 사용자 확인 후)
**작성 세션:** 기획/설계 세션

---

## 1. 배경 / 문제

약관·정책에 **시행일(부칙 공고일 ≠ 시행일)** 이 걸린 기능을 출시할 때, "기능을 언제 배포하고, 효력은 언제 발생하며, 시행 전까지 코드를 어떻게 관리하나"에 대한 일관된 규칙이 없다. 2026-06-02 사용자 질문으로 드러난 니즈:

- 부칙이 "공고 6/2 / 시행 6/9"면 관련 기능을 반드시 6/9에 배포해야 하는가?
- 코드는 미리 배포하고 효력만 시행일에 발생시킬 수 있는가? (자동 ON 포함)
- 시행이 30일 후인 기능을 그 전에 베타 운영할 수 있는가?
- 시행일 대기 중인 "잠든 코드" 위에 다른 작업이 얹힐 때 어떻게 관리하나?

이 규칙은 위 질문들의 표준 답을 영구 규칙으로 고정해, 앞으로 시행일이 걸린 기능마다 같은 판단을 반복하지 않게 한다.

| pg_cron 날짜 자동 실행 선례 | 있음 — 마이그레이션 166(위반기록 3년 자동 삭제), 다이제스트 메일 cron 등 |

### 이 제안과 충돌 가능성 있는 기존 동작
- `policy.md` "미구현 기능 선제 반영 금지" ↔ 본 규칙 "코드는 미리 배포 가능" — **층이 달라 충돌 아님**(아래 §4 3층 모델). 문서(약관)는 미리 못 쓰고, 코드는 미리 올릴 수 있으며, 집행만 시행일에 묶인다.
- `policy-versioning` "효력 동기화" ↔ 본 규칙 "배포-효력 분리" — **모순 아님**. policy-versioning은 *효력*을 시행·동의에 맞추라는 정책 층, 본 규칙은 그 효력을 *코드에서 게이팅*하는 메커니즘 층. 보완 관계.

### 미해결 백로그·관련 작업
- `policy-versioning` 사양서: 약관 버전관리·변경 통지·동의 시스템 (개발 미진입). 본 규칙과 짝.
- 약관 보강 4종 + 마이그레이션 166: 운영 배포·앱 공지 미완(시행일 2026-06-09). 메모리 `project_terms_reinforcement.md`. **본 규칙의 첫 적용 후보.**

## 3. 의심·경우의 수 (planning.md 규칙 B)

### 깨질 가능성

| # | 카테고리 | 시나리오 | 대응 |
|---|---|---|---|
| ① | 법률 — 유리/불리 판정 모호 | "새 알림"은 편의(유리)지만 메일 빈도 증가는 불편(불리). 판정이 갈리면 시행 전 가동 가부가 흔들림 | 애매하면 **불리로 간주** + `/약관확인` + 사용자 확인 (escape hatch) |
| ② | 기술 — 클라 시각 조작 | 인플 기기 시계를 시행일 이후로 조작 → 시행 전에 기능 동작 | **법적 집행은 서버 시각(now())만 신뢰.** 클라 시각은 UI 노출 정도만 |
| ③ | 기술 — 타임존 | "6/9"가 KST/JST 어디 기준인지 코드에 누락 → 9시간 어긋남 | 시각 상수·DB 비교에 항상 `+09:00` 오프셋 명시 |
| ④ | 운영 — 무인 자동 ON 위험 | 자동 ON은 직전 버그 발견돼도 사람 개입 없이 진행됨 | 자동 ON에는 **kill switch + 시행 당일 관리자 통지** 동반 의무 |
| ⑤ | 데이터 — DB는 플래그로 못 가림 | 마이그레이션으로 만든 테이블·트리거는 OFF여도 운영 DB에 존재. 다른 작업이 그 RLS·트리거 변경 시 잠든 기능이 ON 시점에 오작동 | DB 변경은 플래그와 무관하게 항상 영향 범위 점검 |
| ⑥ | 코드 — 잠든 코드 drift | OFF 경로를 아무도 테스트 안 하는 사이 다른 작업이 공유 함수·DOM·i18n 키를 깨뜨림 → 시행일에 폭발 | 개발서버는 **ON 상태로 검증**, 공유 지점 변경 시 OFF 경로 grep, 시행 후 플래그 청소 |
| ⑦ | UX — 인플 혼란 | 시행 전 화면에 새 기능이 보이거나, 시행 당일 갑자기 동작이 바뀜 | 시행 전 노출 차단 또는 "○월 ○일부터" 안내. 인플 친화 문구(`.claude/rules/ui.md`) |

### 현재 구현 충돌점
- 확인 완료 — §2 "충돌 가능성" 참조. 3층 모델로 분리하면 충돌 없음.

### 의도 모호점
- "관련 기능"이 가리키는 게 **유리한 변경(자동 삭제 등)** 인지 **불리한 변경(환수·제재)** 인지에 따라 답이 정반대 → 규칙에 기능 종류 판정을 1단계로 둠.
- "베타 운영"의 정의 — 운영 실회원 대상인지, 테스트 계정/사내인지에 따라 가부가 갈림 → 규칙에 대상별 표.

## 4. 제안 — 규칙 본문 초안

> 아래는 `.claude/rules/release-timing.md`(신규)로 들어갈 본문 초안. 사용자 승인 후 개발 세션 또는 사용자가 규칙 파일로 확정.

### 4-1. 3층 모델 (가장 중요 — 충돌 방지)

시행일이 걸린 기능은 **세 가지를 분리**해서 판단한다. 하나로 묶으면 기존 규칙과 모순돼 보인다.

| 층 | 무엇 | 시점 | 근거 규칙 |
|---|---|---|---|
| **1. 약관 문서** | 약관·개인정보처리방침에 해당 문구를 쓰는 것 | 미리 쓰지 않음 (실제 기능 배포에 맞춤) | `policy.md` "미구현 기능 선제 반영 금지" |
| **2. 코드 배포** | 코드가 서버에 올라가 동작 가능 상태 | **미리 가능** | 본 규칙 |
| **3. 효력(집행)** | 새 조항을 근거로 회원에게 권리·의무를 실제 행사 | **시행일부터** (+동의 절차 완료 시 동의 후) | `policy-versioning` 효력 동기화 |

핵심 한 줄: **"문서는 미리 안 됨 / 코드는 미리 됨 / 집행은 시행일."**

### 4-2. 기능 종류 판정 (1단계 — 항상 먼저)

| 종류 | 정의 | 시행 전 동작 가부 |
|---|---|---|
| 유리한 변경 | 회원에게 이득·보호 (예: 데이터 자동 삭제, 편의 기능) | 미리 가동 무방 (단 문서 문구 일치 위해 시행일 권장) |
| 불리한 변경 | 새 의무·제재·환수·불이익 | 코드는 미리, **집행은 시행일 이후 발생한 사유에만** (소급 금지) |
| 새 개인정보 수집 | 수집 항목·처리 목적 추가 | 시행 + 동의 완료 전 **운영 실회원 대상 수집 금지** |
| **판정 애매** | 유리/불리가 갈림 | **불리로 간주** + `/약관확인` + 사용자 확인 |

### 4-3. 자동 시행 ON — 시각 판정 위치

| 대상 | 판정 위치 | 신뢰도 |
|---|---|---|
| UI 노출·문구 전환 | 클라 상수 `new Date() >= EFFECTIVE`(오프셋 명시) | 조작해도 피해 작음 → 허용 |
| **법적 집행(환수·제재·데이터 처리)** | **서버측**(원격 호출 함수·보안 정책에서 `now() >= '...+09'`) | 서버 시각, 조작 불가 → 필수 |
| 일괄 상태 전환 | pg_cron 예약 (마이그레이션 166 선례) | 서버 → 신뢰 |

자동 ON 채택 시 **kill switch(비상 정지) + 시행 당일 관리자 통지**를 반드시 동반.

### 4-4. REVERB 기본 전략 — 브랜치 우선

| 방식 | 언제 |
|---|---|
| **브랜치 분리 (1순위 기본값)** | 대부분. dev에 보류 → 시행일에 dev→main 머지. 운영 코드베이스에 잠든 코드를 안 남김 |
| 인코드 플래그 (예외) | 무인 자동 전환이 꼭 필요하거나, 시행일에 사람이 머지를 못 누르는 경우만 |

보류 기능이 2개 이상 같은 영역을 건드리면 플래그 중첩이 위험 → 반드시 브랜치 분리.

### 4-5. 잠든 코드(시행 대기 코드) 관리

- **개발서버는 ON 상태로 검증** — OFF만 테스트하면 회귀가 시행일에 폭발
- **공유 지점 변경 시 OFF 경로 점검** — 공통 함수(storage.js)·DB 테이블/트리거·DOM id·i18n 키를 다른 작업이 건드리면 잠든 코드 호출부 grep (reverb-reviewer 역할)
- **DB는 플래그로 못 가림** — 마이그레이션 산출물은 OFF여도 존재. DB 변경은 항상 영향 범위 점검
- **시행 후 플래그 청소** — 플래그는 임시. 시행 직후 분기·OFF 경로 제거

### 4-6. 베타 운영 가부

| 베타 대상 | 시행 전 가능? |
|---|---|
| 단순 UI·유리한 기능 | 자유 |
| 새 개인정보 수집 / 불리한 의무 | ① 베타 참여자 별도 명시 동의(opt-in) / ② 테스트·사내 계정만 / ③ 개발서버만 — 셋 중 하나로 한정 |

운영 실회원 대상으로 동의 없이 새 조항 근거 동작을 시행 전 가동 = 위반 소지.

### 4-7. 적용 제외 (빠른 진행)
- 시행일이 안 걸린 일반 기능 (기존 git.md 워크플로 그대로)
- 약관·정책과 무관한 버그 수정·UI 조정
