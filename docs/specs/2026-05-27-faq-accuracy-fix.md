## 2. 🔴 정정 1 — 신청 승인(당첨) 앱 알림 신설 (Q2-1·Q2-2)

### 문제
- `notifications.kind`(마이그레이션 037·096·105)에 `deliverable_rejected`/`deliverable_changed`/`deliverable_approved`/`application_cancelled`/`message_received`만 존재. **신청 승인(당첨) 알림 종류·생성 코드가 없음.**
- FAQ Q2-1 "결과가 정해지면 앱 알림으로 안내", Q2-2 "당첨 시 앱 알림으로도 안내합니다" → **실제 없는 알림을 약속**. 인플은 응모이력을 직접 봐야만 당첨을 앎.

### 결정 (사용자) — **승인 알림 기능 추가** (FAQ 문구는 유지)
신청이 `pending → approved`로 바뀔 때 인플루언서에게 앱 알림 생성.

### 설계
- `notifications.kind` CHECK에 **`application_approved`** 추가 (신규 마이그레이션 — **155** 예상, 153 미성년·154 LIPS 예약 후)
- `applications` status 트리거: 기존 `trg_application_status_event`(AFTER UPDATE OF status, `record_application_status_event()`)가 이미 있음 → **그 경로 또는 신규 트리거**에서 `pending→approved` 시 `notifications` INSERT (deliverable 알림 패턴 미러링, SECURITY DEFINER + search_path 고정)
- 알림 내용(일본어): title 「キャンペーンに当選しました」, body 「{캠페인명}に当選しました。成果物の提出をお願いします。」 (실제 문구는 인플 안내 규칙 — 쉬운 말)
- `ref_table='applications'`, `ref_id=application_id`
- **인플 알림 모달 클릭 시 분기**: `kind='application_approved'` → 활동관리(`#activity`) 또는 응모이력으로 이동 (`application_cancelled`와 ref_table 공유하므로 kind 한정 필수 — 기존 message_received 분기 패턴과 동일)
- **메일 동반 여부**: 앱 알림 우선. 메일까지 보낼지는 운영 판단(추천: 1차는 앱 알림만, 인플 일일 다이제스트에 이미 「어제 승인」 섹션이 있으므로 중복 회피). 별도 즉시 메일은 보류.
- 중복 방지: 같은 신청 재승인(되돌리기 후 재승인) 시 미읽음 중복 INSERT 차단(기존 패턴 참조)

## 3. 🟠 정정 2 — 팔로워 자격 기준 문구 (Q1-1-c)

### 문제
- FAQ: "**대표로 설정한 SNS**의 팔로워 수로 확인" / 「代表に設定したSNSのフォロワー数で確認します」
- 실제(`application.js` 524~531): **캠페인이 기준으로 지정한 채널(`primary_channel`, 없으면 채널 리스트 첫 번째)**의 팔로워로 검증. 인플이 마이페이지에서 고른 대표 SNS(`primary_sns`)가 아님. + **리뷰어형(monitor)은 팔로워 조건 면제**(언급 없음).
- 부수: 동적 치환 `{current}`도 "대표 SNS 팔로워"로 사양화돼 있음(CLAUDE.md §5-1) → **캠페인 기준 채널 팔로워**로 넣도록 PR B 동적 치환 로직도 수정 필요.

### 정정 문구
**한국어:**
```
이 캠페인은 최소 팔로워 수 조건이 있습니다.
・필요 팔로워: {required}명 이상
・현재 팔로워: {current}명
1. 아래 버튼으로 마이페이지 SNS 계정을 엽니다.
2. 팔로워 수를 지금 최신 인원으로 고칩니다.
3. 고친 뒤 다시 응모해 보세요.
팔로워 조건은 이 캠페인이 기준으로 정한 SNS의 팔로워 수로 확인합니다. 리뷰어형(구매형) 캠페인은 팔로워 조건이 없습니다.
```
**일본어:**
```
このキャンペーンには「最低フォロワー数」の条件があります。
・必要なフォロワー数：{required}人以上
・あなたの現在のフォロワー数：{current}人
1. 下のボタンからマイページのSNSアカウントを開きます。
2. フォロワー数を、いまの最新の人数に直します。
3. 直したあと、もう一度ご応募をお試しください。
フォロワーの条件は、このキャンペーンが基準としているSNSのフォロワー数で確認します。なお、レシート提出型（購入型）のキャンペーンには、フォロワー数の条件はありません。
```

## 4. 🟡 정정 3 — 해시태그 안내 (Q3-3)

### 문제
- FAQ: "**#PR 또는 #広告 중 하나** 필수"

### 정정 문구

```
게시 시 다음 규칙을 지켜주세요.
1. 광고임을 알리는 해시태그를 반드시 넣습니다 — 각 캠페인 가이드라인에 지정된 태그(예: #PR · #広告 · #プロモーション · #アンバサダー 중 지정된 것)를 그대로 사용합니다.
```

```
投稿のときは、つぎの決まりを守ってください。
1. 広告であることを伝えるハッシュタグを必ず入れます。各キャンペーンのガイドラインで指定されたタグ（例：#PR・#広告・#プロモーション・#アンバサダー のうち指定されたもの）を、そのまま使ってください。
2. ブランドから指定されたハッシュタグやアカウントがある場合は、それも入れます。
3. 投稿は、決められた期間そのまま残しておきます。
くわしい条件は、各キャンペーンのガイドラインでご確認ください。
```

## 5-2. 🟠 정정 5 — "제품은 언제 오나요" 모집 유형 미구분 (Q4-1)

### 문제 (사용자 지적 + 추가 발견)
- Q4-1 답변은 **기프팅(무상 제공) 기준**으로만 "당첨 후 제품이 한국에서 발송됩니다"라고 안내.
- 하지만 **리뷰어형(monitor)은 제품이 배송되지 않고**, 캠페인에 안내된 상품 페이지(`campaigns.product_url`)에서 **직접 구매** 후 영수증 제출해야 함. **방문형(visit)은 매장 방문**이라 배송 아님.
- 게다가 Q4-1의 `relevant_stages=['approved_purchase','approved_visit']` — 리뷰어·방문형 단계에 노출되는데 정작 답변은 배송(기프팅) 내용 → **모순**.

### 🔴 추가 발견 — stage 판정이 리뷰어/기프팅을 구분 안 함 (구조 이슈)
- `faqComputeStatus`(`dev/lib/shared.js` 425): **`isVisit`(방문형)만 분기**하고, 리뷰어(monitor)와 기프팅(gifting)은 둘 다 `approved_purchase`/`approved_post`로 동일 처리.
- 결과: **기프팅 당첨자**가 모집 마감 전(recruit phase) 단계면 stage=`approved_purchase`가 되어, **Q3-1(영수증 제출)·Q4-1이 기프팅 당첨자에게도 노출**될 수 있음. 기프팅은 영수증·직접구매가 없으므로 오안내.
- 즉 Q4-1만의 문제가 아니라, **모집 유형 구분이 필요한 모든 노드(Q3-1 영수증·Q4-1 배송)에 영향**.

### 정정안 — 답변 문구에 모집 유형 구분 (stage 보강 없이 우선 대응)
인플루언서는 자기 신청 캠페인 유형을 알므로, 답변에 유형별 안내를 모두 담는다.

**Q4-1 한국어:**
```
제품을 받는 방법은 캠페인 종류에 따라 다릅니다.
■ 무상 제공(기프팅) 캠페인: 당첨 후 제품이 순서대로 발송됩니다. 한국에서 보내므로 도착까지 시간이 걸릴 수 있고, 발송 안내는 앱 알림·메시지로 연락합니다.
■ 구매형(리뷰어) 캠페인: 제품이 따로 배송되지 않습니다. 캠페인에 안내된 상품 페이지에서 직접 구매하신 뒤, 영수증을 제출해 주세요.
■ 방문형 캠페인: 제품 배송이 아니라, 지정된 매장을 직접 방문해 체험합니다.
한참 기다려도 안 오거나 모르겠으면 직접 문의 바랍니다.
```
**Q4-1 일본어:**
```
商品の受け取り方は、キャンペーンの種類によって異なります。
■ 無償提供（ギフティング）のキャンペーン：当選されたあと、商品が順番に発送されます。韓国から送るため、お手元に届くまでお時間をいただくことがあります。発送のお知らせは、アプリの通知やメッセージでご連絡します。
■ 購入型（モニター）のキャンペーン：商品は別途配送されません。キャンペーンで案内された商品ページから、ご自身でご購入のうえ、レシートを提出してください。
■ 訪問型のキャンペーン：商品の配送ではなく、指定された店舗にご自身で訪問して体験していただきます。
しばらく待っても届かないときや、わからないときは、下の「直接お問い合わせ」からご連絡ください。
```

### Q3-1(영수증) 보완
- Q3-1 답변 첫 줄에 **"이 안내는 구매형(리뷰어) 캠페인 전용입니다"** 한 줄 추가 권장 (기프팅 당첨자에게 노출돼도 혼동 방지).
- 일본어: 「※このご案内は、購入型（モニター）のキャンペーン向けです。」

### stage 보강 (선택 — 근본 해결)
- `faqComputeStatus`에 gifting 구분을 넣어 `relevant_stages`로 리뷰어 전용/기프팅 전용 노드를 정확히 나누는 방법. 단 stage 코드 신설(`approved_gifting` 등) + 노드 재태깅 + 인플/관리자 공용 판정 동시 수정이라 범위가 큼. → **1차는 문구 유형 구분으로 대응, stage 보강은 별도 판단.**
