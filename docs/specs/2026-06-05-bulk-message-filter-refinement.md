## 현재 상태 (작성일 2026-06-05 기준)

### 관련 코드·DB·UI 진입점
- **대상 추출 함수** `resolve_bulk_recipients`(마이그레이션 169, 12인자) — 단일 캠페인(`p_campaign_id`) 단위. 첫 조건이 `WHERE a.campaign_id = p_campaign_id`. cancelled 응모 항상 제외. 반환 = **응모(application) ID 배열**
  - 필터: 응모상태 / 영수증 결과물 상태 / 게시물·이미지 결과물 상태 / 인플 보유 SNS 채널 / 도도부현 / 팔로워(채널별·합산) / 인증·위반·블랙리스트
  - 결과물 상태 필터는 **EXISTS 방식 + 그룹 내 OR**: "해당 status 결과물이 하나라도 있으면 통과"
  - "미제출(none)" 정의 = "해당 종류 결과물 행 자체가 없음"
- **발송 함수** `send_application_message_bulk`(마이그레이션 168, 7인자) — 응모 ID 배열을 받아 **응모건마다 1통씩** 메시지 INSERT. 1회 최대 200(=응모건 단위). 알림 제목에 캠페인명 부착
- **UI** `dev/js/admin-messaging.js`
  - 필터 상태 코드: `BULK_APP_STATUSES`(line 958), `BULK_DELIV_STATUSES`(line 963 — `none/pending/approved/rejected`, **draft 없음**)
  - 기본 체크(`renderBulkStatusFilters` line 1167): 응모=**승인만 ON** / 결과물=**승인 제외 나머지 ON**
  - 캠페인 여러 개 선택 시 캠페인별 추출 후 응모 ID **합집합**(`recountBulk` line 1250)
  - 영수증 필터는 리뷰어(monitor) 포함 시에만 적용(`collectBulkFilters` line 1205)
  - SNS 채널 UI 옵션은 4종(Instagram/X/TikTok/YouTube)만 노출 — Qoo10·LIPS·@cosme는 함수 내부에서 "인스타 핸들 보유"로 대리 판정
- **DB API 래퍼** `resolveBulkRecipients`(`dev/lib/storage.js:2846`), `sendApplicationMessageBulk`(line 2807)
- **운영 배포 상태**: 마이그레이션 167~169 + 일괄발송 UI는 운영 배포 완료(메모리 `project_bulk_message_pr3.md`, main b17ae48, 2026-06-04)

### 이 제안과 충돌 가능성 있는 기존 동작
- **응모건 메시지 구조**: 메시지는 반드시 하나의 응모건(`application_id`)에 종속. "사람단위 발송"을 구조적으로 표현할 수 없음 → 본 사양은 **일괄발송을 "신청건별 안내 전용"으로 성격 확정**하고, 사람단위 공지는 별도 주제로 분리(아래 「범위 밖」)
- 마이그레이션 169가 운영 적용 완료 상태 → 함수 재정의(신규 마이그레이션 1개)는 169 위에 DROP+CREATE로 얹어야 함(인자 수 변경. 번호는 개발 세션이 생성 시점에 확정)

## 제안 / 설계

일괄발송을 **신청건별 안내 전용**으로 성격 확정하고, 4개 항목을 정교화한다.

### 항목 A — 카운트 정정 + 기본값 안내 (지점 1·4, 화면만)
1. **카운트 표시**: 현재 "○명" → **"○건 (○명)"** 으로 분리 표시. "건"이 실제 발송 통 수(신청건), "명"이 도달 인원(distinct user). 동일인 중복을 관리자가 즉시 인지
   - 사람 수 계산: 추출 결과(응모 ID 배열)로 `applications.user_id` distinct 카운트. 구현 방식(함수 반환 확장 vs 보조 집계)은 구현 세션 결정
2. **현재 조건 한 줄 요약**: 모달에 체크 조합이 뜻하는 바를 동적 문장으로 표시
   - 예: 기본 조합 → "지금 조건: 당첨됐고 결과물이 아직 다 승인되지 않은 신청건"
   - 응모상태·결과물상태 조합에 따라 문구 분기(완료자/미제출/반려 재요청 등)

### 항목 B — 결과물 "완전 승인만" 옵션 (지점 2, 데이터베이스 함수 변경)
- **정의**: "완전 승인" = 해당 종류 결과물이 1건 이상 존재 AND 미승인(pending/rejected/draft) 결과물 0건
- **함수 변경**(신규 마이그레이션 1개 — 항목 D와 같은 파일): `resolve_bulk_recipients`에 파라미터 2개 추가
  - `p_receipt_all_approved boolean DEFAULT false` — true면 영수증이 "완전 승인"인 응모만
  - `p_post_all_approved boolean DEFAULT false` — true면 게시물·이미지가 "완전 승인"인 응모만
  - true일 때 판정: `EXISTS(승인 결과물) AND NOT EXISTS(pending/rejected/draft 결과물)` (해당 kind 한정)
  - 인자 수 변경 → 169 시그니처 DROP 후 14인자로 CREATE
- **UI**: 결과물/영수증 상태 영역에서 "승인" 체크 시, 그 옆에 토글 「완전 승인만(부분 승인 제외)」 노출. 체크하면 위 파라미터 true 전달
- 효과: 부분승인 10.7% 오포함 제거. "결과물 다 끝난 신청건"을 정확히 추출

### 항목 C — 필터 라벨·타입혼합 안내 (지점 3·5, 화면만)
1. **채널 필터 라벨 명확화**: "채널"/"참여 조건" → **"인플루언서 보유 SNS"** 로 라벨 변경 + 설명 한 줄("캠페인 모집 채널이 아니라 인플루언서가 가진 SNS 기준")
2. **타입 혼합 안내**: 선택 캠페인에 리뷰어와 기프팅·방문형이 섞이면, 영수증 필터 영역에 안내 배너 — "영수증 조건은 리뷰어 신청건에만 적용됩니다. 기프팅·방문형 신청건은 영수증 조건과 무관하게 포함됩니다"

### 항목 D — draft 옵션 보강 (지점 8, 화면+함수 소폭)
- **방향**: "미제출(none)"의 의미를 **"비draft 결과물이 하나도 없음"** 으로 확장(= 결과물 행이 없거나, 있어도 전부 draft인 경우 모두 "미제출"로 취급). 인플 관점에서 draft는 "아직 제출 안 함"이므로 미제출과 동일
- **함수 변경**(항목 B와 같은 신규 마이그레이션 파일): 'none' 판정을 `NOT EXISTS(해당 kind의 status <> 'draft' 결과물)` 로 수정
- UI 옵션은 그대로(draft 별도 노출 안 함 — "미제출"에 흡수)
