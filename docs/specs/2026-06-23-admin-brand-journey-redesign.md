# 관리자 광고주 여정 재설계 — 메뉴 개편 + 오리엔시트 발행·취합 워크플로

**작성일:** 2026-06-23
**작성 주체:** 기획 세션
**관련:** 오리엔시트 사양서 `docs/specs/2026-06-18-brand-self-orient-sheet.md`(§14·§15) / 베타 허브 `docs/specs/2026-06-18-beta-launch-plan.md`
**성격:** 방향·구조 설계 초안. 사용자 확정 = A안(광고주 여정 + 캠페인 운영 2분할). 큰 개편이라 PR 분할·페인 이동은 구현 착수 시 reverb-planner 정밀화.

## 0. 한 줄 요약

광고주 **신청 → 검수·견적 → 오리엔시트 발급·취합 → 캠페인 발행**이 현재 "브랜드 서베이"와 "캠페인" 두 메뉴로 끊겨 있던 것을, **「광고주」(발행 전 여정) + 「캠페인」(발행 후 운영) 2분할**로 재편하고, 오리엔시트 발급·취합·발행을 **신청 상세(파이프라인) 한 흐름**에 통합한다.

## 1. 현재 상태 (planning.md 규칙 A — 검증 완료, 내부 코드 조사 기반)

### 현 좌측 메뉴 (6개 묶음, `dev/admin/index.html` L86-133)
- 공지(공지사항·오픈 예정 기능) / 대시보드(전체 현황)
- **캠페인**: 운영 현황(`brand-ops`) · 캠페인 관리(`campaigns`) · 신청 관리(`applications`, 인플) · 결과물 관리(`deliverables`) · 메시지(`messages`)
- **브랜드 서베이**: 현황 대시보드(`brand-dashboard`) · 회사 관리(`companies`) · 브랜드 관리(`brands`) · 신청 목록(`brand-applications`) · 오리엔시트(`orient-sheets`)
- 회원관리(인플루언서) / 관리자설정(기준데이터·FAQ·관리자계정·오류로그·내계정)

### 현 광고주 운영 흐름
신청 접수(sales) → 검수·견적(브랜드 서베이 > 신청목록, `brand_applications.status` 10단계 new→reviewing→quoted→paid→kakao_room_created→**orient_sheet_sent**→schedule_sent→**campaign_registered**→done/rejected) → 오리엔시트 발급(브랜드 서베이 > 오리엔시트, `create_orient_sheet`) → 캠페인 발행(캠페인 > 캠페인 관리, `campaigns.source_application_id`·`link_campaign_to_application`) → 인플 신청·결과물(캠페인 > …)

### 연결 구조
- `brand_applications.status`(10단계) · `orient_sheets.application_id`(신청↔오리엔시트) · `campaigns.source_application_id`(신청↔캠페인) · `link_campaign_to_application` RPC.
- 회사 > 브랜드 > 신청 > 캠페인 4단 계층(`companies`·`brands`·`brand_applications`·`campaigns`).

### 이 제안과 충돌·문제
- ⚠️ **광고주 흐름이 "브랜드 서베이"와 "캠페인" 두 묶음으로 분리** — 신청→오리엔시트→발행이 메뉴를 넘나듦.
- 신청 → 오리엔시트 → 캠페인 발행 경로가 화면상 끊겨 보임(`link_campaign_to_application` 모달이 더보기에 숨음, brand-applications↔orient-sheets 연결 미약).
- 4단 계층이 화면상 흩어짐(마스터=브랜드 서베이, 운영=캠페인).

## 2. 업계 벤치마크 (외부 조사 — ⚠️ 자동조사 2회 실패, 도메인 지식 기반)

> ⚠️ deep-research 2회 모두 WebSearch 서버 rate limit으로 0건 수집(환경 문제). 아래는 **도메인 지식 기반 일반 패턴**으로, **출처 정밀도 낮음 → 출처 보강은 과제**(환경 회복 후 재조사 또는 플랫폼 화면 자료 확보 시).

- **해외(GRIN·Aspire·Upfluence·CreatorIQ)**: 좌측 메뉴 대체로 대시보드 / 크리에이터 / **캠페인** / 콘텐츠 / 리포트 / 결제 / 설정. **캠페인이 운영 중심 단위**, 브리프(오리엔테이션)는 캠페인 생성의 한 단계, 콘텐츠 검수는 캠페인 하위. 엔터프라이즈(CreatorIQ)는 브랜드/계정 계층 존재.
- **한국 체험단(레뷰)**: 광고주가 캠페인 등록(상품·미션·모집조건) → 신청자 선정 → 콘텐츠 검수(광고주 셀프 등록형).
- **에이전시형 공통**: 클라이언트(브랜드) 워크스페이스 분리 + 공유 링크/승인으로 브랜드 입력 수집(풀 계정 없이). ← REVERB 토큰 오리엔시트가 여기에 해당.
- → 공통 결론: **"발행 전 준비(브랜드·브리프)"와 "발행 후 운영(캠페인·콘텐츠)"이 단계로 구분**되고, 브리프는 발행 파이프라인의 한 단계. **A안과 부합.**

## 3. 의심·경우의 수 (planning.md 규칙 B)

1. **권한별 메뉴 차이**: super_admin / campaign_admin / campaign_manager 가 보는 메뉴·작업이 의도대로 다른지(개편 시 권한 가드 재점검).
2. **비-서베이 오리엔시트**: 광고주 신청 없이 브랜드만으로 발급하는 건(§14)은 신청 파이프라인 밖 → "브랜드 관리"에서 발급 경로 유지 필요.
3. **데이터 통합 아님**: `brand_applications`(신청)와 `campaigns`(발행)는 별 테이블. "파이프라인 단계 표시"는 **두 테이블을 한 흐름으로 보여주는 화면 작업**이지 테이블 통합이 아님(통합 비용·위험 회피).
4. **페인 이동 영향**: 메뉴 재배치 시 라우팅(`data-pane`·해시 딥링크)·새로고침 리다이렉트·사이드바 배지·`refreshPane` 매핑 영향. 기존 북마크·문서 링크 깨짐.
5. **핫스팟 충돌**: `admin-brand.js`·`admin-applications.js`·`admin-orient.js`·`admin-company.js` 다수 동시 수정 → PR 시퀀셜(병렬 금지).
6. **외부 출처 미확보**: 벤치마크 근거가 도메인 지식뿐 → 방향 검증이 약함(출처 보강 과제, §2).
7. **UX**: 광고주 신청 1건의 "현재 단계"가 한눈에 보이고 "다음 단계" 버튼이 명확한지(파이프라인의 핵심). 단계 역행(되돌리기)·중단(반려) 경로.

## 4. A안 — 메뉴 개편 (사용자 확정 2026-06-23)

| 섹션 | 메뉴 | 비고 |
|---|---|---|
| **광고주 (발행 전 여정)** | 현황 대시보드 / 회사 관리 / 브랜드 관리 / **광고주 신청(파이프라인)** / 오리엔시트 현황 | 현 "브랜드 서베이"를 여정 중심으로 확장·개명. 마스터(회사·브랜드) + 신청 파이프라인 |
| **캠페인 (발행 후 운영)** | 운영 현황 / 캠페인 관리 / 인플 신청 관리 / 결과물 / 메시지 | 현 "캠페인" 섹션 유지(발행된 캠페인 운영) |

- **경계 원칙**: **발행 전(광고주 준비) = 「광고주」 / 발행 후(인플 운영) = 「캠페인」.** 캠페인 발행 시점이 두 섹션의 경계.
- 회사 > 브랜드 > 신청 > 캠페인 4단 계층 중 회사·브랜드·신청은 「광고주」, 캠페인은 「캠페인」.
- 명칭(「광고주」 vs 「브랜드 서베이」 유지)은 사용자 확인(§7).

## 5. 오리엔시트 발행·취합 워크플로 (핵심 — 신청 파이프라인 한 흐름)

광고주 신청 1건을 열면 그 안에서 단계가 이어진다:

1. **신청 접수** → 「광고주 신청」 파이프라인에 행/카드(상태 배지)
2. 신청 상세에서 **검토 · 견적 · 입금** 진행(기존 10단계 활용)
3. 「**오리엔시트 발급**」 → 형식·제품 선택(§15) → 토큰 링크 생성 **+ 메일 발송**(§14) → 상태 `orient_sheet_sent`
4. 브랜드 작성·제출 → 신청 상세에 **「취합 완료」 + 내용 미리보기**(읽기 전용)
5. 「**캠페인 발행**」 → 오리엔시트 자동 채움(prefill, §15) → 일본어 보완 → 발행 → 상태 `campaign_registered` → **「캠페인」 섹션 운영으로 이동**
6. **(비-서베이 건)** 신청 없이 **브랜드 관리에서 오리엔시트 발급** → 4·5단계 동일(신청 파이프라인 밖, §14-A)

- 핵심: 오리엔시트 발급·취합·발행이 **「광고주 신청」 한 화면(파이프라인)** 안에서 다음 단계로 이어짐(현재 메뉴를 넘나들던 문제 해소).
- 오리엔시트 별도 메뉴는 **"현황(취합 상태 일람)" 보조**로 두되, 발급·취합·발행 주 진입은 신청 상세(권고, §7 확인).

## 6. PR 분할 (개략 — 착수 시 reverb-planner 정밀화)

> ⚠️ `admin-brand.js`·`admin-applications.js`·`admin-orient.js`·`admin-company.js` 핫스팟 다수 → **시퀀셜**.

- **PR A — 메뉴 재배치(화면 이동)**: 사이드바 2섹션(「광고주」/「캠페인」) 재구성 + `data-pane` 라우팅·딥링크·새로고침 리다이렉트·배지 정리. 기능 로직 변경 없이 위치만.
- **PR B — 신청 파이프라인 화면**: 신청 상세에 단계 표시 + "다음 단계"(검토→견적→오리엔시트 발급→취합→발행) 버튼. brand-applications↔orient-sheets↔campaigns 연결 가시화.
- **PR C — 오리엔시트 발급·취합·발행 통합**: 신청 상세에서 발급(§14·§15)·취합 미리보기·자동 채움 발행 연결. (오리엔시트 사양서 §14~15 구현과 맞물림.)
- 순서·의존성은 오리엔시트 본체 구현(§15 재설계)과 함께 계획.

## 7. 사용자 확인 필요

- **A. 섹션 명칭**: 「광고주」 vs 현 「브랜드 서베이」 유지 vs 다른 명칭(예: 「광고주 관리」).
- **B. 오리엔시트 메뉴**: 신청 파이프라인에 **통합 + 현황 보조 메뉴**(권고) vs 별도 메뉴 유지.
- **C. 파이프라인 표현**: 신청 목록을 **칸반(단계별 열)** vs **리스트+상태 탭**(현행) 중 무엇으로. (현 리스트+탭 유지가 변경 최소.)
- **D. 외부 출처 조사**: 환경 회복 후 deep-research 재시도 여부(벤치마크 근거 보강).
