**상태:** 기획 초안 (개발 미진입)
**작성 세션:** 기획/설계 세션

## 1. 배경 / 문제

서비스 기능이 점점 많아지면서 "전체가 잘 작동하는가 / 불필요·충돌 코드는 없는가 / 정책과 맞는가"를 종합 점검할 체계가 필요하다. 2026-06-02 사용자 니즈.

**진단: REVERB는 이미 도구는 다 갖췄고, 빠진 건 "주기"와 "통합 체크리스트"와 "건강도 기록"이다.**

| 이미 있는 것 (도구) | 빠진 것 (체계) |
|---|---|
| `reverb-reviewer`(매 커밋), `reverb-qa-tester`(플로우), `reverb-supabase-expert`(DB) | **주기(cadence)** — 전수 점검이 "필요할 때 한 번"뿐 (26,600줄 검토는 1회성, 메모리 `project_code_review_cleanup`) |
| `reverb-team-manager` — **에이전트 팀** 건강도 주기 감사 (Light Weekly/Full Monthly) | **통합 체크리스트** — 점검 차원이 여러 규칙·머릿속에 흩어짐 |
| `docs/CODEMAPS/*` 4장, `docs/FEATURE_SPEC.md`, `CLAUDE.md` | **건강도 기록** — 점검 결과를 남겨 추세를 보는 저장소(`docs/health/`) 없음 |
| `/약관확인`·`/배포진단`·`/code-review`·`/db-check` | **서비스 자체를 점검하는 통합 명령** 없음 (개별 명령만) |

핵심 통찰: **`team-manager`가 "에이전트 팀"을 점검하듯, "서비스 자체"를 점검하는 짝**이 없다. 이 사양서는 그 짝을 만든다.

## 2. 현재 상태 (planning.md 규칙 A — 2026-06-02 검증)

### 직접 확인 결과
- 에이전트 5종: `reverb-planner`/`reverb-reviewer`/`reverb-supabase-expert`/`reverb-qa-tester`/`reverb-team-manager`
- `team-manager`는 **에이전트 조직**만 감사(역할 중복·과부하·호출 빈도). 서비스 코드·DB·정책 정합성은 대상 아님.
- 점검 관련 슬래시 커맨드: `/약관확인`(정책), `/배포진단`(배포), `/code-review`(변경 단위), `/db-check`(DB 상태) — **전 차원 통합 명령 없음**
- `docs/CODEMAPS/` 4장(admin-app·brand-survey·data-layer·influencer-app) + README, 2026-06-01 갱신
- `docs/audits/`·`docs/health/` **없음**. `docs/research/`에 1회성 분석만 누적

### 이 제안과 충돌 가능성
- **충돌 없음 — 확인 완료.** 기존 도구를 오케스트레이션할 뿐 신규 에이전트를 만들지 않음. `team-manager`(조직)와 본 체계(서비스)는 대상이 달라 중복 아님.

### 미해결 백로그·관련
- 1회성 전수 검토 실적: 메모리 `project_code_review_cleanup`(26,600줄 7영역 병렬 — 버그 3 + 정리). **이 1회성을 정례화**하는 게 본 사양서.
- 후속 백로그 3건(insertReceipt 고아·영수증 무압축 HEIC·검수 후 camp-applicants stale)이 그 검토에서 나옴 → 발견→백로그 연결 고리의 실제 예시.

### 의도 모호점
- "전체 점검"의 범위 — 코드만? DB·정책·문서·보안·성능까지? → 7차원으로 정의(§4-1)해 모호성 제거.
- "유지보수"가 점검(진단)인지 수리(조치)인지 → 본 체계는 **진단 + 백로그 등록까지**, 실제 수리는 개발 세션이 별도 진행.

## 4. 제안 — 설계

### 4-1. 점검 7차원

| # | 차원 | 무엇을 보나 | 사용 도구 |
|---|---|---|---|
| 1 | 기능 정합성/회귀 | 핵심 플로우(로그인·응모·검수·관리자)가 작동하는가 | `reverb-qa-tester` (full) |
| 2 | 죽은·중복·충돌 코드 + **정합성 미러** | 안 쓰는 함수, 잔존 참조, 중복 로직 + 하드코딩 미러 drift·핫픽스 횡단 재발·캐시 의존 동기함수 (§4-1-2) | 전수 검토(Explore 병렬) |
| 3 | 정책 정합성 | 약관·개인정보처리방침 vs 실제 동작 | `/약관확인` |
| 4 | DB 건강도 | 안 쓰는 테이블·컬럼, 보안 정책 누락, 마이그레이션 drift | `reverb-supabase-expert` |
| 5 | 문서 정합성 | `CLAUDE.md`·`FEATURE_SPEC.md`·CODEMAPS vs 실제 코드 | CODEMAPS 갱신 |
| 6 | 보안 | 보안 정책 감사, 시크릿 노출, 익명 접근면 | `reverb-supabase-expert` + grep |
| 7 | 성능 | 1000행 한계, 인덱스, 로딩, 번들 | 코드 점검 + 운영 실데이터 |
| 8 | 외부 시스템 정합성 | Vercel·Brevo·Supabase 대시보드·DNS 운영 상태 (repo로 추적 안 되는 설정 drift) | 자동 확인 + 사람 체크리스트 혼합 |

### 4-1-1. 차원 8 — 외부 시스템 정합성 (상세)

코드·마이그레이션은 PR로 추적되지만 **대시보드 수동 설정은 repo에 반영 안 됨**(`CLAUDE.md`·`git.md` 반복 강조). REVERB에서 가장 자주 사고 난 영역(배포 큐·메일 한도·cron 미등록)이라 별도 차원으로 둔다.

점검 방식이 갈린다:

| 점검 항목 | 방식 |
|---|---|
| pg_cron job 가동 여부(다이제스트·마이그레이션 166 위반기록 삭제), Edge Function 배포 목록 | ✅ 자동 (SQL `cron.job` / `supabase functions list`) |
| Vercel 배포 상태·최신 커밋 반영 | ✅ 자동 (`gh` deployments) |
| 발신 도메인 DNS (SPF/DKIM/DMARC) | ✅ 자동 (`dig`) |
| 코드 내 하드코딩 URL/key, 마이그레이션 양 서버 적용 여부 | ✅ 자동 (grep/SQL) |
| Brevo 월 쿼터 잔량(20,000통) | ⚠️ 사람 — Brevo 대시보드 확인 요청 |
| Auth Confirm email 토글(운영 ON·개발 OFF), Rate limit(운영 100/h) | ⚠️ 사람 — Supabase 대시보드 확인 요청 |
| Storage 버킷·정책 존재, Vercel 도메인·환경변수 | ⚠️ 사람 — 대시보드 확인 요청 |

→ 완전 자동은 불가. **"자동 확인 가능 항목은 명령이 직접 + 수동 항목은 체크리스트 제시"** 혼합.

### 4-1-2. 차원 2 — 정합성 미러 점검 (상세)

2026-06-02 발견(`@cosme`/LIPS 채널 코드가 화면에 원문 노출)에서 추가된 점검 항목. **죽은 코드·잔존 참조 점검(구조적 결함)으로는 안 잡히는 유형** — 각 부분은 정상인데 조합·타이밍·데이터 drift에서 깨진다. 같은 유형 재발을 잡기 위한 3개 하위 점검:

| 하위 # | 점검 | 방법 | 근거 사례 |
|---|---|---|---|
| 2-A | **하드코딩 미러 ↔ DB 원천 정합** | 코드에 DB 데이터를 하드코딩한 상수(`CHANNEL_LABEL_FALLBACK` 류 라벨·상태·카테고리 미러)가 `lookup_values` 등 원천 목록과 동기화돼 있나. 원천에 있는데 미러에 없는 키 = drift | 마이그레이션 157(LIPS·@cosme 추가) 때 DB만 갱신, `dev/js/ui.js:192` 폴백 상수 누락 → 캐시 미로드 시 코드 원문 노출 |
| 2-B | **이전 핫픽스 패턴의 횡단 재발** | "보장"·"폴백"·"임시"·"진입 전 1회" 류 주석/가드를 grep → 한 곳만 패치한 패턴이 다른 진입 경로에 그대로 남아 있나 | `dev/admin/app.js:91` 주석이 같은 버그를 admin에만 패치 → 다른 경로 재발 |
| 2-C | **비동기 캐시 의존 동기 함수** | 캐시 로드를 전제하는 동기 함수(`getChannelLabel` 등)가 캐시 미로드 진입 경로에서 호출되나 | `getChannelLabel`이 `_lookupCache.channel` 미로드 시 폴백 추락 |

**왜 별도 항목인가**: 전수조사(죽은 코드·stale 문서)는 "구조적으로 틀린 것"에 강하지만, **"각 부분은 맞는데 코드 안 하드코딩 미러의 drift / 런타임 타이밍"** 에 약하다. 이 빈틈이 차원 2와 차원 8(외부 정합) 사이에 있어 별도 명시.

### 4-2. 차원별 주기 차등 (비용 대응)

| 주기 | 차원 | 트리거 |
|---|---|---|
| **배포 전마다** | 1(회귀, 변경 영역만) | 기존 reviewer/qa-tester 게이트 — 이미 가동 중 |
| **월 1회 (빠른 점검)** | 2(죽은 코드)·5(문서 정합성) | `/서비스점검` 기본 모드 |
| **분기 1회 (심층 풀스윕)** | 1~7 전체 | `/서비스점검 심층` |
| **수시** | 3(정책)은 약관 변경 시, 4·6(DB·보안)은 마이그레이션 추가 시, **8(외부 시스템)은 cron·Edge Function·도메인·SMTP·DNS 변경 시** | 해당 작업에 동반 |

→ `team-manager`의 Light Weekly / Full Monthly 패턴을 서비스 버전으로 미러링.

### 4-3. 통합 명령 `/서비스점검` (신규 슬래시 커맨드)

- 기본 모드(빠른): 차원 2·5만, Explore/reviewer 병렬 → 10~20줄 리포트
- 심층 모드(`심층` 인자): 7차원 전체 병렬 점검 → 상세 리포트
- **신규 에이전트 만들지 않음** — 기존 5종 + Explore + 슬래시 명령을 오케스트레이션만

### 4-4. 건강도 리포트 (신규 저장소 `docs/health/`)

- 파일: `docs/health/YYYY-MM-DD-service-health.md`, 점검마다 누적
- 양식(필수 항목):
  - 점검 모드·날짜·점검한 차원 / **점검 안 한 차원 명시**(silent skip 금지)
  - 차원별 발견(없으면 "✅ 이상 없음")
  - **발견 → 백로그 연결**: 메모리 슬러그 또는 사양서 경로로 등록 (조치 누락 방지)
  - 이전 리포트 대비 추세(반복 발견 항목 플래그)

### 4-5. 발견 → 조치 고리 (가장 중요)

점검은 **진단 + 백로그 등록**까지만. 실제 수리는 개발 세션이 백로그를 받아 별도 진행. 리포트에서 발견 항목은 반드시:
- 메모리(`project_*` 또는 `feedback_*`)에 등록하거나
- 사양서(`docs/specs/`)로 승격하거나
- 즉시 수정 가능하면 개발 세션에 인계
