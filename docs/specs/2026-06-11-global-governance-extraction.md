## 현재 상태 (2026-06-11 검증)

### 진단 — 왜 적용이 안 되나
- **규칙·에이전트·커맨드는 현재 작업 폴더의 `.claude/`만 읽힌다.** 상위 폴더(`projects/`)나 옆 프로젝트의 `.claude/`는 자동으로 안 온다.
- 글로벌 `~/.claude/`에는 `CLAUDE.md`(짧은 개인 규칙)와 `settings.json`만 있고 **`agents/`·`commands/`·`rules/` 폴더가 비어 있다.** → REVERB의 규칙 15개·에이전트 4개·커맨드 19개는 REVERB 폴더 안에서만 작동.
- **jfun-work는 이미 손으로 복사**해 갔다: `interaction.md`·`quality.md`·`request-validation.md`(REVERB와 동일 이름) + `playwright.md`·`reverb-data.md`(jfun 특화) + 에이전트 `reverb-data-expert.md`. ⚠️ `session-roles.md`는 복사 안 됨 → jfun에서 "고문세션" 호출 불가.
- **결과: drift(버전 어긋남).** REVERB `interaction.md`를 개선해도 jfun 복사본은 안 따라온다.

### 확인된 Claude Code 메커니즘 (claude-code-guide 에이전트 확인)
- `CLAUDE.md`는 현재 폴더 → 상위 → 홈까지 **거슬러 올라가며 전부** 로드된다. 즉 `projects/CLAUDE.md`를 두면 그 아래 모든 프로젝트가 자동 로드.
- `@~/.claude/rules/파일.md` 형태 **import 지원**(홈 `~` 경로 가능, 최대 4단계 재귀). 와일드카드 import(`*.md`) 지원 여부는 **미검증** → 개별 파일 나열로 안전하게.
- **`agents/`·`commands/` 폴더는 상위탐색 안 됨** — 글로벌(`~/.claude/`) 또는 현재 프로젝트에만. 같은 이름이면 프로젝트가 글로벌을 덮어씀.
- **`.claude/rules/` 자동 주입은 이 환경(ECC 하네스) 동작**으로 보임 — REVERB `CLAUDE.md`는 rules를 `@import`하지 않는데도 세션 시작 시 15개가 "project instructions"로 주입됨. **글로벌 `~/.claude/rules/`가 같은 방식으로 자동 주입되는지는 미검증** → 1단계에서 글로벌 `CLAUDE.md`의 `@import`로 로드를 **명시 보장**하는 게 안전(자동 주입에 의존하지 않음).

### 사용자 결정 (2026-06-11)

## 의심·경우의 수 (이 작업에서 깨질 수 있는 것)

1. **글로벌 `~/.claude/rules/` 자동 주입 미보장** — 이 환경이 글로벌 rules 디렉토리를 안 읽으면 규칙이 조용히 무시된다. → 글로벌 `CLAUDE.md`에서 `@import`로 명시 연결 + 1단계 직후 **다른 폴더에서 실제 로드되는지 검증**(아래 검증 절차) 필수.
2. **중복 잔존 = drift 재발** — 글로벌로 올린 규칙을 REVERB 레포에서 안 지우면 같은 내용이 두 곳에 살아 또 어긋난다. 단일화는 **올리기 + 레포에서 빼기**가 한 세트(3단계).
3. **파일명 충돌 시 동작 불명** — 글로벌 `rules/planning.md`와 레포 `rules/planning.md`가 둘 다 있으면 병합인지 덮어쓰기인지 이 환경에서 불확실. → **2층은 파일명을 분리**(글로벌=원칙 슬림 / 레포=도메인 보강, 다른 파일명)하거나, 공통은 레포에서 완전 제거해 충돌을 없앤다.
4. **과잉 일반화로 REVERB 규칙 약화** — 공통으로 올리며 REVERB 배포 세부(`dev`/`main`·`build.sh`·`globalreverb`)를 지우면 REVERB 워크플로가 헐거워진다. → 도메인 세부는 **레포에 반드시 남긴다**.
5. **"동일해야 한다"의 한계** — `interaction.md`의 "약어 풀이·초등학생 눈높이"는 *비개발자 사용자+인플루언서* 종속. jfun 사용자층이 같으면 공통 OK지만, 다른 프로젝트에서 안 맞는 항목이 나오면 글로벌이 아니라 그 레포에서 끄도록 분리 여지를 남긴다.
6. **에이전트는 그대로 공유 불가** — `reverb-reviewer`·`reverb-supabase-expert`는 REVERB 도메인 지식 내장. 글로벌엔 "역할·호출시점 골격"만, 도메인 변형은 레포.

## 규칙 변경 시 적용 범위 판정 게이트 (고문 세션 의무 — 글로벌 session-roles에 명문화)

**절차:**
1. 사용자가 규칙·정의·프로세스를 새로 만들/바꾸자 하면, 고문은 (반대론자 모드로 충돌·경우의수 검증한 뒤) **적용 범위를 추천과 함께 제시**하고 `AskUserQuestion`으로 확인:

   - ② **이 프로젝트만** (레포 `.claude`)
   - ③ **일부만 공통** (원칙은 글로벌, 도메인 세부는 레포 — 가장 흔함)
2. **고문이 먼저 추천**(사용자 분류 부담 경감):
   - 작업방식·소통·문서관리·세션역할 → **공통 후보**
   - 배포·DB·UI·약관·빌드 등 기능 종속 → **프로젝트 한정**
3. **글로벌(공통)행으로 판정되면 추가 의무 3종:**

   - 변경 전 `~/.claude` 백업 — git 비추적이라 되돌릴 안전망 없음
   - 기존 글로벌 규칙과 충돌·중복 점검 (`planning.md` 규칙 B)

**이 게이트 자체가 공통 규칙** → 1단계에서 글로벌 `session-roles`(원칙 슬림판)에 포함되어 모든 프로젝트 고문세션이 따른다.

### 2단계 — 완료 (2026-06-11)

### 4단계 (커맨드) — 시도 후 철회 (2026-06-11)
- 글로벌 중립판 커맨드 2개(`새세션`·`세션종료`)를 `~/.claude/commands/`에 두었으나 **우선순위 오류로 철회**.
- **틀린 가정**: "같은 이름이면 프로젝트 커맨드가 글로벌을 덮어쓴다"고 봤으나, **실측 결과 반대**였다. REVERB 폴더에서 `/세션종료` 호출 시 글로벌 중립판이 떴다(자체 도메인판 아님).
- **확인된 사실**: 슬래시 커맨드 우선순위 = **Enterprise > Personal(글로벌) > Project**. 즉 **글로벌이 프로젝트를 덮어쓴다.** 같은 이름 커맨드는 글로벌이 이겨 프로젝트 자체판을 가린다.
- **부작용**: 글로벌 중립판이 REVERB 자체판(빌드 자동 실행·`reverb-reviewer`·`origin/dev` 기반·운영배포 확인)을 가려 워크플로 디테일 손실.

- **에이전트 골격 글로벌화도 미실행(백로그)**: `reverb-*` 4종은 도메인 지식이 깊어 일반판 효용 낮음.

## 최종 결론 (2026-06-11)

**원래 목표(프로젝트마다 진행방식·소통·문서관리 동일) 달성.** 글로벌 `~/.claude/rules/`에 공통 규칙 6개(`common-*`). 어느 폴더에서든 고문세션 호출·공통 규칙 작동. 적용범위 판정 게이트로 향후 규칙은 자동으로 글로벌/레포 분류. 레포 규칙 중복은 충돌 없이 공존(단일화 강제 안 함).

**커맨드는 공유 안 함**(4단계 철회): 슬래시 커맨드 우선순위가 글로벌 > 프로젝트라, 글로벌에 두면 REVERB 자체판을 가린다. 커맨드는 각 레포에 둔다.

**핵심 교훈**: 글로벌과 프로젝트에 **같은 이름**을 두면 글로벌이 이긴다(규칙·커맨드·스킬 공통). 그래서 규칙은 `common-*` 접두사로 이름을 분리해 공존시켰고, 커맨드는 이름이 같아 충돌 → 철회. 향후 글로벌 공유는 **이름 충돌 회피**가 전제.
