# Qoo10 랭킹 모니터링 — 기술 검증(PoC) 지시서

**작성일:** 2026-05-27
**작성 세션:** 기획/설계
**대상 세션:** 개발 세션 (PoC 실행)
**배경:** 챌린저스의 「랭킹 모니터링」(Qoo10 순위 추이 그래프 + 카테고리 순위 캡처)을 우리도 제공하기 위함. 사용자 결정 = 방식 C(자체 스크래퍼). 단 Qoo10이 이미 자동 접근을 차단(523)하고 있어, **전체 설계 전에 "실제로 수집되는지" 작은 검증부터** 한다.

---

## 0. 챌린저스 수집 방식 (확인됨)
- 화이트큐브(챌린저스) 백엔드 채용공고에 "Python·Go 로 웹 크롤링/스크래핑으로 데이터 수집" 명시 → **공식 API 아님, 자체 크롤링**. 전담 인력(인턴 포함) 운영.
- 시사점: 기술적으로 가능하나 **전담 유지보수가 필요한 영역**이고, **약관·차단 리스크는 우리가 직접 감수**해야 함. 우리는 Qoo10 셀러 캠페인 파트너라 더 민감.

---

## 1. PoC 목적 (이것만 확인하면 됨)

**"가짜 브라우저(Playwright 등)로 Qoo10 일본 랭킹 페이지에 접근해 순위·제품 데이터를 추출할 수 있는가?"**

세부 검증 3가지:
1. **접근 가능성** — 자동 브라우저로 페이지가 열리는가, 아니면 523/Cloudflare 봇 챌린지·CAPTCHA가 뜨는가
2. **추출 가능성** — 열린다면, JavaScript 렌더링 후 DOM 에서 (순위·제품명·가격·리뷰수) 를 안정적으로 뽑을 수 있는가 (구조·선택자 확인)
3. **차단 민감도** — 1회/소수 요청에 바로 차단되는가, 아니면 느린 간격이면 견디는가

## 2. 대상 페이지 (확인된 URL)
- 베스트셀러 전체: `https://www.qoo10.jp/gmkt.inc/Bestsellers/`
- 뷰티 카테고리: `https://www.qoo10.jp/gmkt.inc/BestSellers/Default.aspx?g=2` (`g=` 가 카테고리 그룹)
- 카테고리별 코스메 특집: `https://www.qoo10.jp/gmkt.inc/Special/Special.aspx?sid=339049`
- (챌린저스 화면은 「스킨케어 > 기초화장품」 수준의 세부 카테고리 + 「ランキング(랭킹)」 정렬 탭을 캡처 — 세부 카테고리 파라미터도 PoC 중 파악)

## 3. PoC 방법 (개발 세션)
- 도구: Playwright (실제 브라우저 엔진). **headless=false 로도 시도**(차단 회피가 아니라, 실제 렌더링·차단 여부를 눈으로 확인하기 위함)
- 실제 User-Agent 사용, **접속자를 숨기지 않음**
- **단 1회 수동 실행**, 데이터 **저장하지 않음**(검증 목적). 반복 루프·스케줄 실행 금지
- 막히면(523/CAPTCHA) **우회 시도하지 말고** 그 사실을 결과로 기록
- 요청 간 충분한 지연(예: 페이지당 수 초 이상). 동시 다발 요청 금지

> ⚠️ 이건 "차단을 뚫는" 검증이 아니라 "정중하게 접근했을 때 되는지" 검증이다. 적극적 우회(접속 지문 위장·IP 분산 등)는 PoC 범위 밖이며, 약관·파트너 관계 리스크가 커 별도 승인 없이 진행 금지.

## 4. 성공/실패 기준 → 결과별 다음 행동

| PoC 결과 | 판정 | 다음 행동 |
|---|---|---|
| 접근 OK + 추출 OK + 1회 차단 안 됨 | **GO** | 전체 설계(§6)로 진행. 인프라·스케줄·저장·표시 사양 확정 |
| 접근 OK + 추출 OK + 쉽게 차단됨 | **조건부** | 차단 빈도·조건 파악. "느린 주기(하루 1회 등) + 정중 접근"으로 견디는지 재검증. 안 되면 외부 스크래핑 API(방식 B) 또는 Qoo10 공식 문의로 선회 |
| 접근 자체가 막힘(523/CAPTCHA) | **STOP** | 자체 스크래퍼로는 비현실적. 방식 B(외부 API) 또는 Qoo10 파트너 데이터 문의로 방향 전환을 사용자에게 보고 |

## 5. 약관·법적 리스크 (PoC 단계에서도 유지)
- Qoo10 이용약관 원문 확보가 어려웠음(페이지도 봇 차단). 자동 수집 금지 조항 가능성 높음 → **전체 설계로 넘어가기 전, 약관 원문을 사람이 직접 확인**하는 절차를 GO 판정과 함께 둔다.
- 수집 데이터는 공개 랭킹·가격(공개정보)에 한정. 개인정보·로그인 영역 접근 금지.
- 우리 약관·개인정보처리방침에는 영향 없음(우리 회원 데이터 아님). 단 운영 리스크로 PR 본문·사양서에 기록.

## PoC 검증 결과

**검증일:** 2026-05-27
**검증 세션:** 개발 세션
**도구:** Playwright(실제 크롬 엔진) — 접속자 위장 없음, 우회·반복·저장 없음, 정중하게 1회씩 2개 페이지만 접근
**판정: GO** (사양서 §4 「접근 OK + 추출 OK + 1회 차단 안 됨」)

### 검증 항목별 결과

| 항목 | 결과 | 근거 |
|---|---|---|
| ① 접근 가능성 | **OK** | 베스트셀러 전체(`/gmkt.inc/Bestsellers/`)·뷰티 카테고리(`/gmkt.inc/BestSellers/Default.aspx?g=2`) 둘 다 정상 로드. 차단 페이지(523)·Cloudflare 봇 검사·CAPTCHA 전혀 미발생. 본문 약 24,000자 정상 렌더링 |
| ② 추출 가능성 | **OK** | 한 상품 카드에서 순위·브랜드·제품명·리뷰수·정가/할인가·배송구분·상품링크 전부 안정적 추출. 순위 마커는 `<span class="rank">N</span>` 형태로 명확. 카테고리별 1~200위 전체 수집 가능 |
| ③ 차단 민감도 | **단기 차단 없음** | 같은 세션 연속 2페이지 접근에도 차단 안 됨. 단 정기·반복 수집 시의 장기 차단 여부는 1회 검증으로 단정 불가 |

### 추출 구조 (확인된 선택자)
- 순위: `.rank` (= `<span class="rank">N</span>`), 페이지당 정확히 200개
- 제품명: 상품 카드 내 `img[alt]` (전체 제품명 보존)
- 브랜드: 카드 텍스트의 `公式 {브랜드}` 패턴
- 리뷰수: 카드 텍스트의 `(숫자)` 패턴 (예: `(8,547)`)
- 가격: 카드 내 `円` 텍스트 (정가+할인가 연속, 예: `1,600円599円` → 파싱 분리 필요)
- 상품 URL: 카드 내 `a[href]` → `https://www.qoo10.jp/item/...`

### 캡처 증빙
- Playwright 스크린샷으로 `ランキング200 (総合)` 헤더 + 순위 카드가 그대로 캡처됨 → 챌린저스식 "카테고리 순위 캡처 이미지" 리포트 제작 가능 확인

### 전체 설계로 넘기는 전제 (기획 세션이 §6 상세화 시 반영)
1. **Qoo10 약관 원문을 사람이 직접 확인** — 자동 수집 금지 조항 가능성. 우리는 Qoo10 셀러 캠페인 파트너라 운영 리스크 더 민감.
2. **차단 민감도는 "단기 안전"까지만 확인됨** — 정기 수집(스케줄) 시 차단이 나타날 수 있으므로 인프라는 **저빈도(하루 1회 수준) + 정중 접근**을 기본 전제로 설계. 차단 실제 발생 시 외부 스크래핑 서비스(방식 B) 선회 분기 유지.
3. 가격 문자열은 정가/할인가가 붙어 나오므로 저장 모델에서 분리 파싱 규칙 필요.
4. 광고주용 성과 리포트(권고 G)와의 연계: 캠페인 제품을 상품 URL 기준으로 랭킹 시계열에 매칭 가능.
