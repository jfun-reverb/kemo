# 캠페인 상태 안내 개선 + 관리자 오류 메시지 한글화 (+ 종료 캠페인 편집 버그)

**작성일:** 2026-06-04
**작성 세션:** 기획/설계
**상태:** 초안 (사용자 정책 확정, 개발 미착수) — ⚠️ PR 1 은 운영 버그 수정(우선)

---

## 배경 (사용자 요청)

1. 「캠페인 상태별 클라이언트 노출 안내」 모달을 **더 알아보기 쉽게** 정리: ① 상태를 관리자 화면 한글 표기로 ② "전이" 용어를 초등학생도 알 만한 쉬운 말로 ③ **어떤 상태에서 무엇을 수정 가능/불가**인지 추가
2. 관리자 캠페인 편집 「변경 저장」 시 영어 오류 노출:
   `저장 오류: new row for relation "campaigns" violates check constraint "campaigns_status_check" [ERR_UNHANDLED]`
3. "관리자 페이지에서 오류 메시지는 한글로 알려주면 좋겠다"

→ 조사 결과 2번은 **단순 한글화 누락이 아니라 실제 버그**(종료 상태 캠페인 편집 시 빈 상태값 저장 시도)로 판명.

**사용자 확정 정책:**
- 미분류 오류(한글 매핑에 없는 것) = **일반 한글 안내 + 식별코드만**, 영어 원문은 화면에서 숨기고 오류 로그에만 기록

## 현재 상태 (작성일 2026-06-04 기준 — planning.md 규칙 A)

### 캠페인 상태 (6단계) — 실제 한글 라벨
`dev/lib/shared.js` 626~643 `campaignStatusLabelKey`/`CAMPAIGN_STATUS_LABEL`:
| status | 한글 라벨 | 일본어 | 의미 |
|---|---|---|---|
| draft | 준비 | 準備中 | 작성·검토 중 |
| scheduled | 모집예정 | 近日公開 | 모집 시작 전 |
| active | 모집중 | 募集中 | 모집 진행 중 |
| closed | 모집마감 | 募集締切 | 모집 종료·결과물 받는 중 (submission_end 경과 시 라벨이 「종료」로 표시) |
| ended | 종료 | 終了 | 결과물까지 완료 |
| expired | 노출종료 | 掲載終了 | 운영자가 노출 OFF |

### 자동 상태 변경 (모달이 "전이"라 부르는 것) — `dev/lib/storage.js` 125~214
- `autoOpenCampaigns` recruit_start 도래 → scheduled→active
- `autoCloseCampaigns` deadline 경과 → active→closed
- `autoEndCampaigns` submission_end 경과 → closed→ended
- expired 는 `toggleCampaignVisibility` OFF 만 (자동 없음)

### 상태별 수정 제약 (실측 — 안내 ③에 들어갈 내용)
- **DB 트리거**(마이그레이션 075·108·156 `block_closed_campaign_caution_participation_update`): `closed`·`ended`·`expired` 에서 **주의사항(caution)·참여방법(participation)·NG(ng) 변경 차단**. 날짜 컬럼은 차단 안 함.
- **클라 검증**(`dev/js/admin.js` `validateCampDateRanges` 1137·`saveCampaignEdit` 1697): 날짜는 **순서·정합성**만 검증(모집마감>시작, 제출마감>구매/방문 종료, 구매·방문은 모집~제출 윈도우 내). `active`/`scheduled` 로 저장 시 deadline 경과면 차단.
- 결론: **날짜류는 거의 모든 상태에서 수정 가능**(정합성만 맞으면). 막히는 건 ①모집마감 이후 주의사항·참여방법·NG ②마감 지난 캠페인을 모집중/모집예정으로 되돌리기 ③날짜 정합성 위반.

### 오류 한글화 구조 — `dev/js/admin-core.js` 99~115 `friendlyError`
- 13개 패턴 한글 매핑(23505·42501·23503·23502·네트워크·429·404·408·401 등)
- ❌ **23514(check_violation)·P0001(트리거 RAISE) 매핑 없음** → 영어 원문 + `[ERR_UNHANDLED]` 노출
- DB 트리거가 던지는 **한글 메시지**(「모집마감 캠페인의 주의사항은…」)도 매핑 안 돼 `[ERR_UNHANDLED]` 가 붙음
- raw 노출 호출부(friendlyError 미경유): `admin-accounts.js`(220), `admin-notices.js`(163·178·314·325), `admin-influencers.js`(664·677·751·817·834·847), `admin-lookups.js`(250·267·283·300·800), `admin.js`(3551·3569·3593)
- 인플루언서용은 별도 `friendlyErrorJa`(`dev/js/ui.js` 115). client_error_logs(마이그레이션 165) 에 원문 기록 경로 존재

### 🐞 버그 — `campaigns_status_check` 위반의 진짜 원인
- 편집 폼 상태 드롭다운 `editCampStatus`(`dev/admin/index.html` 349~351): **옵션이 draft/scheduled/active/closed/expired 5개뿐, `ended` 없음 + `closed` 라벨이 "종료"로 오기**(실제 closed=모집마감).
- `campaigns_status_check` 는 ended 포함 6개 허용. **운영 DB 확인 완료(2026-06-04)**: `CHECK (status = ANY (ARRAY['draft','scheduled','active','closed','ended','expired']))` — DB 정상, ended 허용됨.
- **확정 원인**: 자동으로 `ended` 가 된 캠페인을 편집 → 드롭다운에 ended 옵션이 없어 select 값이 **빈 문자열**로 저장 시도 → `status=''` 가 check 위반 → 영어 오류. **DB 문제 아니라 프론트 드롭다운 누락이 원인.**
- **영향**: 종료된 캠페인은 현재 편집·저장 불가(날짜만 바꾸려 해도 status='' 로 통째 막힘 — 사용자가 "날짜 안 고쳐짐"으로 체감). 운영 영향 범위 = 운영 `status='ended'` 캠페인 수(첫 진단 쿼리로 확인 권고).

## 의심·경우의 수 (planning.md 규칙 B)

1. **(버그 핵심) ended 드롭다운 처리** — 단순히 `ended` 옵션을 추가하면 관리자가 **임의로 ended/closed 상태를 직접 바꿀 수 있게** 됨(현재 status 드롭다운은 값 그대로 UPDATE). 자동 전이로만 되던 상태를 사람이 손으로 바꾸는 부작용. → 두 방향: (가) 옵션 추가하되 deadline 검증처럼 가드 / (나) 종료/모집마감 상태일 땐 드롭다운을 **현재 상태 표시·고정**(임의 변경 차단)하고 빈값 저장만 방지. 개발/planner 정밀 판단 필요.
2. **(데이터) 운영 ended 캠페인 수** — 0건이면 잠복 버그(아직 안 터짐), 다수면 이미 편집 불가 상태. 운영 진단 SQL 선행.
3. **(UX) 안내 모달 정보량** — 노출+자동변경+수정제약을 한 모달에 다 넣으면 복잡. 초등학생 눈높이 위해 **3섹션 명확 구분 + 표 간결**. 제목도 「노출 안내」→「상태 안내(노출·자동변경·수정 가능 여부)」로 확장.
4. **(UX) 오류 한글화 일관성** — 미분류 오류를 일반 안내로 덮으면 운영자가 원인 추적 불가 → 식별코드 + client_error_logs 원문 기록 병행 필수.
5. **(기술) DB 한글 메시지 통과** — 트리거가 던지는 한글 메시지(주의사항 잠금 등)는 그대로 보여주는 게 친절. friendlyError 가 "한글이면 통과" 분기 필요(영어 원문만 일반안내로 치환).
6. **(권한·환경) friendlyError 적용 범위** — raw 노출 호출부 일괄 교체 시 회귀 위험 낮으나, 각 호출부의 prefix("저장 오류:" 등) 유지하며 메시지부만 friendlyError 통과.

### 현재 구현 충돌
- 안내 모달은 ended·NG 차단·closed→ended 자동변경을 누락 중 → 재작성으로 해소.
- friendlyError 보강은 추가형이라 충돌 없음. 단 미분류→일반안내 전환 시 기존에 영어라도 보던 정보가 사라지므로 로그 기록 필수.

### 의도 모호점
- 버그 수정 방식 1번(가/나) — 사용자 확인 또는 개발/planner 위임.

### PR 2 — 관리자 오류 메시지 한글화

- 미분류 오류 발생 시 **원문을 client_error_logs/console 에 기록**(운영자 추적용).
- raw 노출 호출부(위 5개 파일) 일괄 `friendlyError()` 경유로 교체(prefix 유지).
- reverb-reviewer + qa Light.

### PR 3 — 캠페인 상태 안내 모달 재작성
모달(`dev/admin/index.html` `campStatusHelpModal` 2220~2288) 제목·내용 교체. **확정 콘텐츠**:

**제목:** 「캠페인 상태 안내 — 노출·자동 변경·수정 가능 여부」

**① 상태와 인플루언서 노출**
| 상태 | 뜻 | 인플루언서에게 보이나요 |
|---|---|---|
| 준비 | 아직 만드는 중 | 안 보임 |
| 모집예정 | 곧 모집 시작 | 보임 (近日公開) |
| 모집중 | 지금 모집 중 | 보임 (募集中) |
| 모집마감 | 모집 끝·결과물 받는 중 | 보임 (募集締切), 끌 때까지 |
| 종료 | 결과물까지 다 끝남 | 보임 (終了), 끌 때까지 |
| 노출종료 | 운영자가 화면에서 내림 | 안 보임 |

**② 상태가 저절로 바뀌는 경우** ("전이"→"저절로 바뀜")
- 모집 시작일이 되면 → 모집예정에서 **모집중**으로 저절로
- 모집 마감일이 지나면 → 모집중에서 **모집마감**으로 저절로
- 결과물 제출 마감일이 지나면 → 모집마감에서 **종료**로 저절로
- **노출종료**는 저절로 안 됨 — 운영자가 「캠페인 노출」을 꺼야만

**③ 상태에 따라 고칠 수 있는 것 / 없는 것**
| 무엇을 | 준비·모집예정·모집중 | 모집마감·종료·노출종료 |
|---|---|---|
| 모집 기간(시작·마감일) | 고칠 수 있음 | 고칠 수 있음 |
| 결과물 제출 마감일 | 고칠 수 있음 | 고칠 수 있음 |
| 구매·방문 기간 | 고칠 수 있음 | 고칠 수 있음 |
| 주의사항·참여방법·NG사항 | 고칠 수 있음* | **못 고침** |

\* 신청자가 1명이라도 있으면 "정말 바꿀까요?" 확인창이 뜸.
\* 「모집중」·「모집예정」으로 저장하려면 모집 마감일이 아직 안 지났어야 함.
\* 날짜는 서로 순서가 맞아야 저장됨(모집마감>시작, 제출마감>구매·방문 종료 등).

→ 핵심 한 줄: **날짜는 거의 다 고칠 수 있고, 모집마감 이후엔 주의사항·참여방법·NG만 잠긴다.**
