## 2. 데이터베이스 영향

### 2-1. `brand_applications.products` jsonb 스키마 확장

`products` 컬럼은 jsonb 배열이라 **마이그레이션 없이 키 추가 가능**. 행 1건 구조 (제안):

```jsonc
{
  "name": "립프롬 신상 마스카라",     // 기존
  "url": "https://...",                // 기존 (선택)
  "unit_price_jpy": 1980,              // 기존 (단가)
  "qty": 30,                           // 기존 (수량)
  "recruit_fee_krw": 50000,            // 신규 — 행별 모집비 (₩, 선택, 빈/null=0)
  "status": "pending"                  // 기존(2026-04-30 100/101 마이그레이션)
}
```

키 이름 `recruit_fee_krw`는 제안 — 개발1 신규 등록 모달 구현과 일관성을 맞추기 위해 **개발1과 키 이름 합의 후 확정**.

### 2-2. `submit_brand_application` RPC

- anon 익명 신청 RPC. 인자로 `payload jsonb` 받음
- 모집비도 `payload.products[].recruit_fee_krw`로 자연스럽게 통과 — RPC 시그니처 변경 불요
- 단 RPC 안 검증 로직이 «음수 차단·최대값 제한» 등 강제하면 그 검증에 모집비 항목도 추가 (선택)

### 2-3. 견적 자동 계산 트리거 (영향 확인 필요)

- 현재 `brand_applications`에 INSERT/UPDATE 시 `total_jpy`·`total_qty`·`estimated_krw`를 자동 계산하는 트리거가 있는지 작업 시작 시 확인 필요 (CLAUDE.md 「products 재계산」 언급 있음)
- 트리거가 있다면 **`estimated_krw` 계산식에 `sum(products[].recruit_fee_krw)` 합산** 추가 마이그레이션 1건
- 트리거가 없고 클라이언트에서 계산만 한다면 sales 폼·관리자 모달의 합계 식만 수정 (개발1 합의 식 따름)
- 마이그레이션이 필요한 경우 파일명: `supabase/migrations/{다음번호}_brand_app_recruit_fee_estimate.sql`

### 2-4. RLS 변경 없음

- `products` jsonb 안 키 추가는 기존 RLS 정책에 영향 없음
- `submit_brand_application` RPC SECURITY DEFINER 그대로

## 4. 관리자 페이지 영향

### 4-1. 「신규 등록 모달」 (개발1이 진행 중)

- 본 사양의 모집비 열을 개발1 작업에 **합류시켜 한 PR에서 처리**
- 개발1과 합의해야 할 항목:
  - jsonb 키 이름 (`recruit_fee_krw` 제안)
  - 행별 합계 계산식 (사용자가 개발1에 별도 전달했다고 명시 — 정확한 식은 개발1 결정 따름)
  - 모집비 열 표시 순서·라벨 한국어 표기
- 일감 인수 시점에 개발1의 진행 상태(파일 위치·함수 이름) 점검 후 모집비 패치 추가

### 4-2. 「수정 모달」 (`/admin#brand-applications` 상세 모달)

- 제품 테이블에 모집비 열 추가 (신규 등록 모달과 동일 구조)
- 행별 편집 가능 — 기존 단가·수량 편집 패턴 그대로
- 저장 시 `version` 낙관적 락 그대로 사용
- 최종 견적(`final_quote_krw`) 수동 입력 필드는 유지 — 관리자가 모집비 포함 여부와 무관하게 협의 결과를 직접 입력 가능
- 행별 합계·전체 합계 표시는 신규 등록 모달과 동일 식 사용

### 4-3. 「브랜드 서베이 현황 대시보드」 (`/admin#brand-dashboard`)

- 견적 합계 KPI(예상·확정)는 기존 `estimated_krw`·`final_quote_krw` 합산 — 모집비가 §2-3 트리거 갱신으로 자동 반영되면 별도 코드 변경 없음
- 별도 「모집비 합계」 KPI 신설은 본 사양 범위 외 — 추후 필요 시 후속 사양

### 4-4. 엑셀 내보내기

- 「브랜드 서베이 엑셀 내보내기」(2026-04-22) 컬럼에 「모집비 합계(₩)」 1열 추가 (선택 — 운영팀 활용 가치 있을 시)

## 7. QA 시나리오 (개발서버 → reverb-qa-tester)

1. sales 페이지 reviewer 폼 — 모집비 빈 값으로 신청 → 정상 접수 + jsonb 안 `recruit_fee_krw` null 또는 0
2. sales 페이지 seeding 폼 — 제품 3행 중 1행만 모집비 입력 → 정상 접수 + 그 행만 값 채워짐
3. 관리자 페이지 수정 모달 — 모집비 열 편집 후 저장 → 정상 반영 + version 충돌 차단 그대로 작동
4. 관리자 페이지 신규 등록 모달(개발1 작업) — 모집비 열 입력 후 저장 → 정상 INSERT
5. 운영팀 알림 메일 수신 → 모집비 열·합계 정상 노출
6. 광고주 확인 메일(reviewer/seeding) 수신 → 모집비 열 정상 노출
7. 견적 합계(`estimated_krw`)에 모집비 합계 자동 합산 확인 (§2-3 트리거 갱신 후)
8. 행별 합계 계산식 — 개발1 합의 식대로 정상 작동 (sales 폼·관리자 양쪽)
9. 음수·문자 입력 차단 확인
10. 브랜드 서베이 엑셀 내보내기 — 모집비 합계 컬럼 정상 출력 (선택 구현 시)

## 9. 충돌 점검

### 9-1. 개발1 신규 등록 모달 작업

- 합의 항목: jsonb 키 이름 / 행별 합계 식 / 라벨 한국어 표기

### 9-2. 다른 일감과의 영역
- 응모 취소·관리자 메일 수신 분리·NG 사항 번들화 — 본 사양과 다른 파일 영역. 충돌 없음

### 9-3. Edge Function 동시 작업 점검
- `notify-brand-application` Edge Function은 admin-email-subs 작업에서도 수신자 쿼리 교체로 손댐 (그 작업 완료된 가정). 본 사양은 메일 템플릿 HTML만 갱신 — 영역 분리 명확
