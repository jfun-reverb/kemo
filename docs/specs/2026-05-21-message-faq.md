## 0. 범위

- **포함:**
  - 응모건의 **실제 상태를 보여주는 개인화 한 줄**(심사중/당첨/구매기간/제출기한/반려 등)
  - 선택지 트리형 FAQ — 관리자가 질문·답변을 한국어·일본어로 직접 등록
  - 응모 단계에 맞춘 **맞춤 FAQ 우선 노출**
  - 답변에 앱 화면 이동 버튼 + "직접 문의" 탈출구
  - 최소 측정(어떤 FAQ가 조회됐고 결국 사람에게 갔는지)
  - **관리자 응대 보조: 인플루언서가 본 FAQ 열람 이력 + 응모건 상태를 관리자 메시지 화면에 표시**(§3-1·§3-2)
- **제외 (별도 검토):** 한국어↔일본어 번역 (개인정보 외부 전송·국외이전·약관 개정이 얽혀 추후 신중 검토)
- **제외 (별도 일감):** 응모 취소 오류 버그 — 개발 세션에서 별도 진행 중

## 1. 배경 — 실측 분석 결론

실제 LINE 대화 7건 분석 결과:
1. **문의 대부분이 이미 앱에 있는 기능을 LINE으로 하던 것** — 결과물(투고) 제출, SNS 등록, 심사 상태 확인이 상위
2. **상위 문의의 본질이 "내 신청이 지금 어떤 상태인가"** — 고정 답변으로는 해소 불가, 개인 상태를 보여줘야 함
3. 정형 사용법은 FAQ로 흡수, 기능 결함·예외는 사람 연결

### 1-1. 문의 3분류 (설계의 출발점)

| 유형 | 예시 | 해법 |
|---|---|---|
| **A. 사용법 안내** | "결과물 어떻게 올려요", "SNS 어디서 등록해요" | 고정 FAQ 답변 + 해당 화면 이동 버튼 |
| **B. 내 상태 확인** | "심사 언제 끝나요", "당첨됐나요", "배송 언제" | **응모건의 실제 상태를 개인화 한 줄로 표시** + 단계별 맞춤 FAQ |
| **C. 기능 결함·예외** | "취소가 에러로 안 돼요", "상품이 누락돼 왔어요" | 기능 수정(별도) + 빠른 사람 연결 |

> 가장 빈번한 문의가 B이므로, **개인화 상태 표시가 이 기능의 핵심 가치**다. 고정 FAQ(A)는 보조.

## 3. 개인화 상태 한 줄 — 매핑 설계 (B 문의 해소의 알맹이)

응모건의 `applications.status` + 캠페인 일정(`recruit/purchase/visit/submission` 기간)을 조합해 상태 문구를 도출한다. **클라이언트에서 이미 가진 데이터(`_myApps`, `allCampaigns`, `_myDelivsByApp`)로 계산 가능** → 신규 서버 호출 불필요.

| 응모 상태 / 단계 | 상태 한 줄 (일본어 기준, 한국어 병기) | 맞춤 FAQ 우선 노출 |
|---|---|---|
| pending (심사중) | 「現在、審査中です。結果は別途ご連絡します」 (심사 중) | 심사 기간·결과 통보 방법 |
| approved + 구매기간 전 | 「当選されました！まもなく購入のご案内が始まります」 (당첨) | 진행 안내·일정 |
| approved + 구매기간 중 (monitor) | 「商品ご購入後、レシートをご提出ください（提出期限 MM/DD）」 (영수증 제출) | 영수증 제출 방법·필수항목 |
| approved + 방문기간 중 (visit) | 「ご訪問後、投稿をご提出ください」 (방문 후 제출) | 방문 방법·게시물 제출 |
| approved + 제출 단계 | 「成果物の提出期限：MM/DD」 (제출 기한) | 게시물 URL 등록·기한 |
| 결과물 제출됨·검수중 | 「ご提出いただいた成果物を確認中です」 (확인 중) | 검수 소요·승인 통보 |
| 결과물 일부만 반려 (monitor 등 2종) | 「一部の成果物が差し戻されました。差し戻し分をご確認の上、再提出ください」 (일부 반려·재제출) | 재제출 방법 (+ 반려된 결과물 사유 표시) |
| 결과물 전부 반려 | 「成果物が差し戻されました。理由をご確認の上、再提出ください」 (반려·재제출) | 재제출 방법 (+ 사유 표시) |
| 결과물 전부 승인 | 「全てのミッションが完了しました。ありがとうございました」 (완료) | (없음) |
| rejected (비승인) | 「今回はご縁がありませんでした」 (비선정) | 다음 기회·재신청 |
| cancelled (취소됨) | 「取り消された応募です」 (취소됨) | 재신청 가능 여부 |

- 상태 한 줄 옆에 **관련 화면 바로가기 버튼**(예: 반려 → "결과물 제출 화면", 심사중 → "응모이력")
- 문구는 `dev/js/messaging.js`에 상태→문구 매핑 함수로 구현(i18n 키 사용). 캠페인 일정 비교는 기존 `_computeCancelPhase` 패턴 재사용

### 3-0. 상태 한 줄 판정 우선순위 (구현 필수)

> **추가 결정 (2026-05-21, 검토 보강).** 한 응모건이 위 매핑의 여러 행에 동시 해당할 수 있다(예: 「구매기간 중」인데 이미 영수증을 제출함). 구현 함수는 **반드시 아래 순서로 판정**해, 위에서 먼저 맞는 행에서 멈춘다.

1. `status == cancelled` → 취소됨
2. `status == rejected` → 비선정
3. `status == pending` → 심사 중
4. `status == approved` 이면 **결과물(`_myDelivsByApp`) 상태를 먼저 본다** (일정보다 우선):
   - 결과물이 1건 이상 존재하고 **전부 approved** → 완료
   - 결과물 중 **일부만 rejected** (다른 건 pending/approved) → 일부 반려·재제출
   - 결과물 **전부 rejected** → 반려·재제출
   - 결과물이 있고 **pending(검수 대기) 포함, rejected 없음** → 확인 중
5. 결과물이 **아직 없으면** 캠페인 일정으로 판정:
   - 구매기간 전 → 당첨(안내 대기)
   - 구매기간 중(monitor) → 영수증 제출
   - 방문기간 중(visit) → 방문 후 제출
   - 제출 단계(구매/방문 후 ~ `submission_end`) → 제출 기한
6. 어디에도 안 맞으면(일정 데이터 누락 등) → **무난한 기본 문구로 폴백**(§12)

- "일부만 반려"는 monitor처럼 결과물이 2종(영수증 + 리뷰 이미지)인 캠페인에서 발생. 단일 결과물 캠페인은 "전부 반려"와 동일.
- 결과물 상태를 일정보다 먼저 보는 이유: 인플루언서가 이미 행동(제출)했다면, 그 다음 할 일(재제출/대기)을 안내하는 게 정확하기 때문.

### 3-1. 관리자 측에도 같은 상태 표시 (응대 보조)

> **추가 결정 (2026-05-21, 실사례 기반 — 부록 B 참조).** 인플루언서가 "내가 지금 어떤 단계인지" 몰라 문의하면, 관리자도 답하기 전에 그 사람 응모건을 일일이 조회해야 한다. **인플루언서 화면에 띄우는 개인화 상태 한 줄을, 관리자가 답하는 메시지 화면에도 그대로** 띄워 응대를 돕는다.

- 위치: 관리자 메시지 화면(`#adminPane-messages` 스레드 영역 + 응모행 「메시지」 모달, `dev/js/admin-messaging.js`)
- 표시 내용: 해당 응모건의 **캠페인 모집 타입**(리뷰어=영수증 / 무료제공·방문형=게시물 URL) + **현재 단계 상태 한 줄**(§3 매핑) + 결과물 제출 여부
- 언어: 관리자 화면이므로 **한국어** 표기 (§3 매핑의 한국어 병기를 사용)
- 효과: "이 사람 지금 영수증 단계인지 게시물 URL 단계인지"를 관리자가 조회 없이 즉시 파악 → 부록 B 같은 "구매형/무료제공형 혼란" 문의에 정확·신속 응대
- 데이터: 인플루언서 측과 동일 계산. 단 관리자 화면은 `_myApps`가 없으므로 응모건 조회 시 캠페인 일정·타입을 함께 가져와 계산(기존 admin-messaging 응모건 로딩에 컬럼 추가)
- **⚠️ 결과물 상태도 함께 조회 필수 (2차 검토 보강):** §3-0 판정 순서는 결과물(`deliverables`) 상태를 캠페인 일정보다 먼저 본다. 관리자 측에서도 그 응모건의 결과물 상태(전부 승인/일부 반려/검수중 등)를 함께 가져오지 않으면 인플루언서 화면과 **다른 상태 한 줄**이 나온다(§10 PR B2 검증의 "동일 결과 대조"가 깨짐). admin-messaging 응모건 로딩 쿼리에 캠페인 일정·타입 + **결과물 상태 집계**를 함께 추가.

### 3-2. 관리자에게 "인플루언서가 본 FAQ 이력" 표시 (응대 맥락)

> **추가 결정 (2026-05-21, 사용자 요청).** 인플루언서가 어떤 질문을 선택하고 어떤 자동 답변을 받았는지를 관리자가 볼 수 있으면, **직접 문의로 넘어온 뒤 추가 상담**할 때 "이미 무슨 안내를 받았는지" 알고 시작할 수 있다. 같은 안내를 반복하지 않고, 막힌 지점부터 이어서 응대 가능.

- **데이터 출처:** `faq_interactions`(§4-2). 이미 `influencer_id`·`application_id`·`faq_node_id`·`action`·`created_at`을 기록하므로 **테이블 변경 없이** 활용 가능.
- **표시 위치:** 관리자 메시지 화면(`#adminPane-messages` 스레드 + 응모행 메시지 모달, `dev/js/admin-messaging.js`)에 **「FAQ 열람 이력」 접이식 패널**.
- **표시 내용 (시간순):** 「언제 / 어떤 질문(제목) / 받은 답변 요약 / 결과(봤음·해결됨·직접문의로 전환)」.
  - 답변 본문은 기록된 `faq_node_id`로 현재 노드에서 역참조해 표시(질문 제목 + 답변).
  - ⚠️ FAQ 답변이 나중에 수정되면 과거 열람 시점과 달라질 수 있음 → 패널에 「현재 등록된 답변 기준」 안내 문구. 시점 스냅샷 보존은 v2(필요 시 `faq_interactions`에 답변 텍스트 스냅샷 컬럼 추가).
- **직접 문의(handoff) 강조:** 인플루언서가 FAQ를 거쳐 직접 문의로 넘어온 경우, 그 직접 문의 메시지 **바로 위에 "직전 열람: [질문 제목]" 컨텍스트 칩**을 관리자 화면에만 표시 → 관리자가 "이 사람 OO 안내를 봤는데도 안 풀려서 물어본 것"을 즉시 파악.
- **언어:** 관리자 화면이므로 질문 제목·답변은 **한국어(`label_ko`/`body_ko`)** 우선 표시(인플루언서가 본 언어가 일본어여도 관리자는 한국어로 이해).
- **개인정보:** FAQ 열람 이력은 인플루언서 본인의 앱 내 행동 로그(외부 전송·신규 수집 없음). 관리자 SELECT는 이미 `is_admin()` 한정.

### 3-3. 상태 ↔ `relevant_stages` 태그 매핑 (단계 태깅 정합성)

> §3 매핑(노출용 문구, 10케이스)과 `faq_nodes.relevant_stages`(맞춤 노출 태그, §4-1)를 **1:1로 연결**한다. 구현·운영 등록 시 이 표를 기준으로 단계를 태깅한다.

| §3 상태 케이스 | `relevant_stages` 태그값 |
|---|---|
| 심사 중 | `pending` |
| 당첨(구매기간 전) | `approved_purchase` (또는 visit형이면 `approved_visit`) |
| 영수증 제출(구매기간 중) | `approved_purchase` |
| 방문 후 제출(방문기간 중) | `approved_visit` |
| 제출 기한 | `approved_post` |
| 확인 중(검수) | `approved_post` |
| 일부 반려 / 전부 반려 | `approved_post` |
| 완료 | `done` |
| 비선정 | `rejected` |
| 취소됨 | (태그 없음 — 맞춤 노출 대상 아님) |

- enum 확정값: `pending` / `approved_purchase` / `approved_visit` / `approved_post` / `rejected` / `done` (6종). `NULL` = 모든 단계 노출.
- 답변 문서([`message-faq-answers.md`](../research/2026-05-21-message-faq-answers.md))의 각 질문 "단계" 칸도 이 6종으로 통일.

### 3-4. "왜 안 됐는지" 사유 표시 정책 (3구분 — 2026-05-21 결정)

> 인플루언서가 "왜 거절/반려됐나"를 물을 때, **거절의 성격에 따라 시스템이 줄 수 있는 정보가 다르다.** 세 가지를 구분해 처리한다.

| 구분 | 사유 데이터 | 정책·처리 | 답변 문서 |
|---|---|---|---|
| **① 응모 비승인** (심사 탈락, `applications.status=rejected`) | **없음** (저장 칸 없음, 종합·정성 판단) | **구체 사유 미공개 + 완곡 답변.** 비선정자 개별 미통보가 현 정책(캠페인 `notice6`) | Q2-3 |
| **② 결과물·영수증 반려** (`deliverables.status=rejected`, `reject_reason` 있음) | **있음** | **실제 사유를 상태 한 줄·메시지에 동적 표시** + "자세한 내용은 「成果物提出」 화면에서 확인 후 재제출" 유도 | Q3-5 + §3 상태 한 줄 |
| **③ 응모 차단** (응모 버튼이 막힘, 아직 응모 전) | **있음** (조건·본인 값 모두 클라이언트에 존재) | **모집조건 대비 미달을 구체 수치로 안내.** 팔로워 「필요 {required}명 / 현재 {current}명」, 필수정보 미등록 항목 명시 | Q1-1-a~e (특히 Q1-1-c) |

- **핵심 통찰:** "왜 안 되지?"의 진짜 수요는 ①(비승인)이 아니라 ③(응모 차단)이다. 응모가 됐다 = 정량 조건을 이미 통과했다는 뜻 → 비승인된 사람은 대개 미달 항목이 없다. 그래서 미달 안내는 ③ 시점에서 구체화한다.
- **② 반려 사유 동적 표시 구현:** §3 상태 한 줄의 「일부 반려」/「전부 반려」 케이스에 해당 응모건의 `deliverables.reject_reason`을 끼워 표시. 사유가 길거나 여러 건이면 요약 + "화면에서 전체 확인" 버튼. `reject_reason`은 `lookup_values(kind='reject_reason')` 코드일 수도, 자유 입력일 수도 있으므로 **노출 시 코드→일본어 라벨 변환**(없으면 자유 텍스트 그대로) + `esc()` 이스케이프 필수.
- **③ 응모 차단 구체 안내 구현:** 캠페인 `min_followers` vs 본인 대표 SNS 팔로워, 필수정보 미등록 항목을 응모 차단 알럿/메시지에서 치환. 신규 서버 호출 불필요(클라이언트 데이터).

### 4-2. 측정 테이블 `faq_interactions` (최소)

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid PK | |
| `influencer_id` | uuid | 본인 |
| `application_id` | uuid NULL | 어떤 응모건에서 |
| `faq_node_id` | uuid NULL (ON DELETE SET NULL) | 본 질문 (handoff 는 NULL 가능) |
| `action` | text CHECK(`viewed`\|`resolved`\|`handoff`) | 봤다 / 해결됐다 / 직접문의로 갔다 |
| `view_count` | int DEFAULT 1 | `viewed` 멱등 누적 횟수 (resolved/handoff 는 의미 없음) |
| `created_at` | timestamptz | 최초 발생 시각 (viewed 는 최초 열람) |
| `last_viewed_at` | timestamptz NULL | `viewed` 마지막 갱신 시각 |

- **RLS:** INSERT `authenticated`(본인 행), SELECT `is_admin()`
- 용도 ①(개선 진단): "조회 많은데 handoff(직접문의 전환)도 많은 질문 = 답변이 부실" → 운영자가 답변 개선
- 용도 ②(응대 맥락, §3-2): 관리자 메시지 화면에서 **이 응모건/인플루언서가 본 FAQ 이력**을 시간순 표시 → 추가 상담 시 "이미 받은 안내" 파악. `application_id`로 응모건 단위 필터
- 과도 수집 방지: action 3종만, 본문·식별정보 미저장 (답변 텍스트는 `faq_node_id` 역참조)

#### 중복 증가 방지 (구현 정책 — 검토 보강 2026-05-21)

> `viewed`를 매 열람마다 그대로 INSERT하면 같은 사람·같은 질문에 행이 무한히 쌓여, §3-2 관리자 열람 이력 패널이 같은 질문으로 도배된다. 아래 정책으로 억제한다.

- **`viewed`는 (influencer_id, application_id, faq_node_id) 단위로 멱등(UPSERT)** — 같은 응모건에서 같은 질문을 다시 봐도 행을 새로 만들지 않고 `last_viewed_at`만 갱신.
  - 이를 위해 `faq_interactions`에 `view_count int DEFAULT 1` + `last_viewed_at timestamptz` 컬럼을 두고, `created_at`은 최초 열람 시각으로 보존.
  - 부분 유니크 제약: `(influencer_id, application_id, faq_node_id) WHERE action='viewed'`.
  - **⚠️ 기록은 원격 호출 함수(RPC)로 감쌀 것 (2차 검토 보강):** 행 단위 보안 정책(RLS)이 "본인 행 INSERT"만 허용하면 충돌 시 갱신(`ON CONFLICT DO UPDATE`)이 막힌다(UPDATE 정책 부재). 익명 폼 INSERT를 RPC로 감싸는 프로젝트 규칙([`.claude/rules/supabase.md`])과 동일하게, `record_faq_interaction(application_id, faq_node_id, action)` 같은 `SECURITY DEFINER` + `SET search_path=''` 함수로 감싸 멱등 UPSERT를 처리한다. 함수 내부에서 `influencer_id = auth.uid()` 강제 → 본인 행만 기록(변조 차단).
- **`resolved`/`handoff`는 그대로 append** — "해결됐다/사람에게 갔다"는 이벤트성이라 횟수가 의미 있음(같은 질문을 봤지만 결국 또 직접 문의로 갔다 = 부실 신호).
- §3-2 열람 이력 패널은 `viewed`를 질문당 1줄(횟수·마지막 시각 포함)로, `handoff`는 발생 시점마다 표시.
- 보관 정책: 별도 만료 없음(행동 로그 경량). 운영 중 비대해지면 v2에서 오래된 `viewed` 정리 잡 검토.

```
   │
   └─ 메시지 0건 → 상태 한 줄 + FAQ
        │

        │
        ├─ 맞춤 카테고리/질문 (relevant_stages 일치) 위로 정렬
        │
        ├─ 카테고리 칩 → 질문 항목 목록 → 질문 선택
        │     ├─ handoff=false → 답변 카드 (본문 + 화면 이동 버튼 + [해결됐어요]/[직접 문의])

        │     └─ handoff=true → 바로 메시지 작성 모드 → 'handoff' 기록
        │

```

- 언어: 인플루언서 KO/JA 토글에 따라 `_ja`/`_ko` 선택 (기본 ja)
- 화면 이동: `navigate(action_target)` (해시 라우팅)

### 5-1. 답변 본문 동적 치환 (플레이스홀더 — 구현 필수)

> **누락 보강 (2026-05-21 인계 점검).** 답변 본문(`faq_nodes.body_ja`/`body_ko`)에 `{required}` 같은 **플레이스홀더 토큰**을 넣고, 화면에 그릴 때 실제 값으로 치환한다. Q1-1-c(팔로워 부족)가 이미 `{required}`/`{current}`를 본문에 포함하므로 **이 치환 기능이 없으면 인플루언서에게 `{required}人以上`이 그대로 노출**된다(시드 후 즉시 발생).

- **치환 시점:** 답변 카드를 그리기 직전, `messaging.js`에서 본문 문자열의 토큰을 실제 값으로 바꾼다(서버 호출 없음, 클라이언트 데이터 사용)
- **지원 토큰 화이트리스트(초기):**
  - `{required}` = 응모 차단된 캠페인의 `min_followers`
  - `{current}` = 본인 대표 SNS(primary_channel, 없으면 첫 채널) 팔로워 수
  - (확장 시 이 목록에 추가 — 임의 토큰 허용 금지)
- **값이 없을 때(데이터 누락):** 그 토큰이 들어간 **줄을 통째로 생략**하고 나머지 안내만 노출(답변 문서 Q1-1-c 메모 그대로). 토큰을 빈칸·0으로 남기지 않는다
- **치환은 텍스트 노드에만**, 치환 값은 `esc()` 이스케이프(숫자라도 일관 적용). 토큰 문법은 `{영문소문자}` 고정 — 사용자 입력 답변에 우연히 들어간 중괄호 오인 방지 위해 화이트리스트 매칭만 치환
- **맥락 전달:** Q1-1-c는 §3-4 ③(응모 차단) 케이스와 짝. 응모 차단 알럿에서 진입하든 FAQ에서 직접 열든, 같은 치환 함수로 동일 수치를 보여준다

## 6. 진입 동선 (FAQ를 만나게 하는 설계)

> "메시지 0건일 때만" 노출하면 LINE에 익숙한 인플루언서는 만날 일이 적다. 동선을 넓힌다.

1. **응모이력 카드의 메시지 버튼** — 기존 진입점. 0건이면 FAQ가 먼저
2. **결과물 반려 알림** 클릭 시 → 활동관리로 가되, 메시지 진입에서 "재제출 방법" FAQ 우선 노출
3. **(운영 정책) LINE 응답 매크로** — LINE 문의 시 "앱 메시지에서 바로 확인하실 수 있어요" 안내로 앱 메시지로 유도 (메시지 기능 운영 배포 후)
4. (v2) 신규 응모 직후 "문의는 여기서" 온보딩 툴팁

## 7. 답변 작성 가이드 (톤·쉬운 설명·단계 안내)

### 7-1. 대원칙 (2026-05-21 사용자 명시 — 모든 인플루언서 안내에 영구 적용)

> **인플루언서는 시스템·개발 이해도가 매우 낮다고 전제한다.** 모든 자동응답 문안은 **초등학생에게 설명하듯 쉽게**, 처리 방향은 **단계별(1·2·3)로 끊어서** 안내한다.

- **쉬운 말로:** 전문용어·시스템 용어·영어 약어 금지. 부득이하면 쉬운 풀이를 괄호로. (예: `URL` → 「リンク（URL）」, "인증" → 「メールのリンクを開く」처럼 동작으로 풀어 설명)
- **한 문장 = 한 동작:** 길게 잇지 말고 짧게 끊는다. 한 문장에 할 일 2개 넣지 않기.
- **처리 방향은 번호 단계:** "무엇을 → 어디서 → 어떻게"를 `1. / 2. / 3.` 순번으로. 누르는 버튼 이름은 「」로 정확히.
- **마지막은 항상 안전망:** 막혔을 때 「下の「直接お問い合わせ」からご連絡ください」로 닫는다(handoff 질문 제외).
- **'왜 안 되는지'보다 '어떻게 하면 되는지':** 원인 설명은 짧게, 해결 단계를 길게.

### 7-2. 작성 템플릿 (item 답변 공통 골격)

```
（1文の導入：何についての案内か）
つぎの順番でおこないます。
1. （最初の動作）
2. （つぎの動作。押すボタンは「」で）
3. （…）
（補足が必要なら1〜2文。なければ省略）
わからないときは、下の「直接お問い合わせ」からご連絡ください。
```

- 단계가 없는 단순 안내(예: "메일은 며칠 걸려요")는 번호 없이 2~3문장으로. 단 **할 일이 있으면 반드시 번호 단계**.
- 화면 이동 버튼이 붙는 질문은 본문에서 「下のボタン」으로 그 버튼을 가리킨다.

### 7-3. 톤 (기존 유지)

- 실측 대화의 인플루언서 말투는 매우 정중(「お世話になっております」「よろしくお願いいたします」) — 답변도 **정중하되 간결한 です・ます체**, 과한 경어·딱딱함 지양
- 이모지·아이콘 본문 금지(프로젝트 규칙). 단계·강조는 텍스트(번호·「」)로
- 부정적 상황(반려·비선정)은 완곡하게(「ご縁がありませんでした」), 그래도 **다음에 할 일은 단계로 명확히**
- 한국어(관리자 등록용)와 일본어(노출용)는 **의미 동일·단계 수 동일**하게 작성 (한쪽만 단계가 더 많으면 안 됨)

## 8. 관리자 등록 화면 (`dev/js/admin.js`)

### 8-0. 위치·레이아웃 (2026-05-21 확정)

- **위치: 별도 페인 `#adminPane-faq`** (`/admin#lookups` 탭 안에 끼우지 않음). 사이드바에 신규 메뉴 **"자주 묻는 질문"** 추가. 메뉴 그룹은 응모건 메시지 관련이므로 캠페인 신청·메시지 근처 또는 "관리자 설정" 그룹(구현 시 확정, `campaign_admin` 이상 노출)
- **이유:** 기준 데이터(`lookup_values`)는 짧은 코드값 목록이지만, 자동응답은 긴 답변·2단 트리·화면이동 버튼·미리보기·측정이라 화면 성격이 완전히 다름. 별도 테이블(`faq_nodes`)을 두는 것과 같은 맥락
- **레이아웃: 좌우 2단 (마스터-디테일)**
  - **좌측 — 카테고리 목록:** 카테고리명(한국어) + 그 안의 질문 수, 순서변경 모드, 활성 토글, "+ 카테고리 추가"
  - **우측 — 선택한 카테고리의 질문 항목 목록:** 질문 제목(한국어) + **측정 배지**(조회수 · 직접문의 전환수). 직접문의 전환율이 높은 답변은 경고 표시(부실 답변 식별), 순서변경, 활성 토글, "+ 질문 추가"
  - 좌측에서 카테고리를 고르면 우측이 그 카테고리 질문으로 갱신
- 다른 8개 목록 페인과 동일하게 IntersectionObserver 점진 렌더는 불필요(항목 수 적음). 단 `admin-pane-list` 레이아웃 패턴(제목·헤더 고정, 본문 스크롤)은 따름

### 8-1. 질문 편집 모달

### 8-4. 관리자 응대 보조 화면 (`dev/js/admin-messaging.js`)

> 등록 화면(§8, admin.js)과 **별개**. 관리자가 인플루언서에게 답하는 메시지 화면에 응대 맥락을 더한다.

## 12. 리스크 / 미결

- `admin.js` 핫스팟 → 시퀀셜만
- "응모 취소 에러" 버그는 본 사양 범위 밖 (개발 세션 별도)
- 상태 한 줄 매핑이 캠페인 일정 데이터 정확도에 의존 — 일정 누락 캠페인은 'other'로 폴백, 무난한 기본 문구 필요(§3-0 6단계)
- **비로그인·이메일 미인증 단계는 FAQ에 도달 못 함 (구조적 한계)** — `faq_nodes` SELECT가 `authenticated` 전체이고, 메시지 기능 자체가 응모건 기반이라 응모 전·미인증 사용자는 진입점이 없다. 따라서 카테고리 1(申し込み・応募, "응모가 안 돼요" 계열)은 정작 막힌 당사자가 못 본다. 본 FAQ로는 응모 성공한 사용자만 커버됨을 인지 — 응모 전 단계 안내는 별도 동선(가입 화면 도움말·LINE 매크로)으로 보완(v2)
- **정산 규칙 미확정** — 답변 문서 Q5-1(리워드 지급 시점·조건)이 "운영 규칙 확정 후 보강"으로 잠정 작성됨. 실제 정산 규칙(지급 주기·원천징수·환율·조건)이 확정되면 Q5-1 문안을 보강해야 함. 그 전까지는 "승인 후 PayPal 지급, 시기는 캠페인별" 일반 안내로 노출
- 측정 데이터 활용(부실 답변 자동 알림 등)은 v2
- 번역 도입 시 본 FAQ와 통합 그림 재검토

## 부록 A. 실측 기반 초기 FAQ 콘텐츠 초안

> **답변 본문(일본어 노출용 + 한국어 등록용) 전문은 [`docs/research/2026-05-21-message-faq-answers.md`](../research/2026-05-21-message-faq-answers.md) 에 1차 확정.** 아래는 카테고리·질문 골격이며, 실제 등록 문안은 답변 문서가 source of truth.
> 운영 시작 즉시 가치를 내도록, 실측 7건에서 도출한 초기 카테고리·질문. (요지)는 한국어 기획 메모.

### 카테고리 1. 신청·응모 (申し込み・応募)
- Q. 신청(응모)이 안 돼요 — (요지) 이메일 인증·필수정보·팔로워·중복·마감 5가지 원인별 안내. 화면이동: 기본정보
- Q. 신청을 취소하고 싶어요 — (요지) 응모이력 ⋮ 메뉴 안내. 화면이동: 응모이력. ※취소 버그 수정 후 정확화
- Q. 다시 신청할 수 있나요 — (요지) 취소 후 재신청 가능 안내

### 카테고리 2. 심사·결과 (審査・結果)
- Q. 심사는 언제 끝나나요 — (요지) **상태 한 줄로 1차 해소** + 일반 소요 안내. 단계태깅: pending
- Q. 당첨됐는지 어떻게 아나요 — (요지) 알림·상태 표시 위치 안내. 단계태깅: pending
- Q. 왜 비승인(선정 안 됨)됐는지 알고 싶어요 — (요지) 종합·정성 판단이라 개별 사유 미공개, 완곡 안내 + 재응모 권유. 단계태깅: rejected (답변 문서 Q2-3, 정책 §3-4 ①)

### 카테고리 3. 결과물 제출 (成果物の提出)
- Q. 영수증을 어떻게 올리나요 — (요지) 주문번호·구매일·구매금액 필수. 화면이동: 활동관리. 단계태깅: approved_purchase
- Q. 게시물 주소(URL) 등록이 안 돼요 — (요지) 자동 채널 인식·수동 선택 안내. 화면이동: 활동관리. 단계태깅: approved_post
- Q. 게시물 규칙(해시태그)이 있나요 — (요지) #PR/#広告 필수 + 브랜드 지정 태그·게시 유지 기간. 단계태깅: approved_post (답변 문서 Q3-3)
- Q. 제출 기한이 지났어요 / 못 맞출 것 같아요 — (요지) 기한 연장 상담 → handoff. 단계태깅: approved_post

## 부록 B. 실사례 검증 케이스 (2026-05-21)

> 실제로 접수된 인플루언서 문의. **이 설계(특히 §3·§3-1 개인화 상태 한 줄)가 왜 핵심인지**를 보여주는 검증 사례. 개발 세션은 이 케이스를 보며 우선순위를 잡을 것.

### 원문 (일본어)
> お世話になっております。(投稿のInstagram URL) こちら投稿させていただきました。ご確認よろしくお願いします。
> アプリ上ですと、購入して投稿？と書いてあるのですが商品が自宅に届いたのでそのままレビューしたのですが、アプリの処理はどうしたらいいでしょうか？
> こちらのスクショのページの処理がわかりません。インスタのdmにて連絡してるのですが返信がなくこちらに連絡させていただきました。

### 문의 해부
| 인플루언서가 한 행동/혼란 | 본질 | 본 설계의 대응 |
|---|---|---|
| 게시물을 인스타에 올리고 URL을 메시지로 보냄 | 결과물 제출을 **앱이 아닌 메시지·DM으로** 하고 있음 (실측 1위) | Q3-2 답변 + 화면 이동 버튼으로 앱 「成果物提出」 유도 |
| "구매해서 투고라고 적혀있는데 무료로 받았다" | **내 캠페인이 구매형인지 무료제공형인지 모름** | §3 개인화 상태 한 줄이 타입·단계를 자동 표시 |
| "이 화면(스크린샷) 처리를 모르겠다" | 결과물 제출 화면 사용법 모름 | Q3-1/Q3-2 + 단계 맞춤 노출 |
| "DM 답장이 없어 여기로 왔다" | 채널 분산 → 관리자 응대 지연 | 앱 메시지로 창구 일원화 (실측 통찰 1) |

### 이 케이스가 입증하는 것
1. **개인화 상태 한 줄(§3)이 없으면** 인플루언서는 매번 "내가 뭘 해야 하나"를 사람에게 물어야 한다.
2. **관리자 측 상태 표시(§3-1)가 없으면** 관리자는 답하기 전에 매번 그 사람 계정·캠페인을 조회해야 한다. 이번에 실제로 "계정·캠페인을 확인해 답해줄 수 있냐"는 요청이 나왔다 = 조회 부담의 증거.
3. **고정 FAQ만으로는 부족** — "구매형이냐 무료제공형이냐"는 응모건마다 달라 개인화 없이는 정답을 못 준다 (B 문의의 전형).

### 즉시 활용 (자동응답 등록 전 임시)
관리자 응대용 일본어 답변 2종(무료제공형/구매형)은 [`docs/research/2026-05-21-message-faq-answers.md`](../research/2026-05-21-message-faq-answers.md) Q3-1·Q3-2 와 본 세션 대화에 정리됨. 타입 확인 후 해당 문안 사용.
