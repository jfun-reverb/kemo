# 인플루언서 피드백·소통 채널 역할 분담 사양

> **작성일:** 2026-05-21
> **작성:** 기획/설계 세션
> **상태:** 설계 검토 (구현 미착수)
> **목적:** 인플루언서가 알림·화면·메시지·메일·LINE 등 **여러 채널에서 같은 피드백을 중복으로 받아 정신없어지는 것**을 막는다. 채널마다 역할을 하나로 고정하고, 응모건 메시지·자동응답(FAQ) 운영 배포 시 **LINE 1:1 문의를 앱 메시지로 옮기는 전환 계획**을 정의한다.
> **연관:** [`docs/specs/2026-05-15-application-messaging.md`](./2026-05-15-application-messaging.md)(응모건 메시지), [`docs/specs/2026-05-21-message-faq.md`](./2026-05-21-message-faq.md)(자동응답 FAQ)

---

## 0. 핵심 결론 (먼저 읽기)

1. **채널마다 역할을 하나로 고정** — 같은 내용을 새 채널에 또 밀어넣지 않는다.
2. **메시지로 대체 가능한 LINE은 "특정 응모 건에 대한 1:1 문의"뿐** — 메시지 기능이 응모건(신청 건) 단위라서 그렇다. 신규 캠페인 홍보·가입 전 문의·탈퇴 같은 응모건 밖 소통은 메시지로 못 옮긴다 → LINE/메일/알림 유지.
3. **반려·검수 결과는 통지를 더하지 말고, 후속 대화만 메시지로 수렴** — 지금도 알림+화면 배너+메일 3곳에서 통지된다.

## 1. 현재 인플루언서 채널 전수

### 1-1. 받는 채널 (운영 → 인플루언서)

| 채널 | 무엇을 받나 | 성격 | 구현 |
|---|---|---|---|
| 앱 알림(종) | 결과물 승인/반려/변경, 응모 취소, 메시지 도착 | 1회성 푸시 신호 | `notifications` 5종 |
| 활동관리 화면 | 반려 사유 배너, 재제출 폼 | 할 일의 정본 | `mypage.js` |
| 응모이력 화면 | 신청 상태(심사/당첨/반려/취소) | 상태 정본 | `mypage.js` |
| 응모건 메시지 | 운영팀 답장 + (신규)상태 한 줄·FAQ | 양방향 대화 | `messaging.js`(PR 1·2) |
| 이메일 | 일일 다이제스트(신청·승인·반려·마감), 검수 결과, 캠페인 홍보 | 앱 밖 리마인더 | Edge Function 3종 |
| LINE @reverb.jp | (외부) 실제 문의가 몰림 | 외부 1:1 | 외부 |

### 1-2. 보내는·문의하는 채널 (인플루언서 → 운영)

| 채널 | 용도 | 비고 |
|---|---|---|
| 응모건 메시지 | 특정 응모 건 문의 | 운영 미배포(약관 D-7 대기) |
| LINE | 모든 문의(현재 주력) | 응모건 무관 문의도 가능 |
| (없음) | 가입 전·계정 문제 | 현재 LINE뿐 |

## 2. 채널 역할 분담 원칙 (채널 = 목적 하나)

| 채널 | **단 하나의 역할** | 하지 말 것 |
|---|---|---|
| 앱 알림(종) | "방금 무슨 일이 생겼다" 신호 + 정본 화면으로 보내기 | 사유 전문·긴 설명 넣기 |
| 활동관리·응모이력 | 내 상태·할 일의 **정본**(반려 사유 전문은 여기) | — |
| 응모건 메시지 | 운영팀과 **양방향 대화**(질문·예외·개별 상담) | 알림으로 충분한 통지를 중복 푸시 |
| 자동응답 FAQ | 묻기 전 **셀프 해결 + 상태 요약** | — |
| 이메일 | 앱 안 볼 때 **외부 리마인더**(다이제스트·마감) | 앱 알림과 동일 내용 즉시 중복 |
| LINE | **응모건 밖 소통**(신규 캠페인 홍보·가입 전·탈퇴) | 응모건 1:1 문의(→ 메시지로 이관 대상) |

> 원칙: **하나의 사건은 "통지 채널 1~2 + 정본 화면 1 + 대화 채널 1"**로 한정. 통지(알림·메일)는 짧게 신호만, 내용은 정본 화면, 대화는 메시지 한 곳.

## 3. 중복 진단 — "반려"가 지금 몇 곳에서 오나

결과물 반려 시 현재 **동시 3곳**:
1. 앱 알림 `deliverable_rejected`
2. 활동관리 빨간 배너(사유 전문)
3. 검수 결과 이메일

→ 여기에 "반려를 메시지로도 통지"를 더하면 4번째. **금지.** 대신 알림·배너에서 **"이 건 문의하기" 동선만 메시지로 연결**(FAQ "재제출 방법" 우선 노출). 통지 수는 그대로, 되묻는 창구만 일원화.

## 4. 메시지·FAQ 결합 — 추천 / 금지

| 결합 | 판정 |
|---|---|
| ✅ 반려·검수 결과의 **후속 대화**를 메시지로 수렴(통지 추가 X, 동선만) | 추천 |
| ✅ LINE 1:1 문의 → 앱 메시지로 유도(§6) | 추천(운영 배포 후) |
| ✅ 알림 클릭 → 맞춤 FAQ 노출(반려 알림→"재제출 방법") | 추천(FAQ §6) |
| ⚠️ 메시지 상단 상태 한 줄로 마감·검수중 표시 | 이미 FAQ §3. 이메일 마감 리마인더와 역할 분담(메시지=수동 확인, 이메일=푸시) |
| ❌ 반려를 메시지로 **또** 통지 | 4번째 채널, 과부하 |
| ❌ 마감 임박을 메시지로 별도 푸시 | 이메일+상태 한 줄로 충분 |

## 5. LINE 유도 지점 전수 + 메시지 전환 가부

> 메시지 기능은 **응모건 단위**다. "특정 응모에 대한 1:1 문의"만 메시지로 옮길 수 있고, 나머지는 LINE을 유지해야 한다.

| # | 위치(코드) | 현재 문구(요지) | 분류 | 메시지 전환? |
|---|---|---|---|---|
| A1 | `dev/js/ui.js:715,723` (푸터·문의 안내) | "ご質問は公式LINEまでお問い合わせください" | 1:1 문의 유도 | **△ 부분** — 응모건 문의는 메시지로, 일반 문의는 LINE 병기 |
| A2 | `dev/lib/legal.js:146` (약관 화면) | "韓国語版確認 or 公式LINE 문의" | 1:1 문의 유도 | △ — 메시지 안내 추가 가능, 비로그인엔 LINE 유지 |
| A3 | 인플 메일 4줄 푸터 + deliverable 메일 `{{help_line_url}}` | "お問い合わせは LINE @reverb.jp まで" | 1:1 문의 유도 | **○ 검수 메일은 메시지 링크 우선** + LINE 보조 |
| B1 | i18n `lineAddCta`/`lineAddSub` | "LINE 추가하고 최신 캠페인 정보 받기" | 마케팅·정보 푸시 | **✕ 유지** — 신규 캠페인 푸시는 응모건 밖 |
| B2 | `dev/js/application.js` 응모완료 LINE QR 카드, `dev/index.html:367` | 응모 직후 LINE 추가 권유 | 마케팅·정보 푸시 | ✕ 유지 |
| C1 | `winner_announce` 기본값 "選考後、LINEにてご連絡"(admin.js 다수 + 캠페인 상세) | 당첨 발표를 LINE으로 | 당첨 발표 채널 | **△ 검토** — 앱 알림으로 대체 가능하나 운영 프로세스 변경(§7) |
| D1 | 탈퇴 토스트 "退会申請…運営にLINEでご連絡"(`mypage.js:463`, i18n) | 탈퇴 요청은 LINE | 응모건 밖 요청 | ✕ 유지(응모건 아님) — 단 별도 창구 검토 |
| E1 | `lineId`/`lineIdHint` 프로필 입력, admin 목록 LINE 컬럼 | LINE ID 수집·표시 | 연락처 데이터 | ✕ 유지(식별·비상 연락) |

> 광고주(brand) 메일의 LINE(`brand-ack-*`)은 **인플루언서 대상이 아니므로 본 검토 범위 밖** — 영업 시퀀스 유지.

## 6. 메시지 운영 배포 시 LINE→메시지 전환 체크리스트

> 응모건 메시지가 운영 배포되는 시점(메시지 PR 5 약관 통지)과 **함께** 적용. 그 전에 먼저 바꾸면 인플루언서가 안 열리는 메시지로 안내받는다.

- [ ] **A3 검수 결과 메일** — `{{help_line_url}}` 옆/위에 "アプリのメッセージからもお問い合わせいただけます" + 해당 응모건 메시지 딥링크(가능하면). LINE은 보조로 병기
- [ ] **A1 푸터·문의 안내(ui.js)** — 로그인 상태: "応募に関するお問い合わせはアプリのメッセージから" 우선, 일반/계정 문의는 LINE 병기
- [ ] **A2 약관 화면(legal.js)** — 로그인 상태면 메시지 안내 추가, 비로그인은 LINE 유지
- [ ] **C1 당첨 발표 문구** — §7 결정에 따름. 바꾼다면 `winner_announce` 기본값 + 기존 캠페인 문구 일괄 점검
- [ ] **유지 확정(B1·B2·D1·E1)** — 바꾸지 않음을 명시(실수로 제거 방지)
- [ ] 전환 후에도 **LINE을 완전히 끄지 않는다** — 응모건 밖 문의·비로그인·긴급 연락의 폴백

## 7. 메시지로 대체 불가 — 남겨야 할 LINE 역할

메시지는 응모건 단위라 아래는 **반드시 LINE(또는 다른 채널)이 남아야** 한다.

1. **신규 캠페인 홍보·정보 푸시**(B1·B2) — 응모 전 단계. 메시지 진입점 없음
2. **가입 전·로그인 불가·계정 분실 문의** — 앱에 못 들어온 사람
3. **탈퇴 등 계정 단위 요청**(D1) — 특정 응모건이 아님
4. **긴급·장애 시 폴백** — 앱 메시지가 안 될 때
5. **당첨 발표**(C1) — 앱 알림으로 옮길 수 있으나(§7 검토), 옮기기 전까지 LINE 유지

> ⚠️ "메시지 생겼으니 LINE 다 없애자"는 위험. **LINE은 응모건 밖 소통의 정본으로 남기고, 응모건 1:1만 메시지로 이관**한다.

## 9. 리스크 / 미결

- **C1 당첨 발표를 LINE→앱 알림으로 옮길지** 미결 — 운영팀이 LINE으로 개별 안내하는 현 방식의 장단(친밀감 vs 누락) 검토 필요
- **D1 탈퇴 창구** — 메시지 부적합. 별도 "계정 문의" 폼/메뉴를 둘지 검토
- 메일 딥링크 — 검수 메일에서 특정 응모건 메시지로 바로 여는 링크가 기술적으로 가능한지 확인(없으면 "앱에서 확인" 일반 안내)
- 전환 후 LINE 문의량 감소 추이 측정 → 과도 이관 시 부작용(앱 미숙 사용자) 모니터링
- 비로그인·다국어(KO/JA) 안내 문구 일관성
