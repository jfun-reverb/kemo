# 인플루언서 피드백·소통 채널 역할 분담 사양

> **작성일:** 2026-05-21
> **작성:** 기획/설계 세션
> **상태:** 설계 검토 (구현 미착수)
> **목적:** 인플루언서가 여러 채널에서 같은 피드백을 중복으로 받아 정신없어지는 것을 막고, **LINE 소통을 최대한 줄여 관리자 응대 창구를 앱 한 곳(받은편지함)으로 통합**한다.
> **연관:** [`docs/specs/2026-05-15-application-messaging.md`](./2026-05-15-application-messaging.md)(응모건 메시지), [`docs/specs/2026-05-21-message-faq.md`](./2026-05-21-message-faq.md)(자동응답 FAQ)

## 0. 핵심 결론 (먼저 읽기) — LINE 최소화 확정

> **사용자 결정(2026-05-21):** LINE 소통을 **최대한 줄이고**, 관리자가 보는 창구를 **앱 한 곳(받은편지함)으로 통합**한다. "관리자가 관리할 채널이 너무 분산"이 핵심 문제.

1. **채널마다 역할을 하나로 고정** — 같은 내용을 새 채널에 또 밀어넣지 않는다.
2. **LINE을 최대한 앱으로 흡수:**
   - 응모 건 1:1 문의 → 응모건 메시지 (이미 설계됨)
   - **응모건 밖 일반 문의 → 신규 "일반 문의" 창구 신설(§10)** ← LINE 축소의 열쇠
   - 당첨 발표 → 앱 알림 + 메시지
   - 캠페인 홍보 → 캠페인 홍보 메일(이미 있음) + 앱 홈
3. **LINE에 끝까지 남는 건 "앱에 못 들어오는 경우"뿐** — 가입 전·로그인 불가·장애 폴백(§7). 그 외 LINE 유도 문구는 앱으로 교체.
4. **반려·검수 결과는 통지를 더하지 말고, 후속 대화만 메시지로 수렴** — 지금도 알림+화면 배너+메일 3곳에서 통지된다.

## 1. 현재 인플루언서 채널 전수

### 1-1. 받는 채널 (운영 → 인플루언서)

| 채널 | 무엇을 받나 | 성격 | 구현 |
|---|---|---|---|
| 앱 알림(종) | 결과물 승인/반려/변경, 응모 취소, 메시지 도착 | 1회성 푸시 신호 | `notifications` 5종 |
| 활동관리 화면 | 반려 사유 배너, 재제출 폼 | 할 일의 정본 | `mypage.js` |
| 응모이력 화면 | 신청 상태(심사/당첨/반려/취소) | 상태 정본 | `mypage.js` |
| 응모건 메시지 | 운영팀 답장 + (신규)상태 한 줄·FAQ | 양방향 대화 | `messaging.js`(PR 1·2) |
| 이메일 | 일일 다이제스트, 검수 결과, 캠페인 홍보 | 앱 밖 리마인더 | Edge Function 3종 |
| LINE @reverb.jp | (외부) 실제 문의가 몰림 | 외부 1:1 | 외부 |

### 1-2. 보내는·문의하는 채널 (인플루언서 → 운영)

| 채널 | 용도 | 현재 |
|---|---|---|
| 응모건 메시지 | 특정 응모 건 문의 | 운영 미배포(약관 D-7 대기) |
| **일반 문의 창구** | 응모건 밖 문의 | **없음 → 신설(§10)** |
| LINE | 모든 문의(현재 주력) | 가장 분산의 원인 |

## 2. 채널 역할 분담 원칙 (채널 = 목적 하나)

| 채널 | **단 하나의 역할** | 하지 말 것 |
|---|---|---|
| 앱 알림(종) | "방금 무슨 일이 생겼다" 신호 + 정본 화면으로 보내기 | 사유 전문·긴 설명 넣기 |
| 활동관리·응모이력 | 내 상태·할 일의 **정본**(반려 사유 전문은 여기) | — |
| 응모건 메시지 | 특정 응모 건의 **양방향 대화** | 알림으로 충분한 통지를 중복 푸시 |
| **일반 문의 창구** | 응모건 밖 **양방향 대화**(계정·정산 전반·기타) | — |
| 자동응답 FAQ | 묻기 전 **셀프 해결 + 상태 요약** | — |
| 이메일 | 앱 안 볼 때 **외부 리마인더**(다이제스트·마감) | 앱 알림과 동일 내용 즉시 중복 |
| **LINE** | **앱에 못 들어오는 경우만**(가입 전·로그인 불가·장애 폴백) | 로그인 사용자의 문의(→ 앱 메시지로) |

> 원칙: 하나의 사건은 "통지 1~2 + 정본 화면 1 + 대화 채널 1"로 한정. **로그인한 인플루언서의 모든 대화는 앱 메시지(응모건 + 일반 문의) 한 곳으로.**

## 3. 중복 진단 — "반려"가 지금 몇 곳에서 오나

결과물 반려 시 현재 **동시 3곳**: ① 앱 알림 `deliverable_rejected` ② 활동관리 빨간 배너(사유 전문) ③ 검수 결과 이메일.

→ "반려를 메시지로도 통지"를 더하면 4번째. **금지.** 대신 알림·배너에서 **"이 건 문의하기" 동선만 메시지로 연결**(FAQ "재제출 방법" 우선). 통지 수는 그대로, 되묻는 창구만 일원화.

## 6. LINE → 앱 전환 체크리스트

> 응모건 메시지 + 일반 문의 창구가 **운영 배포된 시점**(메시지 PR 5 약관 통지)과 **함께** 적용. 그 전에 문구부터 바꾸면 인플루언서가 안 열리는 메시지로 안내받는다.

- [ ] **A1·A2 문의 안내(ui.js·legal.js)** — 로그인: "お問い合わせはアプリのメッセージから"(응모건은 해당 카드, 일반은 「お問い合わせ」 메뉴). 비로그인만 LINE
- [ ] **A3 메일 푸터** — 검수 메일은 해당 응모건 메시지 딥링크 우선, 일반 메일은 일반 문의 안내. LINE은 폴백으로 작게
- [ ] **B1·B2 홍보 LINE** — 신규 캠페인 안내는 홍보 메일·앱 홈 중심으로. LINE 추가 카드는 축소(완전 제거는 마케팅팀 협의)
- [ ] **C1 당첨 발표** — 앱 알림 신설(`notifications` 신규 kind) + 메시지. `winner_announce` 기본값·기존 캠페인 문구 일괄 갱신
- [ ] **D1 탈퇴** — 일반 문의 창구 또는 마이페이지 탈퇴 기능으로
- [ ] **E1 LINE ID** — 가입 필수에서 선택으로 검토(비상 연락 용도만 남김)
- [ ] **LINE 완전 종료 금지** — §7 폴백은 유지

## 7. LINE에 끝까지 남기는 것 (최소)

앱에 **못 들어오는** 상황만 LINE(또는 이메일)이 폴백으로 남는다. 그 외는 모두 앱으로.

1. **가입 전 문의** — 계정이 없어 앱 메시지 진입 불가
2. **로그인 불가·계정 분실** — 앱에 못 들어옴
3. **장애·긴급 폴백** — 앱 메시지가 동작 안 할 때
4. (과도기) 앱 사용 미숙 사용자 — 전환 안착까지 한시 유지

> 이 4가지 외에는 LINE 유도 문구를 남기지 않는다. 단 "LINE 채널 자체를 폐쇄"하는 게 아니라 **앱 안에서 LINE으로 보내는 유도를 제거**하는 것.

## 9. 리스크 / 미결

- **일반 문의 창구 데이터 모델**(§10) — 기존 `application_messages` 확장 vs 별도 테이블. supabase-expert·planner와 결정
- **당첨 발표 LINE→앱 알림 전환**(C1) — 운영팀의 친밀한 개별 안내 방식 변화. 누락 위험 vs 일원화 이득
- **전환 과도기 모니터링** — LINE 의존이 큰 기존 사용자가 앱 미숙으로 이탈하지 않도록 단계적 전환 + 안내
- **비로그인·다국어(KO/JA) 안내 문구** 일관성
- **메일 딥링크** — 검수 메일에서 특정 응모건 메시지로 바로 여는 링크 기술 가능성 확인

## 10. 신규 — 일반 문의 창구 (응모건 비종속 메시지)

> **결정(2026-05-21):** 응모건에 매이지 않는 인플루언서↔관리자 문의 창구를 신설한다. **LINE 일반 문의를 앱으로 흡수**하는 핵심 기능. **상세 설계는 [`general-inquiry-desk.md`](./2026-05-21-general-inquiry-desk.md)로 분리**(아래는 요약).

### 10-1. 목적
- 계정·정산 전반·"이거 어떻게 해요" 등 **특정 응모건이 아닌 문의**를 앱에서 받는다
- 관리자는 LINE 앱 대신 **앱 받은편지함 하나**에서 응모건 메시지 + 일반 문의를 함께 응대

### 10-2. 인플루언서 진입점
- 마이페이지 메뉴 또는 GNB 햄버거에 **「お問い合わせ」** 항목 신설
- 진입 시 **FAQ 먼저 노출**(응모건 무관 카테고리: 계정·정산·기타) → 안 풀리면 직접 문의(메시지 작성)
- 응모건 메시지와 동일한 게시판형 모달 UX 재사용

### 10-3. 관리자 화면
- 받은편지함(`#adminPane-messages`)에 **「일반 문의」 탭/필터** 추가 — 응모건 메시지와 한 화면에서 응대
- 응대 완료·강제 숨김·열람 이력 등 기존 메시지 기능 그대로 재사용

### 10-4. 데이터 모델 (미결 — 별도 결정)
- **옵션 A:** 기존 `application_messages`에서 `application_id`를 NULL 허용으로 확장 + 스레드 구분 컬럼. 기능 재사용 최대, 단 RLS·뷰·받은편지함 쿼리 전면 영향
- **옵션 B:** 별도 테이블(`general_inquiries` + 메시지). 격리 깔끔, 단 메시지 기능(첨부·회수·숨김·응대완료)을 상당 부분 중복 구현
- 추천 방향: **A를 우선 검토**(코드 재사용)하되, RLS·뷰 영향 범위를 supabase-expert가 평가 후 확정
- 첨부·압축·회수·숨김·응대완료·rate limit은 기존 응모건 메시지 정책 그대로 적용

### 10-5. 약관·개인정보
- 신규 개인정보 수집 없음(기존 메시지와 동일 성격) → 개인정보처리방침 개정 불필요. 단 LINE 안내 통지·운영 배포는 메시지 본체 약관 통지와 함께

### 10-6. 미결
- 데이터 모델 A/B 확정
- 비로그인 차단(일반 문의도 로그인 필수 — 가입 전 문의는 §7 LINE 폴백)
- 일반 문의용 FAQ 카테고리 구성(자동응답 FAQ와 공유 vs 분리)
- 운영 배포 시점(응모건 메시지와 동시 vs 이후)
