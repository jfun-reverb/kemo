# 일반 문의 창구 (응모건 비종속 메시지) 사양

> **작성일:** 2026-05-21
> **작성:** 기획/설계 세션
> **상태:** 설계 초안 (데이터 모델 미확정 — supabase-expert 결정 대기)
> **목적:** 응모건(신청 건)에 매이지 않는 인플루언서↔관리자 문의 창구를 신설해, **LINE으로 오던 일반 문의를 앱으로 흡수**하고 관리자 응대 창구를 앱 받은편지함 하나로 통합한다.
> **상위 결정:** [`docs/specs/2026-05-21-feedback-channel-roles.md`](./2026-05-21-feedback-channel-roles.md) §10 (LINE 최소화)

## 0. 한 줄 정의

응모이력 카드가 아니라 **마이페이지의 「お問い合わせ」 메뉴**로 들어가는, **특정 응모 건과 무관한** 인플루언서↔관리자 양방향 메시지. 관리자는 응모건 메시지와 **같은 받은편지함**에서 응대한다.

---

## 1. 범위

- **포함:**
  - 인플루언서: 마이페이지 「お問い合わせ」 진입 → FAQ(응모건 무관 카테고리) 먼저 → 안 풀리면 직접 문의(메시지)
  - 관리자: 받은편지함(`#adminPane-messages`)에 「일반 문의」 구분 추가 — 응모건 메시지와 한 화면
  - 기존 메시지 기능(첨부·압축·회수·강제숨김·응대완료·읽음·rate limit) 재사용
- **제외:**
  - 가입 전·로그인 불가 문의 → 구조상 LINE/이메일 폴백 유지(채널 역할 사양서 §7)
  - 한국어↔일본어 번역(별도)
  - 일반 문의 전용 신규 알림 종류는 기존 `message_received` 재사용 검토(§6)

---

## 2. 핵심 설계 결정

| 항목 | 결정 | 이유 |
|---|---|---|
| 진입점 | **마이페이지 「お問い合わせ」 메뉴** (+ GNB 햄버거에도 노출 검토) | 응모건과 무관하므로 응모이력 카드가 아닌 독립 진입 |
| FAQ 우선 | **진입 시 일반 카테고리 FAQ 먼저** | LINE 일반 문의의 상당수도 셀프 해결 가능(자동응답 FAQ 재사용) |
| 스레드 단위 | **인플루언서당 1 스레드** | 일반 문의는 응모건처럼 여러 개로 쪼갤 필요 없음. 한 사람=한 대화 |
| 관리자 화면 | **기존 받은편지함에 통합** (별도 페인 아님) | "창구 하나로 통합"이 목표. 응모건 메시지와 같은 화면에서 응대 |
| 기능 재사용 | 첨부·회수·숨김·응대완료·읽음·rate limit **그대로** | 응모건 메시지와 정책 동일 |
| 운영 배포 | **응모건 메시지 운영 배포(약관 D-7 통지)와 함께 또는 직후** | LINE 전환 문구도 이때 동시(채널 역할 §6) |

## 3. 인플루언서 화면 흐름

```
마이페이지 → 「お問い合わせ」
   │
   ├─ 메시지 0건 → 일반 FAQ(계정·정산·기타) 먼저 + [直接お問い合わせ]
   │
   └─ 메시지 1건+ → 기존 대화 스레드
        └─ 하단 입력창에서 추가 문의 (텍스트 + 이미지 첨부, 응모건 메시지와 동일 UX)
```

- 게시판형 풀스크린 모달 — 응모건 메시지(`messaging.js`)의 모달·압축·회수 로직 재사용
- 응모건 메시지와 시각적으로 동일하되, **상단에 "응모건 상태 한 줄"은 없음**(특정 응모건이 아니므로). 대신 "운영팀 문의" 헤더

---

## 4. 관리자 화면 흐름

- 받은편지함(`#adminPane-messages`)에 **「일반 문의」 필터/구분** 추가
  - 기존: 캠페인 선택 → 응모건 스레드 목록
  - 추가: 「일반 문의」를 고르면 인플루언서별 일반 문의 스레드 목록
- 스레드 열면 응모건 메시지와 동일한 답장·응대완료·숨김 UI
- **응모건 상태 한 줄(자동응답 FAQ §3-1)은 일반 문의엔 미표시** (특정 응모건 없음). 대신 그 인플루언서의 **기본 정보·최근 응모 요약** 정도를 보조로 노출 검토(v2)
- 미응대 배지: 응모건 미응대 + 일반 문의 미응대를 **합산**해 사이드바에 표시(통합 목표)

## 5. 데이터 모델 — 미확정 (supabase-expert 결정 대기)

> **중요:** `application_messages`는 `application_id NOT NULL`을 중심축으로, 행 단위 보안 정책(RLS)·요약 뷰(`application_message_summary`)·모든 원격 함수(`get_application_messages` 등)가 이 값에 의존한다. 일반 문의를 어떻게 담을지가 이 작업의 최대 분기점이다.

| 옵션 | 방식 | 장점 | 단점·위험 |
|---|---|---|---|
| **A. 기존 테이블 확장** | `application_messages.application_id`를 NULL 허용 + 소유자 식별 컬럼(`target_influencer_id`) 추가 | 첨부·회수·숨김·응대완료 로직 100% 재사용 | **RLS·요약 뷰·모든 원격 함수가 `application_id` 중심** → NULL 분기를 전부 추가해야 함. 본인 식별을 `target_influencer_id`로 바꾸는 광범위 수정. 운영 대기 중인 PR 1·2 구조에 큰 변경 |
| **B. 별도 테이블** | `general_inquiry_messages`(influencer_id 기반) 신설 | 기존 구조 무손상, RLS 단순(본인 행) | 첨부·회수·숨김·응대완료·읽음·rate limit RPC를 상당 부분 **복제** |
| **C. 스레드 추상화** | `message_threads`(kind=application\|general) 도입 + 메시지가 thread 참조 | 가장 깨끗·확장성 | PR 1·2 테이블 **대수술**. 운영 미배포라도 검증 끝난 구조를 갈아엎는 위험 |

- **기획 추천(잠정):** **B(별도 테이블)** 를 1순위로 검토. 일반 문의는 응모건 메시지보다 단순(상태 한 줄·반려 사유 연동 없음)해서 복제량이 제한적이고, 운영 대기 중인 응모건 메시지 구조를 건드리지 않아 안전. 단 **공통 로직(압축·회수·숨김 RPC)을 함수로 공유**해 중복 최소화.
- **확정 권한:** supabase-expert가 RLS·뷰·RPC 영향 범위를 평가해 A/B/C 최종 결정. 그 결과를 본 §5에 기록.

## 6. 기존 메시지 기능 재사용 + 알림

- **재사용:** 이미지 압축(`image-compress.js`), 첨부(Storage `application-message-attachments` 또는 신규 버킷), 25분 본인 회수, 관리자 강제 숨김 + 이력, 응대 완료 마킹, 개인별 읽음, rate limit(발신 100/h·회수 50/h)
- **알림:** 관리자가 일반 문의에 답장 시 `message_received` 알림 재사용(자동응답 FAQ·응모건 메시지와 동일 kind). ref_table은 일반 문의 식별자로 분기 — 알림 클릭 시 「お問い合わせ」 모달 오픈
- **메일 지연 큐:** 응모건 메시지 PR 4 정책(있으면) 동일 적용 — 별도 결정

---

## 7. FAQ 연계

- 일반 문의 진입 시 노출할 FAQ는 **응모건 무관 카테고리** (계정·프로필, 보수·정산 일반, 그 외) — 자동응답 FAQ(`faq_nodes`)에서 `relevant_stages IS NULL`(단계 무관) 항목 중심
- 응모건 단계 맞춤 노출(자동응답 FAQ §3)은 일반 문의엔 미적용(응모건 없음)
- 자동응답 FAQ의 카테고리를 **공유**하되, 일반 문의에서는 "신청·심사·결과물" 같은 응모건 종속 카테고리는 숨기거나 뒤로

## 9. 검증 계획

- 인플루언서: 「お問い合わせ」 진입 → FAQ 노출 → 직접 문의 발신 → 첨부·회수 → 관리자 답장 수신·알림
- 관리자: 받은편지함에서 일반 문의 스레드 응대 → 응대완료 → 미응대 배지에 합산 반영
- 권한: 본인 일반 문의만 조회(RLS), 관리자 전체 조회
- 신규 RPC 스모크 호출(메모리 규칙)
- 에이전트: supabase-expert(데이터 모델·RLS), reviewer, qa-tester
