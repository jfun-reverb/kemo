## 2. 캠페인 홍보 메일 관리자 수신 — 사양

### 2-0. 사용자 결정 사항 (2026-05-27 확정)
| 항목 | 결정 |
|---|---|
| 누가 받나 | 관리자 계정 「메일 받기 설정」에서 캠페인 홍보 메일 토글을 켠 관리자 |
| 발송 타이밍 | 인플루언서 홍보 메일과 **동일하게 주 2회 (월·목 한국시간 오전 9시)**, 같은 자동 스케줄 |
| 내용 | 관리자는 자격 매칭이 불가능 → **그날 홍보 대상이 된 캠페인 전체(신규 + 마감 하루 전)**를 담은 **1통** |
| 발송 방식 | 관리자 **1명당 1통 분리 발송** (다른 관리자 이메일이 받는 사람 칸에 노출 안 됨) |
| 메일 언어 | 제목·머리말·안내 문구는 **한국어**, 캠페인 카드(제목·리워드·모집유형)는 원본 그대로 **일본어** |

### 2-1. 구현 방식 (planner 탐색 → 추천안 채택)

| 항목 | 채택안 | 이유 |
|---|---|---|
| 발송 위치 | **인플 홍보 메일 함수의 "첫 배치"에 관리자 발송을 추가** | 자동 스케줄·중복 차단 잠금장치를 그대로 재활용. 인플 발송은 여러 번 나눠 호출(chained)되는데, 첫 배치에서만 관리자 발송 → **중복 발송이 구조적으로 차단** |
| 캠페인 풀 조회 | **신규 전용 조회 함수 `get_promo_digest_campaign_pool(날짜)` 추가** | 인플 대상 조회 함수와 **같은 캠페인 조건을 SQL 한 곳에서 동기화** → 인플이 보는 캠페인과 관리자가 보는 캠페인이 어긋나지 않음 |
| 메일 템플릿 | **관리자용 메인 틀(wrapper) 1종 신규 추가**, 섹션·카드 틀은 인플 것 재사용 | 인플 틀의 개인화 인사·수신거부 링크·클릭 추적 토큰은 관리자에게 불필요. 카드는 토큰만 빼고 재사용 |
| 실패 기록 | **가벼운 처리** (별도 테이블 없이 함수 로그 + 실행 로그에 요약) | 관리자는 소수, 첫 배치 1회 잠금으로 멱등 충분 |
| 메일 종류 코드 | `campaign_promo` | 기능명과 직관적으로 일치 |

### 2-2. 캠페인 풀 정의 (인플과 일치시켜야 함)
- **신규**: `status='active'` + `first_active_at` 의 한국시간 날짜 = 그날 + `deadline >= 오늘` + (리뷰어형이면 승인 수 < 모집 인원)
- **마감 하루 전**: `status='active'` + `deadline = 내일` + (리뷰어형 모집 인원 조건)
- 인플 대상 조회 함수에 있는 "이미 응모/노출/클릭한 캠페인 제외" 조건은 **관리자 풀에서는 제외**(관리자는 응모 주체가 아님). 즉 관리자는 그날 풀 **전체**를 본다.

### 2-3. 구현상 핵심 주의점
- **관리자 발송 조건은 "인플 대상자 수"가 아니라 "캠페인 풀 유무"**. 인플 대상자가 0명이어도(예: 모두 이미 응모/클릭) 캠페인 풀은 있을 수 있으므로, 관리자 발송 판단/풀 조회를 **인플 0건 조기 종료보다 앞**에 둔다.
- 관리자 발송 실패는 try/catch 로 격리 → 인플 발송 전체를 막지 않음.
- 디버그용 단일 수신자 모드(`testRecipient`)에서는 관리자 발송을 건너뛴다.

### 2-4. 메일 종류 추가 → 관리자 화면 자동 노출
- `lookup_values` 에 `('admin_email_kind', 'campaign_promo', '캠페인 홍보 메일', 'キャンペーン宣伝メール', ...)` 한 줄만 추가하면, 관리자 계정 「메일 받기 설정」 모달에 토글이 **자동으로** 나타남 (관리자 앱 코드 수정 불필요).
- 수신자 조회는 기존 `get_subscribed_admin_emails('campaign_promo')` 함수를 그대로 사용.

### 구현 중 기술 결정 사항
- 관리자 발송 위치: mutex 직후 + 인플 0건 조기 종료보다 **앞**(§2-3) — 인플 대상 0명이어도 풀 있으면 관리자 발송. `isFirstBatch` 가드로 chained 배치 중복 발송 차단. `testRecipient` 모드는 상단 early-return 으로 도달 불가.
- 관리자 발송 실패는 자체 try/catch 격리 — 인플 발송/mutex/finalizeRun 흐름 불간섭.
- 캠페인 상세·monitor 승인수는 관리자 풀 전용으로 재조회(인플 campaignMap 과 독립) — 인플 0건 조기 종료를 통과 못 하는 구조라 공유 불가.
- 관리자 본문은 `renderSection`/`renderCampaignCard` 재사용(token="" — 클릭 추적 안 함).
- `get_subscribed_admin_emails` 반환 `TABLE(email text)` → `adminEmails: string[]` 타입 명시 + `filter((e): e is string => e.length>0)` 로 deno check 통과.
- 개발 DB 적용: 152 SQL Editor 실행 + 시드 4행 확인 + 함수 스모크(빈 풀 정상 반환) 완료. 개발 Edge Function 배포 = 환경 동기화(발송 테스트는 정책상 운영에서).
- 운영 적용(PR 3, 2026-05-27 완료): **전체 dev→main 머지 없이** 운영 DB 152 SQL Editor + 운영 Edge Function deploy 로만 반영(보류 기능 main 머지 회피 — [[project_pending_features_prod_deploy]] 전체 머지 시 코드 형상 일치). 발송 검증은 정책상 수동 강제 발송 대신 다음 정기 자동 발송(목 09:00) 모니터링으로 대체 — 비정기 수동 호출 시 인플루언서 대량 발송 위험 때문.
