# 1. 참여방법 번들

<!-- NOTION:SYNCED 2026-04-15 -->

> 위치: 관리자 페이지 → **기준데이터** → **참여방법** 탭
> 배포일: 2026-04-15

## 1-1. 개념 한눈에

**참여방법 번들** = 캠페인 상세 페이지의 "참가방법 STEP 1~N" 영역을 재사용 가능한 묶음으로 관리하는 기능.

- 이전에는 캠페인마다 동일한 단계(신청 → SNS 게시 → LINE 링크 전송)를 반복 입력
- 이제는 **번들 1번 만들어두면 캠페인 등록 시 드롭다운으로 선택**만 하면 됨
- 모집 타입(리뷰어/기프팅/방문형)별로 기본 번들 3개가 이미 시드되어 있음

### 핵심 원칙: **스냅샷 복사**
- 번들을 선택해서 캠페인을 저장하면 그 시점의 단계가 **캠페인에 복사되어 저장됨**
- 나중에 번들을 수정해도 **기존 캠페인은 영향받지 않음**
- 신규 캠페인부터 새 번들 내용이 적용됨
- 캠페인 폼에서 해당 캠페인만 **개별 단계 수정** 가능 (번들 원본은 그대로)

---

## 1-2. 기본 제공 번들 (시드 데이터)

초기 배포 시 아래 3개가 자동 등록됨:

| 번들명 | 모집 타입 | STEP 1 | STEP 2 | STEP 3 |
|---|---|---|---|---|
| 리뷰어 기본 | monitor | 신청 폼 제출 | 제품 사용 후 SNS 리뷰 게시 | LINE으로 게시 링크 전송 |
| 기프팅 기본 | gifting | 신청 폼 제출 | 제품 사용 후 SNS 리뷰 게시 | LINE으로 게시 링크 전송 |
| 방문형 기본 | visit | 신청 폼 제출 | 매장/팝업 방문 후 SNS 게시 | LINE으로 게시 링크 전송 |

각 STEP은 한국어·일본어 제목/설명을 모두 가지며, 인플루언서에게는 일본어가 노출됨.

---

## 1-3. 캠페인 등록 시 사용법

### 1-3-1. 번들 선택
1. 관리자 → **캠페인 등록/편집** 진입
2. **콘텐츠 가이드 섹션** 아래 "참여방법" 영역 확인
3. 드롭다운에서 모집 타입에 맞는 번들 선택
   - 예: 모집 타입이 `리뷰어`면 "리뷰어 기본"이 후보로 표시됨
4. 하단 미리보기에 STEP 1~N이 자동으로 채워짐

### 1-3-2. 이 캠페인만 부분 수정
- 특정 캠페인에서 STEP 2 설명만 바꾸고 싶은 경우:
  1. 번들 선택 후 STEP 2 제목/설명 인라인 편집
  2. 저장하면 **이 캠페인에만 적용**되는 스냅샷이 저장됨
  3. 번들 원본은 변경되지 않음

### 1-3-3. "번들 다시 불러오기" 버튼
- 개별 수정한 걸 원본 번들 내용으로 되돌리고 싶을 때 클릭
- 단, 저장 전에만 의미 있음 (저장 후에는 다시 번들 선택 → 저장해야 함)

---

## 1-4. 번들 관리 (기준데이터 → 참여방법 탭)

### 1-4-1. 권한
- **campaign_admin 이상**만 진입 가능
- campaign_manager는 조회 불가

### 1-4-2. 새 번들 등록
1. **기준데이터** 사이드 메뉴 → **참여방법** 탭
2. `+ 새 번들` 클릭
3. 입력:
   - **번들명** (한국어/일본어) — 관리자에게만 보임
   - **적용 모집 타입** — monitor/gifting/visit 복수 선택 가능
     - 아무것도 선택 안 하면 전 타입 공통 번들로 취급
   - **STEP 1~6** — 각 단계마다 한국어·일본어 제목/설명
     - STEP 최소 1개, 최대 6개 권장
4. **저장**

### 1-4-3. 수정
- 번들 수정은 **신규 캠페인부터 적용**. 기존 캠페인 스냅샷은 영향 없음.
- 단계 순서 변경은 드래그 또는 위/아래 버튼

### 1-4-4. 비활성(soft delete)

> 💬 **soft delete (소프트 삭제)**: 실제로 삭제하지 않고 "숨김" 처리하는 방식. 언제든 다시 활성화할 수 있어 안전. 반대는 hard delete (완전 삭제).

- 번들 목록 우측 토글로 `active` 끄기
- 비활성 번들은 캠페인 등록 드롭다운에서 숨겨짐
- 기존 캠페인 스냅샷에는 영향 없음 ✅

### 1-4-5. 삭제(hard delete)

> 💬 **hard delete (하드 삭제)**: DB에서 데이터를 완전히 제거. 복구 불가. soft delete(숨김 처리)와 달리 되돌릴 수 없음.

- 사용 중인(캠페인에 선택된 적 있는) 번들은 **원칙적으로 삭제 지양** — 비활성 처리 권장
- 삭제 시 `participation_set_id`는 `NULL`로 바뀌고, `participation_steps` 스냅샷은 유지
- → 기존 캠페인은 계속 정상 동작, 다만 "원본 번들로 돌아가기" 연결만 끊어짐

## 1-5. 자주 묻는 질문

### Q1. 이전에 만든 캠페인도 번들로 연결되나요?
**아니요.** 이전 캠페인은 기존 하드코딩 단계가 그대로 표시됩니다. 필요하면 해당 캠페인을 편집 화면에서 번들을 선택·저장하면 번들 기반으로 전환됩니다.

### Q2. 번들을 수정했더니 기존 캠페인도 바뀌나요?
**안 바뀝니다.** 캠페인 저장 시점에 단계가 복사되어 저장되므로, 번들을 나중에 수정해도 **이미 등록된 캠페인은 그대로** 유지됩니다. 이것이 "스냅샷" 방식의 핵심입니다.

### Q3. 모집 타입이 여러 개(예: 리뷰어+기프팅 동시)인 캠페인은?
- 번들 드롭다운에 **두 타입 모두에 태깅된** 번들만 표시됨
- 공통 번들을 만들고 싶으면 `recruit_types`에 `monitor, gifting` 둘 다 선택
- 또는 `recruit_types`를 비워두면 모든 타입에서 선택 가능

### Q4. STEP을 5개 이상 넣어도 되나요?
최대 6개까지 권장. 7개 이상도 저장은 되지만 모바일 레이아웃에서 과도하게 길어질 수 있으니 주의.

### Q5. 번들을 잘못 만들었는데 복구되나요?
- 활성 상태만 바꾼 거라면 다시 `active` 토글
- 내용을 덮어쓴 경우는 **복구 불가** → 중요한 번들은 변경 전에 스크린샷/백업 권장
- 기존 캠페인은 스냅샷이라 영향 없음 (재사용 원본만 영향)

### Q6. 인플루언서 화면에는 어떻게 보이나요?
캠페인 상세 페이지의 `참加方法` 섹션에 STEP 1~N이 순서대로 표시됨. 일본어만 노출.

---

## 1-6. 운영 주의사항

- **번들 원본 삭제 전에는 반드시 `active=false`로 먼저** — 예상치 못한 캠페인 편집 시 번들 재선택 불가 리스크 방지
- 번들명(name_ko)은 **유니크**. 중복 불가
- 번들 업데이트는 **개발서버에서 먼저 검증 후 운영 적용** 권장
- 캠페인 편집 중 번들 변경은 **기존 설정 덮어쓰기**이므로 주의

## 1-7. 문제 발생 시

| 증상 | 원인 | 해결 |
|---|---|---|
| 드롭다운에 번들이 안 보임 | `active=false` 또는 `recruit_types` 불일치 | 번들 활성화 + 타입 확인 |
| 저장 시 "번들명 중복" 에러 | `uq_participation_sets_name_ko` 위반 | 번들명 변경 |
| 캠페인 상세에 STEP이 안 보임 | `participation_steps` 스냅샷 없음 (이전 캠페인) | 캠페인 편집에서 번들 선택 후 재저장 |
| 번들 수정했는데 캠페인에 반영 안 됨 | 정상 동작 (스냅샷 방식) | 해당 캠페인 편집 → "번들 다시 불러오기" → 저장 |

# 2. 대시보드

> 위치: 관리자 페이지 로그인 직후 첫 화면 (왼쪽 사이드바 맨 위 **대시보드** 메뉴)
> 배포일: 기본 기능 (2026-04-15 문서화)

## 2-1. 개념 한눈에

대시보드는 **지금 우리 서비스가 어떻게 돌아가고 있는지 한눈에 보는 화면**입니다. 엑셀 요약표라고 생각하시면 됩니다.

- 캠페인이 몇 개인지
- 회원이 몇 명이고 최근 얼마나 늘었는지
- 신청이 얼마나 들어왔고 몇 건이 승인됐는지
- 가장 최근 신청자가 누구인지

매일 아침 업무 시작 전에 **잠깐 확인하면 상황 파악**이 됩니다.

---

## 2-2. 화면 구성 (위에서 아래로)

### ① KPI 카드 (4개 숫자)
화면 맨 위에 큰 숫자 카드 4개가 나열됩니다.

| 카드 | 의미 |
|---|---|
| **캠페인수** | 지금까지 등록된 전체 캠페인 개수 (상태 무관) |
| **인플루언서수** | 가입한 일본인 회원 총 수 |
| **신청수** | 누적 신청 건수 (승인·비승인 모두 포함) |
| **승인수** | 그중 관리자가 승인한 건수 |

> 💡 **"신청수 / 승인수" 비율**을 보면 승인률을 가늠할 수 있어요. 너무 낮으면 선정 기준이 타이트하거나, 너무 높으면 자격 조건이 느슨한 것일 수 있습니다.

### ② 캠페인 분포 카드
- **상태별**: 준비 / 모집예정 / 모집중 / 일시정지 / 종료 별로 몇 개씩 있는지
- **채널별**: Instagram / X / TikTok / YouTube / Qoo10 에 몇 개씩 올라가 있는지
  - 한 캠페인이 채널을 여러 개 선택한 경우(예: Instagram+X), 각 채널에 **중복 집계**됩니다. 그래서 합계가 총 캠페인수보다 많을 수 있음.

### ③ 회원가입 추이 차트
선(line) 그래프로 **날짜별 가입자 수**를 보여줍니다.
- 상단에 `7일 / 30일 / 전체` 버튼 — 클릭하면 기간 전환
- `오늘 가입` / `이번주 가입` 작은 숫자가 함께 표시됨
- 갑자기 치솟는 날이 있으면 광고·이벤트 효과 가능성

### ④ 프로필 완성률
회원들이 **어떤 정보를 얼마나 입력했는지** 퍼센트로 표시:
- Instagram 계정 입력률
- X / TikTok / YouTube 입력률
- 배송지 입력률
- PayPal 이메일 입력률

> ⚠️ 낮은 항목이 있으면 신청 시 걸림돌이 될 수 있습니다. 예: PayPal 입력률이 30%면 정산 가능한 회원이 적다는 뜻.

### ⑤ 최근 신청 테이블
화면 맨 아래 **가장 최근 신청 5~10건**이 목록으로 표시됩니다.
- 누가 어떤 캠페인에 신청했는지
- 언제 신청했는지
- 현재 상태(심사중 / 승인 / 비승인)
- 클릭하면 **신청 관리** 화면으로 바로 이동

---

## 2-3. 자주 확인하는 순서 (권장 루틴)

매일 아침 업무 시작 시:

1. **KPI 카드 4개** 훑어보기 — 어제까지 숫자가 정상 범위인지
2. **상태별 분포** 확인 — "모집예정" 캠페인이 곧 `active`로 넘어갈 시기인지
3. **회원가입 추이** 7일 탭 클릭 — 최근 가입 흐름 정상인지
4. **최근 신청 테이블** 확인 — 심사 대기 건수 누적 여부
5. 심사 대기가 많으면 바로 **신청 관리** 메뉴로 이동

---

## 2-4. 자주 묻는 질문

### Q1. 숫자가 실시간으로 반영되나요?
**거의 실시간**입니다. 페이지 새로고침(F5)하면 최신 데이터가 표시됩니다. 자동 업데이트는 되지 않으므로 오래 켜놓으셨다면 새로고침을 권장합니다.

### Q2. 채널별 분포 합계가 전체 캠페인 수보다 많아요. 오류인가요?
**정상입니다.** 한 캠페인이 Instagram·X·TikTok 여러 채널을 동시에 모집하면 각 채널에 1씩 추가로 집계되기 때문입니다.

### Q3. "프로필 완성률"이 낮을 때 어떻게 하나요?
회원에게 **마이페이지 입력을 독려**하는 게 근본 대책입니다. 또는 캠페인 신청 시 해당 정보가 없으면 자동으로 입력 화면으로 유도되도록 설계돼 있으니, 시간이 지나면 자연스럽게 올라갑니다.

### Q4. 대시보드에 표시되는 숫자가 실제와 다른 것 같아요.
- 먼저 브라우저 새로고침 (`Cmd+R` 또는 `F5`)
- 그래도 이상하면 개발팀(LINE `@reverb.jp`)에 스크린샷과 함께 연락

### Q5. 대시보드에서 캠페인이나 회원을 직접 추가할 수 있나요?
**아니요.** 대시보드는 **조회 전용**입니다. 등록·수정은 왼쪽 사이드바의 해당 메뉴(캠페인 관리, 관리자 계정 등)에서 진행합니다.

# 3. 캠페인 관리

<!-- NOTION:SYNCED 2026-04-15 -->

> 위치: 관리자 페이지 → 왼쪽 사이드바 **캠페인 관리** 메뉴
> 배포일: 기본 기능 (2026-04-15 문서화)

## 3-1. 개념 한눈에

캠페인 관리 메뉴는 **우리 서비스에 올라가는 모든 캠페인을 만들고 관리하는 곳**입니다.

실무에서 가장 많이 쓰는 메뉴이며, 여기서 하는 일은 크게 5가지입니다:

1. 새 캠페인 **등록**
2. 기존 캠페인 **수정**
3. 비슷한 캠페인 **복제** (템플릿처럼 활용)
4. 캠페인 **상태 변경** (준비 → 모집중 → 종료)
5. 목록 **순서 조정** (인플루언서 화면에 노출되는 순서)

> 💡 캠페인 등록·수정 폼 자체는 내용이 많아서 별도 섹션(§4 캠페인 등록/편집 폼)에서 다룹니다. 이 섹션은 **목록 화면에서 할 수 있는 일**을 다룹니다.

---

## 3-2. 목록 화면 구성

캠페인 관리 메뉴를 클릭하면 아래와 같이 **표(테이블) 형태**로 캠페인이 쭉 나열됩니다.

### 상단 필터 영역
- **상태 필터**: 드롭다운으로 특정 상태만 보기 (전체 / 준비 / 모집예정 / 모집중 / 일시정지 / 종료)
- **타입 필터**: 전체 / 리뷰어 / 기프팅 / 방문형
- **검색창**: 캠페인 제목 또는 브랜드명으로 검색

### 테이블 컬럼 (왼쪽에서 오른쪽)
| 컬럼 | 설명 |
|---|---|
| 썸네일 | 캠페인 대표 이미지 미리보기. 오른쪽에 `1/5` 같은 표시는 **등록된 이미지 장수** |
| 제목 | `[타입] 캠페인 제목` 형식. 클릭하면 **모바일 크기 미리보기**가 뜸 |
| 브랜드 | 광고주 브랜드명 |
| 조회 | 인플루언서가 캠페인 상세를 연 횟수 (▲▼로 정렬 가능) |
| 신청 | 누적 신청 건수. `3/10 명` 같이 **승인수/모집정원** 함께 표시됨. 옆에 대기(심사중) 배지가 붙을 수 있음 |
| 등록일 | 캠페인이 처음 등록된 날짜 (▲▼ 정렬) |
| 수정일 | 최근 수정된 날짜 (▲▼ 정렬) |
| 상태 | 드롭다운으로 바로 변경 가능 (아래 3-5 참고) |
| D-day | 게시마감·모집마감까지 남은 일수 (`D-3` 등) |
| 액션 | 복제·삭제 버튼 |

### 헤더 클릭 정렬
- **조회 / 신청 / 등록일 / 수정일** 컬럼은 헤더를 클릭하면 ▲(오름차순) ↔ ▼(내림차순) 전환
- 예: "신청 많은 순" 정렬하면 인기 캠페인이 위로 올라옴

---

## 3-3. 캠페인 미리보기

### 사용 방법
1. 목록에서 **캠페인 제목을 클릭**
2. 화면 중앙에 **모바일 크기 미리보기 모달**이 뜸
3. 인플루언서가 실제로 보는 화면과 동일한 레이아웃
4. 미리보기 안에서 **편집 버튼**을 누르면 바로 편집 화면으로 이동

### 언제 쓰나요?
- 등록·수정 후 실제 화면 확인
- 고객(광고주)에게 "이렇게 나옵니다" 공유 전에 확인
- 다른 캠페인 복제 전에 원본 내용 훑어볼 때

> 💡 모바일 미리보기는 **실제 서비스와 동일하게 렌더링**되므로 문구·이미지·단계 모두 최종 상태 그대로 나옵니다.

---

## 3-4. 캠페인 복제

### 사용 방법
1. 목록에서 복제할 캠페인 행의 **복제 버튼** 클릭
2. 동일한 내용으로 **새 캠페인이 생성**되며, 상태는 `준비(draft)`로 자동 설정
3. 제목에 `(복제)` 같은 표시가 붙을 수 있음 — 바로 편집해서 제목 수정

### 언제 쓰나요?
- 비슷한 브랜드의 다른 제품 캠페인
- 이전 시즌 캠페인을 재활용해서 새 시즌 시작
- 채널만 바꿔서 동일 제품 다시 모집

> ⚠️ **복제 후 반드시 확인할 항목**: 제목, 마감일, 슬롯 수(모집인원), 이미지. 실수로 이전 캠페인 정보가 남아있지 않은지 체크.

---

## 3-5. 캠페인 상태 (매우 중요)

캠페인은 **5가지 상태**를 가지며, 상태에 따라 인플루언서 화면 노출이 달라집니다.

### 상태 종류

| 상태 | 한국어 | 인플루언서에게 보이나요? | 언제 쓰나요? |
|---|---|---|---|
| `draft` | 준비 | **안 보임** | 등록 중·초안. 아직 공개할 준비 안 됨 |
| `scheduled` | 모집예정 | 보임 (`近日公開` 표시) | 모집 시작 전 미리 홍보 |
| `active` | 모집중 | 보임 (`募集中` 표시, 신청 가능) | 실제 신청 받는 기간 |
| `paused` | 일시정지 | **안 보임** | 긴급히 신청을 막아야 할 때 |
| `closed` | 종료 | 조건부 보임 (`募集締切` 오버레이) | 모집 끝. 신청 불가 |

### 상태 변경 방법
목록의 **상태 드롭다운을 클릭**해서 원하는 상태로 바로 변경. 저장 버튼 필요 없음.

### 상태 흐름 (권장 순서)
```
등록 → 준비(draft)
     ↓ 내용 확정
     모집예정(scheduled)  ← 홍보 시작
     ↓ 모집 시작일 도래
     모집중(active)  ← 신청 받기
     ↓ 모집 마감일 도래 or 정원 충족
     종료(closed)
```

### 🚨 자동 종료
- **마감일(deadline)이 지난 `active` 캠페인은 자동으로 `closed`로 바뀝니다** (새로고침 시)
- 관리자가 따로 신경 안 써도 됨

### 🚫 저장이 차단되는 경우
- **마감일이 이미 지난 캠페인**을 `모집중(active)` 또는 `모집예정(scheduled)` 상태로 저장하려고 하면 차단됨
- 해결: 마감일을 미래 날짜로 수정하거나, 상태를 `종료(closed)`로 두기

---

## 3-6. 캠페인 삭제

### 사용 방법
1. 목록에서 해당 캠페인 행의 **삭제 버튼** 클릭
2. **확인 모달**이 떠서 "정말 삭제하시겠습니까?" 물어봄
3. 확인하면 삭제

### ⚠️ 중요: 삭제 시 연쇄 삭제
- 캠페인을 삭제하면 해당 캠페인에 들어온 **모든 신청(applications)도 함께 삭제**됩니다
- 복구 불가
- **가급적 삭제 대신 `종료(closed)` 상태 변경을 권장**

### 언제 삭제해도 되나요?
- 잘못 등록해서 신청이 0건일 때
- 테스트용으로 만든 캠페인
- 어차피 재활용할 데이터가 없을 때

---

## 3-7. 목록 순서 조정 (순서변경 모드)

### 왜 필요한가요?
- 인플루언서 화면에서 **위에 뜨는 캠페인이 더 많이 보임**
- 프로모션 중요도에 따라 노출 순서 조정 필요

### 사용 방법
1. 캠페인 관리 페이지에서 **순서변경 버튼** 클릭 → 순서변경 모드 진입
2. 각 행 옆에 **↑↓ 드래그 핸들** 표시됨
3. 원하는 위치로 드래그
4. 다시 **순서변경 버튼** 클릭 → 저장 + 일반 모드로 복귀

> 💡 순서변경 모드 중에는 다른 작업(수정·삭제 등)은 불가. 순서만 집중해서 조정하세요.

---

## 3-8. 신청 상태 표시 이해하기

목록 "신청" 컬럼에 표시되는 것들:

- `3/10 명` = **승인 3건 / 모집정원 10명**
- 옆에 붙는 **작은 배지** = 심사 대기 중(pending) 건수
  - 예: `3/10 명 [대기 2]` → 승인 3, 대기 2, 미승인/모집정원 10

### 🚫 정원 초과 승인 차단
- 승인수가 정원(`slots`)에 도달하면 **승인 버튼이 자동 비활성**
- 실수로 초과 승인하는 사고 방지

---

## 3-9. 자주 묻는 질문

### Q1. 실수로 캠페인을 삭제했어요. 복구되나요?
**복구 불가**입니다. 삭제는 신중히 하시고, **모집이 끝난 캠페인은 `종료(closed)` 상태로 두면** 됩니다.

### Q2. 모집예정(scheduled) 상태에서도 신청이 들어오나요?
**아니요.** 인플루언서 화면에는 `近日公開`로만 표시되고 신청 버튼이 비활성화됩니다. 실제 신청은 `모집중(active)`일 때만 가능.

### Q3. 마감일이 지났는데 상태가 `active` 그대로인데요?
**새로고침하면 자동으로 `closed`**로 바뀝니다. 브라우저에서 `F5`.

### Q4. 캠페인 복제 후 이미지가 안 딸려왔어요.
복제는 **텍스트 정보 위주**이며, 경우에 따라 이미지는 다시 올려야 할 수 있습니다. 복제 후 반드시 이미지 탭 확인.

### Q5. 캠페인 순서를 바꿨는데 인플루언서 화면에 반영이 안 돼요.
- 순서변경 모드에서 **저장(다시 버튼 클릭)**을 꼭 눌렀는지 확인
- 인플루언서가 이미 해당 페이지를 켜놓은 상태면 **새로고침 후**에 반영됨

### Q6. "일시정지(paused)"와 "종료(closed)"의 차이는?
- **일시정지**: 잠깐 멈춤 (나중에 다시 `모집중`으로 복귀 가능)
- **종료**: 모집 완전히 끝. 되살리기 어려움. 마감일이 지난 자연스러운 종료에 주로 사용

---

## 3-10. 운영 주의사항

- **삭제보다 상태 변경 우선** — 데이터 보존이 원칙
- 캠페인 등록·수정 후 **반드시 미리보기로 확인** 후 상태를 `active`로 올리기
- 정원이 적은(slots=1~3) 캠페인은 **초과 승인 주의**
- 순서 조정은 **노출 효과에 영향**이 크므로 프로모션 전략과 연동

## 3-11. 문제 발생 시

| 증상 | 원인 | 해결 |
|---|---|---|
| 상태 드롭다운을 바꿨는데 저장 안 됨 | 마감일 경과된 캠페인을 `active`/`scheduled`로 변경 시도 | 마감일 먼저 수정 |
| 삭제 확인 모달이 안 뜸 | 브라우저 팝업 차단 | 브라우저 설정 확인 |
| 순서변경 모드에서 드래그가 안 됨 | 터치스크린·태블릿 환경 | PC 마우스 권장 |
| 복제 후 일부 필드 비어있음 | 원본 캠페인의 해당 필드가 애초에 비어있음 | 복제본 편집해서 채움 |
| 정원을 늘렸는데 승인 버튼이 여전히 비활성 | 페이지 캐시 | 새로고침 (`Cmd+R`) |

## 3-12. 권한 안내

- **campaign_manager**: 목록 조회·상태 변경은 가능, 삭제는 제한될 수 있음

# 4. 캠페인 등록/편집 폼

<!-- NOTION:SYNCED 2026-04-15 -->

> 위치: 캠페인 관리 목록 → `+ 새 캠페인` 버튼 (등록) / 기존 행 편집 버튼 (편집)
> 배포일: 기본 기능 (2026-04-15 문서화)

## 4-1. 개념 한눈에

캠페인 등록/편집 폼은 **인플루언서에게 노출될 모든 정보를 입력하는 핵심 화면**입니다.

폼은 **4개 섹션**으로 그룹핑돼 있으며, 순서대로 채우면 됩니다:

1. **기본정보** — 제목, 브랜드, 모집타입, 채널, 카테고리, 정원, 마감일 등
2. **제품정보** — 제품명, 상세 설명, 제품 URL, 이미지
3. **모집조건** — 최소 팔로워수, 콘텐츠 종류, 해시태그, 멘션
4. **콘텐츠가이드** — 설명, 어필 포인트, 촬영 가이드, NG사항 (리치 텍스트)

추가로 하단에 **참여방법** 영역이 있습니다 (§1 참조).

> 💡 폼 하단 **저장** 버튼을 누르기 전까지는 아무것도 반영되지 않습니다. 중간에 실수해도 취소하고 빠져나오면 원상태 유지.

---

## 4-2. 기본정보 섹션

### 입력 항목

| 항목 | 설명 | 필수 |
|---|---|---|
| **캠페인 제목** | 인플루언서에게 보이는 제목. 간결하고 명확하게 | ✅ |
| **브랜드명** | 광고주 브랜드 | ✅ |
| **캠페인 상태** | draft/scheduled/active/paused/closed (§3-5 참조) — 2026-04-16부터 **기본정보 섹션 상단**으로 이동됨 | ✅ |
| **모집 타입** | 리뷰어 / 기프팅 / 방문형 중 택 1 (라디오 버튼). 신규 폼 기본값 = 리뷰어(monitor) | ✅ |
| **채널** | 인스타/X/Qoo10/TikTok/YouTube 중 **복수 선택 가능** (체크박스) | ✅ |
| **채널 매칭 (or/&)** | 채널 2개 이상 선택 시만 노출 (아래 4-3 참고) | - |
| **카테고리** | 뷰티/패션/푸드 등. 기준데이터에서 관리 | ✅ |
| **모집 정원 (slots)** | 몇 명 뽑을 건지 | ✅ |
| **최소 팔로워수** | 선택 채널 중 1개 이상이 이 값을 충족해야 신청 가능 | - |
| **모집 마감일 (deadline)** | 신청 받는 마지막 날 | ✅ |
| **게시 마감일 (post_deadline)** | 승인된 인플루언서가 SNS 게시해야 하는 마감 | ✅ |
| **결과물 제출 마감일 (submission_end)** | 이 날짜 이후 결과물(이미지·URL) 제출 불가. 비우면 게시 마감일 폴백 | - |
| **당선 발표 안내 (winner_announce)** | 인플루언서 캠페인 상세에 표시되는 안내 문구. 기본값 `選考後、LINEにてご連絡` — 필요 시 일본어로 자유 수정 가능 (예: "5/1 인스타 공지", "당첨자 개별 메일 연락" 등) | - |

### 마감일 규칙
- **게시 마감일 ≥ 모집 마감일** 필수. 아니면 저장 차단
- **마감일이 이미 지난 날짜**면 `active`/`scheduled` 상태 저장 차단

### 💡 캠페인 상태 드롭다운 위치 변경 (2026-04-16)
기존에는 폼 **하단**에 있었지만, **기본정보 섹션 상단**으로 이동했습니다. 등록 시 상태를 먼저 정한 뒤 나머지를 채우는 순서가 자연스럽습니다.

## 4-3. 채널 매칭 (or/&) — 채널 복수 선택 시

### 무엇인가요?
채널을 2개 이상 선택하면 **인플루언서 상세 화면에 채널 구분자**를 어떻게 보여줄지 선택할 수 있습니다.

- `or` 선택 → `Instagram or TikTok` 으로 표시
- `&` 선택 → `Instagram & TikTok` 으로 표시

### 차이
- **`or`**: "둘 중 아무거나 OK" 느낌
- **`&`**: "둘 다 필요" 느낌

### ⚠️ 중요
**자격 심사 로직은 `or`와 `&` 모두 동일**합니다 (둘 중 하나라도 팔로워수 충족하면 신청 가능).
**표시만 다르고 실제 신청 조건은 동일**합니다. 헷갈리지 마세요.

## 4-6. 콘텐츠가이드 섹션 (리치 텍스트 에디터)

### 4개 필드

| 필드 | 용도 |
|---|---|
| **설명** | 캠페인 전체 안내 문구 |
| **어필 포인트** | 제품의 핵심 장점, 강조할 부분 |
| **촬영 가이드** | "이렇게 찍어주세요" 가이드 (각도, 배경 등) |
| **NG 사항** | 상세 NG (모집조건의 NG와 별개의 긴 설명) |

### 에디터 사용법
콘텐츠가이드 필드는 **Quill 리치 텍스트 에디터**입니다. Word처럼 서식 입력 가능:

> 💬 **Quill**: 웹에서 널리 쓰이는 글자 서식 편집기 라이브러리 이름. 사용자 입장에서는 그냥 "Word 스타일 편집기"라고 생각하시면 됩니다.
> 💬 **리치 텍스트(rich text)**: 굵게·이탤릭·리스트·링크 등 **서식이 있는 글자**. 반대는 plain text(평문, 서식 없는 글자).

- **볼드** / *이탤릭* / 리스트 / 링크 / 제목 / 인용
- 상단 **툴바**에서 선택
- **Notion에서 복사해서 붙여넣기**도 서식 유지됨 ✅

### ⚠️ 이미지 붙여넣기 주의
- Quill 에디터에 이미지를 직접 넣으면 **저장 시 자동 제거**됩니다
- 이미지는 반드시 **제품정보 섹션의 이미지 업로드 영역**을 사용
- 이유: 에디터 내 이미지는 base64로 저장돼 용량 폭증 위험
    - 💬 **base64**: 이미지 파일을 긴 문자열로 변환한 형태. 편집기 안에 넣으면 DB 용량이 수십 배로 불어날 수 있어 금지.

### 이전에 평문으로 등록된 캠페인
- 2026-04-13 이전에 등록된 캠페인은 서식 없는 평문일 수 있음
- 편집 시 자동으로 줄바꿈이 유지된 HTML로 변환됨 — 기존 내용 잃지 않음

## 4-7. 이미지 관리

### 업로드 방법
1. 제품정보 섹션의 **이미지 영역**에 파일 드래그 앤 드롭
2. 또는 영역 클릭 → 파일 선택 다이얼로그
3. 업로드 후 **크롭 모달**이 떠서 영역 조정 가능
4. 저장 시 자동으로 Supabase Storage에 저장됨
    - 💬 **Supabase Storage**: 우리 서비스가 쓰는 파일 저장소 (이미지·문서 보관 공간). 사용자는 그냥 "서버에 자동 저장된다"고 이해하면 충분.

### 제약
- **최대 9장** (img1~img8 + 썸네일)
- 첫 번째 이미지가 **대표 썸네일**로 사용됨
- 이미지 순서는 드래그로 재배열

### 권장 스펙
- 비율: **정사각형(1:1)** 또는 **세로형(4:5)** 추천
- 크기: 최대 2MB 권장 (자동으로 썸네일 변환됨)
- 형식: JPG / PNG / WebP

### ⚠️ 주의
- 이미지 업로드 중 페이지를 벗어나면 업로드가 중단될 수 있음
- 업로드 실패 시 다시 시도. 반복 실패 시 개발팀 문의

## 4-8. 참여방법 영역

콘텐츠가이드 아래 **참여방법** 영역이 있습니다. 여기서 캠페인 상세 페이지의 "참가방법 STEP 1~N"에 들어갈 단계를 선택·편집합니다.

### 사용 방법
1. 모집 타입에 맞는 **번들 드롭다운에서 선택** (예: 모집 타입이 `리뷰어`면 "리뷰어 기본" 번들이 후보로 표시됨)
2. 하단에 STEP 1~N이 자동 채워짐
3. 이 캠페인만 단계를 **인라인 수정** 가능 (STEP 제목/설명을 그대로 편집)
4. 저장 시 현재 STEP이 캠페인에 **스냅샷 복사**됨

### 스냅샷 원칙
- 저장 시점의 단계가 캠페인에 복사되므로, **나중에 번들을 수정해도 기존 캠페인은 영향 없음**
- 신규 캠페인부터 새 번들 내용 적용

### "번들 다시 불러오기" 버튼
- 인라인 수정한 걸 원본 번들 내용으로 되돌릴 때 클릭
- 저장 전에만 유효

### 기본 제공 번들
초기 설치 시 3개 등록돼 있음: **리뷰어 기본 / 기프팅 기본 / 방문형 기본** (각 3단계).

### 번들 자체 관리
번들 목록 편집·추가·순서 변경 등은 **기준데이터 → 참여방법 탭**에서 진행 ([§7-11 참여방법 번들 상세](#7-11-참여방법-번들-상세) 참조).

## 4-9. 저장 및 검증

### 저장 버튼
- 폼 하단의 **저장** 버튼 클릭
- 저장 중에는 로딩 스피너 표시
- 성공 시 토스트(우상단 알림) + 목록으로 자동 이동
    - 💬 **토스트(toast)**: 화면 모서리에 잠깐 떴다가 자동으로 사라지는 작은 알림 메시지. "저장되었습니다" 같은 안내가 보였다 사라지는 형태.

### 저장 전 자동 검증
아래 중 하나라도 해당하면 저장 차단:
- 필수 항목 누락 (제목, 브랜드, 모집타입, 채널, 카테고리, 정원, 마감일)
- **게시 마감일 < 모집 마감일** (순서 역전)
- **모집 마감일 < 오늘** 인데 상태가 `active`/`scheduled`

차단 시 **인라인 경고 메시지**가 해당 필드 옆에 표시됩니다.

### 저장 후 확인 루틴 (권장)
1. 캠페인 관리 목록으로 돌아가기
2. 해당 캠페인 **제목 클릭 → 미리보기 모달**로 실제 화면 확인
3. 이상 없으면 상태를 `모집예정` 또는 `모집중`으로 변경

## 4-10. 자주 묻는 질문

### Q1. 폼 입력 중에 페이지를 벗어나면 내용이 사라지나요?
**네, 사라집니다.** 긴 내용을 작성할 때는 **중간에 임시 저장** 권장. 상태를 `준비(draft)`로 두고 저장하면 내용이 보존됩니다.

### Q2. 채널 매칭 `or`와 `&`의 실제 차이가 있나요?
**표시만 다르고 신청 자격 조건은 동일**합니다. `&`로 해도 "반드시 두 채널 다 있어야" 되는 건 아닙니다. 자격 조건은 OR 방식 고정.

### Q3. Notion에서 콘텐츠가이드 내용을 복사했는데 서식이 깨져요.
대부분의 경우 Quill 에디터가 서식을 유지합니다. 깨진 부분만 에디터 안에서 다시 포맷해주세요. 표(table)는 Quill이 지원하지 않으니 리스트로 변환.

### Q4. 이미지를 9장보다 많이 올리고 싶어요.
**최대 9장 제한**입니다. 여러 제품 변종을 보여줘야 한다면 대표 이미지에 콜라주로 만들어 올리는 방법 권장.

### Q5. 제품 URL에 `http://` 없이 `example.com`만 입력해도 되나요?
**안 됩니다.** 반드시 `https://` 또는 `http://`로 시작해야 링크가 정상 동작합니다.

### Q6. 최소 팔로워수를 0으로 두면 어떻게 되나요?
**모든 인플루언서가 신청 가능**해집니다. 아무 제약 없이 모집하고 싶을 때 사용.

### Q7. 해시태그에 한자·일본어 써도 되나요?
**네, 됩니다.** 예: `化粧品`, `韓国コスメ`. 단, `#` 기호는 자동 붙으니 입력 금지.

### Q8. 저장 후 내용 수정하면 이미 신청한 인플루언서에게 알림이 가나요?
**아니요.** 수정 알림 기능은 아직 없습니다. 중요한 변경 시 LINE으로 별도 공지 권장.

## 4-11. 운영 주의사항

- **상태는 `준비(draft)`로 시작** → 내용 다 채우고 미리보기 확인 → `모집예정` 또는 `모집중`으로 변경
- **마감일은 여유 있게 설정** (너무 촉박하면 수정해야 할 때 차단될 수 있음)
- **이미지 업로드 후 즉시 저장 권장** (중간에 페이지 벗어나지 않기)
- 해시태그에 **일본 법규 태그(#PR, #広告) 필수 포함**
- 편집 시 채널 복수 선택에서 하나로 줄이면 `channel_match` 값은 기본 `or`로 리셋됨

# 5. 신청 관리

<!-- NOTION:SYNCED 2026-04-16 -->

> 위치: 관리자 페이지 → 왼쪽 사이드바 **신청 관리** 메뉴
> 배포일: 기본 기능 (2026-04-15 문서화)

## 5-1. 개념 한눈에

신청 관리는 **인플루언서가 캠페인에 신청한 내역을 확인하고 승인/거절 결정을 내리는 화면**입니다.

주요 업무:
1. 심사 대기(pending) 건 **확인**
2. 조건에 맞으면 **승인(approved)**, 아니면 **비승인(rejected)**
3. 실수로 처리한 건은 **되돌리기**로 pending 복귀
4. 누가 어떤 조건으로 신청했는지 **인플루언서 상세 프로필** 조회

---

## 5-2. 목록 화면 구성

### 상단 필터
- **캠페인 필터**: 특정 캠페인의 신청만 보기 (드롭다운, 썸네일 포함)
- **타입 필터**: 리뷰어 / 기프팅 / 방문형
- **상태 필터**: 전체 / 심사중 / 승인 / 비승인
- **검색창**: 인플루언서 이름·이메일 검색
- **정렬**: 최신순 / 오래된 순

### 테이블 컬럼
| 컬럼 | 설명 |
|---|---|
| 캠페인 썸네일·제목 | 어떤 캠페인에 대한 신청인지 |
| 인플루언서명 | 클릭 시 상세 프로필 모달 |
| 신청일 | 언제 신청했는지 |
| 대표 SNS | 선택한 메인 SNS + 팔로워 수 |
| 모집정원/승인/대기 | `3/10 (대기 2)` 형식. 캠페인 단위 요약 |
| 상태 | 심사중 / 승인 / 비승인 배지 |
| 액션 | 승인 · 비승인 · 되돌리기 버튼 |

---

## 5-3. 인플루언서 상세 모달

인플루언서명을 클릭하면 **상세 프로필 모달**이 뜸. 심사 전 필수 확인:

- 프로필: 이름(한자/가나), 거주지(일본 내 주소)
- SNS 계정: 각 채널별 ID + 팔로워 수 (링크 클릭 가능)
- 대표 SNS: 인플루언서가 설정한 메인 채널
- 메시지: 인플루언서가 신청 시 남긴 동기 메시지
- 배송지: 제품 배송 받을 주소 (리뷰어/기프팅만)
- PayPal 이메일: 정산용 (해당 캠페인이 유료라면)

### 팔로워 수 확인 팁
- 캠페인의 **최소 팔로워수** 기준을 자동으로 표시
- 미달인 경우 붉은색 표시 — 단, 시스템이 **신청 시점에 이미 차단**하므로 목록에 들어온 신청은 원칙적으로 조건 충족

---

## 5-4. 승인 처리

### 승인 방법
1. 목록에서 해당 행의 **승인 버튼** 클릭
2. 즉시 상태가 `승인(approved)`으로 변경
3. 인플루언서 마이페이지의 "응모이력"에 반영됨

### 🚫 정원 초과 승인 차단
- 이미 승인된 수가 `slots(정원)`에 도달하면 **승인 버튼이 자동 비활성** (회색)
- 알림 모달로 "모집 인원이 가득 찼습니다" 안내

### 승인 후 — 모집 타입별 흐름이 다릅니다

| 모집 타입 | 승인 후 인플루언서가 하는 일 | 결과물 종류 |
|---|---|---|
| **리뷰어(monitor)** | 제품 자비 구매 → **영수증 이미지 제출** | 영수증 이미지 |
| **기프팅(gifting)** | 무료 제품 수령 → 사용 후 SNS 리뷰 게시 → **게시 URL 제출** | 게시물 URL |
| **방문형(visit)** | 매장/팝업 방문 → 현장 **이미지 + SNS 게시 URL 제출** | 이미지 + 게시물 URL |

> ℹ️ **2026-04-16 매핑 정정**: 커밋 `a128738` 에서 역방향으로 잘못 구현된 매핑을 비즈니스 상식대로 복구했습니다 (리뷰어=영수증, 기프팅=URL).

- 인플루언서에게는 **LINE 또는 별도 메일**로 당첨 안내를 운영팀에서 수동 전달 (자동 알림 기능 없음)
- 결과물 검수는 **결과물 관리** 메뉴(§10)에서 진행

### 🚫 응모 차단 규칙 — 모집 타입별 차이

| 모집 타입 | 응모 차단 조건 | 이유 |
|---|---|---|
| **리뷰어(monitor)** | 신청 수 ≥ 모집 정원(slots) 시 **신규 응모 차단** | 전원 승인 = 전원 구매이므로 초과 응모 무의미 |
| **기프팅(gifting)** | 초과 응모 **허용** | 심사 후 선별하므로 다다익선 |
| **방문형(visit)** | 초과 응모 **허용** | 심사 후 선별하므로 다다익선 |

> 💡 **리뷰어 캠페인**은 정원이 차면 인플루언서 화면에서 신청 버튼이 비활성됩니다. **기프팅/방문형**은 정원을 초과해도 신청 가능하지만, 관리자가 승인할 수 있는 수는 정원까지입니다.

---

## 5-5. 비승인 처리

### 비승인 방법
1. 해당 행의 **비승인 버튼** 클릭
2. 상태가 `비승인(rejected)`로 변경

### 비승인 사유
시스템상 사유 입력 필드는 없음. 필요 시 LINE으로 별도 안내 권장.
예시 사유: 팔로워 수 미달, 게시물 품질 낮음, 콘텐츠 방향 불일치, 중복 신청 등

---

## 5-6. 되돌리기 (pending 복귀)

### 언제 쓰나요?
- 실수로 승인/비승인을 눌렀을 때
- 재검토가 필요할 때

### 방법
1. 승인 또는 비승인된 행의 **되돌리기 버튼** 클릭
2. 상태가 `심사중(pending)`으로 복귀
3. `reviewed_by`, `reviewed_at` 기록은 초기화

> ⚠️ 승인 건을 되돌리면 정원 카운트에서도 차감됩니다. 인플루언서에게 이미 당첨 안내를 전달했다면 별도 연락 필요.

---

## 5-7. 자주 묻는 질문

### Q1. 신청한 인플루언서에게 자동으로 알림이 가나요?
**아니요.** 승인/비승인 처리 시 자동 알림 기능은 없습니다. 별도 LINE/메일로 운영팀이 연락.

### Q2. 동일한 인플루언서가 같은 캠페인에 중복 신청할 수 있나요?
**아니요.** 시스템이 중복 신청을 차단합니다.

### Q3. 정원을 넘어선 대기 건은 어떻게 하나요?
- 승인 수가 정원에 도달하면 나머지 대기 건은 **비승인 처리 권장**
- 또는 모집 기간 내 승인자가 이탈하면 대기 중 1명 추가 승인 가능

### Q4. 한번 삭제된 신청은 복구되나요?
**복구 불가.** 삭제보다 `비승인` 처리가 원칙.

### Q5. 심사 대기 건수가 많을 때 효율적으로 처리하려면?
- **상태 필터를 `심사중`으로** 고정
- **캠페인별로 그룹 처리** — 한 캠페인의 신청을 한번에 몰아서 심사

---

## 5-8. 운영 주의사항

- **승인 전 상세 프로필 꼭 확인** — 팔로워 수만 보지 말고 실제 콘텐츠 품질도 체크
- **정원 초과 승인 불가**이므로 우선순위 높은 인플루언서부터 처리
- 되돌리기는 **이미 당첨 안내 전달 전에만** 안전
- 비승인 사유를 기록해두고 싶다면 내부 메모(구글 시트 등) 활용 권장

---

## 5-9. 문제 발생 시

| 증상 | 원인 | 해결 |
|---|---|---|
| 승인 버튼이 회색으로 비활성 | 정원 도달 | 다른 신청을 되돌리거나 정원(slots) 늘리기 |
| 인플루언서 상세 모달에 일부 정보 비어있음 | 인플루언서가 해당 항목 미입력 | 신청 자체는 유효. 필요 시 LINE으로 보완 요청 |
| 되돌리기 후 정원 카운트가 안 줄어듦 | 페이지 캐시 | 새로고침 (`Cmd+R`) |
| 필터 적용 후 목록이 비어있음 | 조건에 맞는 신청이 실제 없음 | 필터 초기화 후 재확인 |

## 5-10. 권한 안내

- 모든 관리자 권한이 **조회**는 가능
- **승인/비승인/되돌리기**는 campaign_admin 이상 (조직 설정에 따라 campaign_manager도 가능)

# 6. 인플루언서 관리

> 배포일: 기본 기능 (2026-04-15 문서화)

## 6-1. 개념 한눈에

가입한 **일본 인플루언서 회원들의 목록을 보고 상세 프로필을 조회**하는 메뉴입니다.

주요 용도:
- 어떤 회원이 있는지 전체 파악
- 특정 채널 인플루언서만 골라서 확인 (예: Instagram 10만 이상)
- 특정 회원의 SNS·주소·PayPal 등 정보 확인
- 마케팅 리스트 추출 (엑셀 복사 등)

---

## 6-2. 목록 화면 구성

### 상단 필터
- **채널 필터**: Instagram / X / TikTok / YouTube / Qoo10 (계정이 있는 회원만)
- **검색창**: 이름·이메일
- **정렬**: 가입 최신순 / 팔로워 많은 순

### 테이블 컬럼
| 컬럼 | 설명 |
|---|---|
| 이름 | 한자 + 가나 (かな) |
| 이메일 | 가입 이메일 |
| 대표 SNS | 본인이 선택한 메인 채널 + 팔로워 수 |
| 가입일 | 언제 가입했는지 |
| 프로필 완성도 | 기본정보/SNS/배송지/PayPal 각각 ✅ or ❌ |
| 신청 이력 | 이 회원이 신청한 총 건수 |
| 상세 | 클릭 시 상세 프로필 모달 |

## 6-3. 인플루언서 상세 프로필 (조회)

회원명 클릭 시 상세 모달이 뜹니다. 확인 가능 정보:

### 기본 정보
- 이름(한자/가나), 생년월일, 성별, 거주지(우편번호·도도부현·시구정촌)
- 전화번호, 가입일

### SNS 계정
- Instagram / X / TikTok / YouTube / Qoo10 각각의 ID + 팔로워 수
- 프로필 링크 클릭하면 실제 SNS 페이지 열림
- **대표 SNS** 표시

### 배송지
- 제품 받을 주소 (우편번호 + 전체 주소)

### 결제 정보
- PayPal 이메일 (정산용)
- 은행 계좌 관련 컬럼은 deprecated (미사용)

### 동의 기록
- 약관 동의일시, 개인정보 동의일시, 마케팅 수신 동의 여부

## 6-4. 자주 묻는 질문

### Q1. 회원 정보를 엑셀로 내보낼 수 있나요?
현재 UI에서는 **별도 export 기능 없음**. 필요 시 개발팀에 요청(DB에서 직접 추출).

### Q2. 특정 회원을 관리자 쪽에서 삭제/탈퇴시킬 수 있나요?
시스템 UI에서는 불가능. 본인 탈퇴 요청 또는 개발팀 수동 처리 필요.

### Q3. 회원의 정보를 관리자가 직접 수정할 수 있나요?
**아니요.** 조회 전용입니다. 수정은 본인이 마이페이지에서 진행해야 합니다.

### Q4. "프로필 완성도"가 낮은 회원은 어떻게 하나요?
회원 본인이 입력해야 합니다. 별도 LINE으로 입력 독려 권장.

### Q5. 가장 팔로워 많은 인플루언서 순으로 보려면?
헤더의 **팔로워 컬럼 ▲▼** 클릭해서 정렬.

# 7. 기준데이터 관리

<!-- NOTION:SYNCED 2026-04-16 -->

> 위치: 관리자 페이지 → **기준데이터** (URL: `/admin#lookups`)
> 배포일: 기본 기능 (2026-04-15 문서화) / 반려사유 탭 추가 (2026-04-16)

## 7-1. 개념 한눈에

기준데이터 = **캠페인 등록 폼·반려 모달 등에서 선택지로 나타나는 항목들의 원본**입니다.

예: 캠페인 폼에서 "채널" 드롭다운에 나오는 Instagram/X/TikTok 등 — 이게 전부 기준데이터에 등록된 것.

### 관리 대상 6종
| 종류 | 설명 |
|---|---|
| **채널** | Instagram / X / TikTok / YouTube / Qoo10 등 |
| **카테고리** | 뷰티 / 패션 / 푸드 / 라이프 등 |
| **콘텐츠 종류** | 게시물 / 릴스 / 스토리 / 영상 등 |
| **NG 사항** | 게시물 금지 내용 (경쟁사 언급, 과장 표현 등) |
| **참여방법** | §1 참여방법 번들 (별도 페이지 참조) |
| **반려사유** | 결과물 반려 모달의 템플릿 (§10 결과물 관리에서 사용) |

각 항목은 **한국어·일본어 두 언어**로 관리됩니다.

## 7-2. 탭 구성

기준데이터 메뉴 진입 시 상단에 6개 탭이 있습니다:
- 채널 / 카테고리 / 콘텐츠 종류 / NG 사항 / 참여방법 / 반려사유

### 공통 UI (참여방법 제외)
- **한국어 이름** / **일본어 이름** 2열 입력
- **순서 (sort_order)** — 드롭다운 등에 표시되는 순서
- **활성/비활성 토글** — 비활성 시 선택지에서 숨김
- **삭제** 버튼

### 채널 탭 전용
- **적용 모집 타입** 체크박스 — monitor / gifting / visit 복수 선택 가능 **(최소 1개 필수)**
- 예: "방문형 전용 채널"을 만들려면 visit만 체크

### 반려사유 탭 전용
- **한국어 이름** = 관리자 드롭다운에 표시되는 이름 (예: "PR 태그 누락")
- **일본어 이름** = 반려 모달에서 사유란에 **자동 채워지는 일본어 문구** (인플루언서에게 전달됨)
- **적용 모집 타입** 체크박스 — monitor / gifting / visit 선택 가능 **(선택 사항, 빈 값 허용)**
  - 체크하지 않으면 "공통" 배지 — 모든 캠페인의 반려 모달에 표시
  - 특정 타입만 체크하면 해당 타입 캠페인의 반려 모달에만 표시
- 예: "게시물 삭제/비공개"는 gifting/visit만 체크 — 리뷰어 캠페인에는 안 뜸

### 기본 제공 반려사유 (5종 시드 + "기타")
| 한국어 이름 | 적용 타입 | 용도 |
|---|---|---|
| PR 태그 누락 | gifting, visit | #PR/#広告/#プロモーション 미포함 (SNS 게시물 전용) |
| 이미지 불선명 | monitor, visit | 영수증·현장 이미지 품질 부족 |
| 금액/날짜 불일치 | monitor | 영수증 정보 확인 불가 |
| 게시물 삭제/비공개 | gifting, visit | SNS 게시물 접근 불가 |
| 해시태그/멘션 누락 | gifting, visit | SNS 게시물의 필수 태그·멘션 누락 |
| 기타 | — (시스템 고정) | 자유 입력용. DB에 없는 옵션 |

## 7-11. 참여방법 번들 상세

### 개념

**참여방법 번들** = 캠페인 상세 페이지의 "참가방법 STEP 1~N" 영역을 재사용 가능한 묶음으로 관리하는 기능.

- 이전: 캠페인마다 동일한 단계(신청 → SNS 게시 → LINE 링크 전송) 반복 입력
- 현재: **번들 1번 만들어두면 캠페인 등록 시 드롭다운으로 선택**
- 모집 타입(리뷰어/기프팅/방문형)별로 기본 번들 3개가 이미 시드되어 있음

### 핵심 원칙: 스냅샷 복사
- 번들을 선택해서 캠페인을 저장하면 그 시점의 단계가 **캠페인에 복사되어 저장**
- 나중에 번들을 수정해도 **기존 캠페인은 영향받지 않음**
- 신규 캠페인부터 새 번들 내용이 적용됨
- 캠페인 폼에서 해당 캠페인만 **개별 단계 수정** 가능 (번들 원본은 그대로)

### 기본 제공 번들 (시드 데이터)

| 번들명 | 모집 타입 | STEP 1 | STEP 2 | STEP 3 |
|---|---|---|---|---|
| 리뷰어 기본 | monitor | 신청 폼 제출 | 제품 사용 후 SNS 리뷰 게시 | LINE으로 게시 링크 전송 |
| 기프팅 기본 | gifting | 신청 폼 제출 | 제품 사용 후 SNS 리뷰 게시 | LINE으로 게시 링크 전송 |
| 방문형 기본 | visit | 신청 폼 제출 | 매장/팝업 방문 후 SNS 게시 | LINE으로 게시 링크 전송 |

각 STEP은 한국어·일본어 제목/설명을 모두 가지며, 인플루언서에게는 일본어만 노출.

### 새 번들 등록
1. 기준데이터 → **참여방법** 탭
2. `+ 새 번들` 클릭
3. 입력: 번들명(한/일), 적용 모집 타입(복수 선택 가능, 비우면 공통), STEP 1~6(각 한/일 제목·설명)
4. 저장

### 수정
- 신규 캠페인부터 적용. 기존 캠페인 스냅샷은 영향 없음
- 단계 순서 변경은 드래그 또는 위/아래 버튼

### 비활성(soft delete)
- 목록 우측 토글로 `active` 끄기
- 캠페인 등록 드롭다운에서 숨겨짐
- 기존 캠페인 스냅샷은 유지 ✅

### 삭제(hard delete)
- 사용 중인 번들은 원칙적으로 **삭제 지양** — 비활성 권장
- 삭제 시 `participation_set_id`는 `NULL`로 바뀌고, `participation_steps` 스냅샷은 유지
- 기존 캠페인은 정상 동작, 다만 "원본 번들로 돌아가기" 연결만 끊어짐

### 자주 묻는 질문

**Q1. 이전에 만든 캠페인도 번들로 연결되나요?**
아니요. 이전 캠페인은 기존 하드코딩 단계가 그대로 표시됩니다. 필요하면 해당 캠페인 편집에서 번들 선택·저장하면 번들 기반으로 전환됩니다.

**Q2. 번들을 수정했더니 기존 캠페인도 바뀌나요?**
안 바뀝니다. 캠페인 저장 시점에 단계가 복사되어 저장되므로, 번들을 나중에 수정해도 이미 등록된 캠페인은 그대로 유지됩니다.

**Q3. 모집 타입이 여러 개(예: 리뷰어+기프팅)인 캠페인은?**
번들 드롭다운에 두 타입 모두에 태깅된 번들만 표시. 공통 번들을 만들고 싶으면 `recruit_types`를 비우거나 여러 타입을 체크.

**Q4. STEP을 5개 이상 넣어도 되나요?**
최대 6개까지 권장. 7개 이상도 저장은 되지만 모바일 레이아웃에서 길어질 수 있음.

### 운영 주의사항
- **번들 원본 삭제 전 반드시 `active=false`로 먼저** — 예상치 못한 캠페인 편집 리스크 방지
- 번들명(name_ko)은 **유니크**, 중복 불가
- 번들 업데이트는 **개발서버에서 먼저 검증 후 운영 적용** 권장
- 캠페인 편집 중 번들 변경은 **기존 설정 덮어쓰기**이므로 주의

### 문제 발생 시

| 증상 | 원인 | 해결 |
|---|---|---|
| 드롭다운에 번들이 안 보임 | `active=false` 또는 `recruit_types` 불일치 | 번들 활성화 + 타입 확인 |
| 저장 시 "번들명 중복" 에러 | 유니크 제약 위반 | 번들명 변경 |
| 캠페인 상세에 STEP이 안 보임 | 스냅샷 없음 (이전 캠페인) | 캠페인 편집 → 번들 선택 후 재저장 |
| 번들 수정했는데 캠페인에 반영 안 됨 | 정상 동작 (스냅샷 방식) | 해당 캠페인 편집 → "번들 다시 불러오기" → 저장 |

# 8. 관리자 계정

<!-- NOTION:SYNCED 2026-04-15 -->

> 위치: 관리자 페이지 → 왼쪽 사이드바 **관리자 계정** (super_admin 전용 메뉴)
> 배포일: 기본 기능 (2026-04-15 문서화)

## 8-1. 개념 한눈에

관리자 계정 메뉴는 **super_admin만 접근 가능**합니다. 여기서 할 수 있는 일:

1. 새 관리자 **초대**
2. 기존 관리자 **권한 변경**
3. 관리자 **삭제** (2가지 방식)
4. 관리자 **비밀번호 재설정 도움**

---

## 8-2. 권한 3단계

| 권한 | 역할 |
|---|---|
| `super_admin` | 전권. 관리자 초대·삭제·권한 변경 포함 |
| `campaign_admin` | 캠페인·기준데이터·신청 관리 가능. 관리자 메뉴 진입 불가 |
| `campaign_manager` | 캠페인·신청 조회·승인 중심. 기준데이터 수정 불가 |

---

## 8-3. 관리자 초대 (가장 중요)

### 흐름
1. **초대 버튼** 클릭 → 이메일, 이름, 역할 입력
2. 시스템이 자동으로:
   - `auth.users` + `auth.identities` 생성
   - `admins` 테이블에 등록
   - 초대 받는 사람에게 **비밀번호 설정 메일** 자동 발송
3. 받는 사람이 메일 링크 클릭 → **직접 새 비밀번호 설정** → 로그인 가능

### ✅ 이 방식의 장점
- super_admin이 **비밀번호를 직접 지정하지 않음** (보안)
- 이메일 유효성이 **자동 검증**됨 (오타 메일로 가입 방지)
- 받는 사람이 원하는 비밀번호로 설정 가능

### 기존 인플루언서 계정을 관리자로 승격
- 이미 인플루언서로 가입한 이메일을 초대하면 → **자동으로 관리자 권한 추가** (기존 프로필 유지)

---

## 8-4. 관리자 삭제 (2가지 방식)

삭제 시 **2가지 중 선택** 모달이 뜸:

### 방식 1: 권한만 해제 (권장)
- `admins` 테이블에서만 제거
- **인플루언서 계정·데이터는 유지**됨
- "관리자는 그만두지만 인플루언서로는 남음" 시나리오

### 방식 2: 완전 삭제
- `auth.users` + `admins` + 관련 `applications` + `receipts` + `influencers` 모두 cascade 삭제
    - 💬 **cascade 삭제(연쇄 삭제)**: 한 항목을 지우면 그에 딸린 관련 데이터까지 **한 번에 함께 삭제**되는 방식. 예: 회원 삭제 시 그 회원의 신청·영수증도 모두 함께 사라짐.
- 복구 불가
- **데이터 프라이버시 요청(회원탈퇴) 시 사용**

### 🚫 자기 자신 삭제 차단
- 본인 계정은 삭제 불가 (시스템이 차단)

---

## 8-5. 비밀번호 재설정 (super_admin 도움)

관리자가 비밀번호를 잊어버렸을 때 2가지 옵션:

### 옵션 1: 메일 재설정 링크 (권장)
- 관리자 행의 액션 메뉴 → **"메일로 재설정"**
- 본인이 메일 링크로 새 비밀번호 설정

### 옵션 2: 수동 비밀번호 지정
- 모달에서 8자 이상 새 비밀번호 입력
- 시스템에서 즉시 설정
- **자기 자신 비밀번호는 이 방식으로 못 바꿈** — 본인은 §9 "내 계정"에서

---

## 8-6. 자주 묻는 질문

### Q1. 초대 메일이 안 왔어요.
스팸/프로모션 함 확인. 그래도 없으면 개발팀(LINE `@reverb.jp`)에 문의. Brevo(메일 발송 서비스) 이슈일 수 있음.

> 💬 **Brevo**: 프랑스의 이메일 발송 대행 서비스. 우리 서비스가 비밀번호 재설정·초대 메일 등을 보낼 때 사용. 현재 **Starter 플랜 (월 20,000건)** 구독 중. 이 회사 서버가 장애나면 메일이 늦어질 수 있음.

### Q2. super_admin을 2명 만들어도 되나요?
**가능**합니다. 오히려 권장 — 혼자 있을 때 계정 잠기면 복구 어려움.

### Q3. campaign_manager와 campaign_admin의 차이?
**manager는 기준데이터 수정 불가, admin은 가능.** 캠페인 등록·수정도 admin만 가능 (조직 설정에 따라).

### Q4. "완전 삭제" 후 다시 같은 이메일로 초대 가능?
**가능합니다.** 완전 삭제 후 auth 레코드가 사라지므로 새 가입처럼 진행 가능.

### Q5. 관리자 권한 변경(승격/강등)은?
목록에서 **역할(role) 드롭다운** 변경 후 저장. super_admin으로 승격도 가능하나 신중히.

---

## 8-7. 운영 주의사항

- **super_admin은 최소 2명 유지 권장** (단독 잠김 방지)
- **삭제보다 "권한 해제" 우선** — 데이터 보존
- 퇴사자는 **즉시 권한 해제** 필요 (내부 정보 유출 방지)
- 초대 시 이메일 오타 확인 (실제로 받아볼 수 있는 주소인지)

---

## 8-8. 문제 발생 시

| 증상 | 원인 | 해결 |
|---|---|---|
| 초대 메일 안 옴 | Brevo 이슈 or 스팸 차단 | 스팸함 확인, 개발팀 문의 |
| 가입 시 "email rate limit exceeded" | 시간당 이메일 발송 한도(100건/h) 초과 | 잠시 후 재시도. 반복 시 개발팀에 한도 상향 요청 |
| "자기 자신은 삭제 불가" 메시지 | 본인 계정 삭제 시도 | 다른 super_admin이 삭제 |
| 메일 재설정 링크 만료 | 링크 유효기간 경과 | 다시 재설정 링크 발송 |
| 승격했는데 권한 반영 안 됨 | 해당 관리자가 이미 로그인 중 | 재로그인 요청 |

---

## 8-9. 권한 안내

- **super_admin만** 이 메뉴 진입 가능
- campaign_admin, campaign_manager는 메뉴 자체가 보이지 않음

3. 새 비밀번호 입력 (8자 이상)
4. 저장

### 비밀번호 정책
- **8자 이상**
- **Supabase HIBP 검사** — 유출된 비밀번호 사용 시 차단
- 대문자·숫자·특수문자 조합 권장 (필수는 아님)
- **보이기/숨기기 토글** 있어서 오타 확인 가능

---

## 9-4. 자주 묻는 질문

### Q1. 비밀번호를 잊어버렸어요.
로그인 화면의 **"비밀번호를 잊으셨나요?"** 링크 → 이메일 입력 → 재설정 메일로 복구

### Q2. 이메일 주소를 바꾸고 싶어요.
시스템에서 직접 변경 불가. **super_admin에게 요청** → 새 이메일로 다시 초대받기 권장.

### Q3. "보이기" 눌러도 비밀번호가 안 보여요.
브라우저 확장 프로그램(비밀번호 관리자)과 충돌 가능. 시크릿 모드에서 시도.

### Q4. 비밀번호 변경 후 다른 기기에서는?
다른 기기의 기존 세션은 당분간 유효할 수 있지만, 토큰 만료 후 재로그인 필요.

---

## 9-5. 운영 주의사항

- 비밀번호는 **3~6개월마다 변경 권장**
- 같은 비밀번호를 다른 서비스와 공유 금지
- 공용 PC에서 로그인 후 반드시 **로그아웃**

- 자기 자신의 권한 레벨·이메일은 변경 불가 (보안)

# 10. 결과물 관리

<!-- NOTION:SYNCED 2026-04-16 -->

> 위치: 관리자 페이지 → 왼쪽 사이드바 **결과물 관리** 메뉴 (URL: `/admin#deliverables`)
> 배포일: 2026-04-16

## 10-1. 개념 한눈에

결과물 관리는 **인플루언서가 제출한 이미지·게시물 URL을 검수하는 화면**입니다.

캠페인에 승인된 인플루언서는 활동관리에서 결과물을 **초안으로 추가 → 한꺼번에 제출**하고, 관리자가 여기서 확인 후 승인/반려 처리합니다.

### 결과물 종류 2가지

| 종류 | 제출 대상 캠페인 | 인플루언서가 제출하는 것 |
|---|---|---|
| **이미지** | 리뷰어(monitor) / 방문형(visit) | 영수증 이미지 또는 현장 사진 |
| **게시물 URL** | 기프팅(gifting) / 방문형(visit) | SNS 게시 URL (Instagram/X/TikTok 등) |

> 💡 **방문형(visit)**은 이미지 + 게시물 URL 둘 다 제출합니다. 나머지는 한 종류만 제출.

### 인플루언서의 제출 흐름 (초안 → 제출 2단계)

```
1. 활동관리 화면에서 "리스트에 추가" 버튼
   → 결과물이 "초안(draft)" 상태로 저장 (관리자 화면엔 보이지 않음)
   → 항목에 "未提出" 배지 + 삭제 버튼 표시

2. 여러 항목을 계속 추가 (이미지 여러 장, URL 여러 개 가능)
   → 필요 없는 초안은 삭제 버튼으로 제거 가능

3. "관리자에게 제출" 버튼
   → 해당 신청의 모든 초안이 일괄 pending(검수대기) 상태로 전환
   → 이 순간부터 관리자 결과물 관리에 보임
```

### 핵심 흐름

```
인플루언서: 초안 추가 → 일괄 제출(pending)
     ↓
관리자가 "결과물 관리"에서 검수
     ↓
승인 → 완료 / 반려 → 사유와 함께 인플루언서에게 알림 → 재제출 가능
```

> ⚠️ **관리자에게는 draft(초안)가 보이지 않음** — 인플루언서가 "관리자에게 제출" 버튼을 눌러야 관리자 화면 목록에 나타납니다.

---

## 10-2. 화면 구성

### 상단 필터 영역

| 필터 | 옵션 | 기본값 |
|---|---|---|
| **타입** | 전체 / 이미지 / 게시물 URL | 전체 |
| **상태** | 검수대기 / 승인 / 반려 / 전체 | **검수대기** |
| **캠페인** | 전체 / 특정 캠페인 선택 | 전체 |
| **검색** | 인플루언서 이름으로 검색 | — |

> 💡 기본 상태가 **검수대기**로 설정되어 있어, 화면 진입 시 바로 처리 필요한 건만 보입니다.

### 테이블 컬럼

| 컬럼 | 설명 |
|---|---|
| 타입 | 이미지 / 게시물 URL 배지 |
| 캠페인 | 캠페인명 + 브랜드명 |
| 인플루언서 | 이름 + 이메일 |
| 제출일 | 최초 제출 날짜 (오래된 순 정렬 = 방치 방지) |
| 검수일 | 관리자 처리 날짜 (미처리 시 "—") |
| 상태 | 검수대기 / 승인 / 반려 배지 |
| 처리 | **상세** 버튼 → 모달 열기 |

### 정렬
- **검수대기**: 오래된 순 (방치 우선 처리)
- **그 외 상태**: 최근 처리 순
- 제출일·검수일·상태 컬럼 헤더 클릭으로 ▲▼ 정렬 가능

---

## 10-3. 상세 모달

테이블에서 **상세** 버튼을 클릭하면 모달이 열립니다.

### 이미지(receipt) 표시 내용
- 이미지 (클릭 시 원본 보기)
- 메모 (선택사항)
- 제출일 / 최종 수정일
- 한 신청에 **여러 이미지**가 있으면 각각 개별 행으로 표시됨

> 💡 리뷰어(monitor) 캠페인은 **영수증 이미지**, 방문형(visit) 캠페인은 **현장 사진** 중심으로 제출됩니다.

### 게시물 URL(post) 표시 내용
- 채널명 (Instagram/X/TikTok 등)
- 게시물 URL (클릭 시 새 탭에서 열림)
- 제출 이력 — 재제출이 있으면 날짜별 목록 표시

### 변경 이력 타임라인
모달 하단에 **상태 변경 이력**이 시간순으로 표시됩니다:
```
제출      · pending → pending · 2026-04-10
반려      · pending → rejected · PR 태그 누락 · 2026-04-11
재제출    · rejected → pending · 2026-04-12
승인      · pending → approved · 2026-04-13
```

### 반려된 결과물
반려 사유가 모달 상단에 빨간 박스로 표시됩니다. 인플루언서 활동관리 화면에도 같은 사유가 표시됩니다.

---

## 10-4. 승인 처리

### 방법
1. 상세 모달에서 내용 확인
2. **승인** 버튼 클릭
3. 즉시 처리 → 토스트 알림 + 모달 닫힘 + 목록 새로고침

### 확인 포인트
- **이미지**: 이미지가 선명한지, 제품/현장이 명확히 보이는지
- **게시물 URL**: 링크가 유효한지, 게시물에 **#PR / #広告 / #プロモーション** 태그가 포함되어 있는지

---

## 10-5. 반려 처리

### 방법
1. 상세 모달에서 **반려** 버튼 클릭
2. 반려 사유 입력 화면 표시
3. **템플릿 선택** 또는 **직접 입력**
4. **반려 처리** 클릭

### 반려 사유 템플릿 (기준데이터에서 동적 로드)

반려 사유 템플릿은 **기준데이터 관리 → 반려사유 탭**에서 관리됩니다 (§7-2 참고). 기본 제공 5종:

| 템플릿 | 적용 모집 타입 | 자동 채워지는 일본어 문구(개요) |
|---|---|---|
| PR 태그 누락 | 기프팅·방문형 | #PR/#広告/#プロモーション 중 1개 누락 안내 |
| 이미지 불선명 | 리뷰어·방문형 | 영수증/현장 이미지 품질 부족, 선명한 사진 재제출 요청 |
| 금액/날짜 불일치 | 리뷰어 | 영수증 금액/구매일 확인 불가 안내 |
| 게시물 삭제/비공개 | 기프팅·방문형 | SNS 게시물 삭제/비공개 전환 안내 |
| 해시태그/멘션 누락 | 기프팅·방문형 | 필수 해시태그·멘션 미포함 안내 |
| 기타 (직접 입력) | — | 위에 해당 안 될 때 자유 입력 |

### 🎯 모집 타입별 자동 필터링
- **리뷰어(monitor) 캠페인**의 결과물(영수증): SNS 관련 템플릿(PR 태그 누락, 게시물 삭제/비공개, 해시태그/멘션 누락)은 표시 안 됨
- **기프팅(gifting) 캠페인**의 결과물(URL): 영수증 관련 템플릿(이미지 불선명, 금액/날짜 불일치)은 표시 안 됨
- **방문형(visit) 캠페인**의 결과물: 이미지·URL 둘 다 있으니 "금액/날짜 불일치"를 제외한 대부분 표시됨

> 💡 템플릿을 선택하면 **일본어 문구가 사유란에 자동 채워지고** (인플루언서에게 그대로 전달됨), **추가 수정도 가능**합니다. 상황에 맞게 내용을 보완해주세요.
>
> 💡 새 반려 사유가 필요하면 기준데이터 관리 → 반려사유 탭에서 추가할 수 있습니다 (campaign_admin 이상).

### 반려 후
- 인플루언서 활동관리 화면에 **빨간 배너**로 반려 사유가 자동 표시됨
- 인플루언서가 **재제출**하면 상태가 자동으로 **검수대기(pending)**로 복귀
- 관리자가 다시 검수하면 됨

---

## 10-6. 되돌리기

### 언제 쓰나요?
- 실수로 승인 또는 반려를 눌렀을 때
- 재검토가 필요할 때

### 방법
1. 승인/반려된 결과물의 상세 모달 열기
2. **되돌리기 (검수대기로)** 버튼 클릭
3. 확인 모달 → "인플루언서가 이미 결과를 확인했을 수 있습니다" 안내
4. 확인 클릭 → 상태가 **검수대기(pending)**로 복귀

> ⚠️ 승인 후 인플루언서에게 이미 완료 알림이 간 경우, 되돌리면 혼선이 생길 수 있으니 **신속하게 처리**해주세요.

---

## 10-7. 동시 처리 충돌 방지

> 💬 **낙관적 락(optimistic lock)**: 두 명이 동시에 같은 데이터를 수정하려 할 때, 먼저 저장한 쪽만 성공하고 뒤늦은 쪽은 차단하는 방식. "충돌 시 알려주고 새로 읽어오기"가 핵심.

### 무슨 상황인가요?
관리자 A와 B가 **동시에 같은 결과물을 검수**하면:
1. A가 먼저 승인 처리 → **성공**
2. B가 뒤이어 반려 처리 시도 → **"다른 관리자가 이미 처리했습니다"** 토스트 표시
3. B의 화면이 자동 새로고침 → 최신 상태 확인

### 대응 방법
- 토스트가 뜨면 **목록이 자동으로 새로고침**되니 새 상태를 확인하면 됩니다
- 의도적으로 덮어쓸 수 없음 (시스템이 차단)
- 팀원 간 **캠페인별로 담당을 나누면** 충돌 최소화

---

## 10-8. 캠페인 진행현황에서 결과물 확인

> 위치: 캠페인 관리 → 캠페인 클릭 → **신청자 보기**

신청자 테이블에 **결과물 상태 요약** 컬럼이 있습니다:
- `승인 1 · 검수대기 1` — 상태별 건수 색상 표시
- `완료` 배지 — 해당 신청자의 결과물이 모두 승인됨

### 완료 판정 기준

| 모집 타입 | 채널 매칭 | 완료 조건 |
|---|---|---|
| 리뷰어(영수증만) | — | 영수증 1장 승인이면 완료 |
| 기프팅(URL만) | or | 1개 채널이라도 URL 승인이면 완료 |
| 기프팅(URL만) | & | 선택 채널 **각각** 최소 1개 URL 승인 필요 |
| 방문형(이미지+URL) | or | 이미지 1장 + 1개 채널 URL 승인이면 완료 |
| 방문형(이미지+URL) | & | 이미지 1장 + 선택 채널 **각각** 1개 URL 승인 필요 |

> 예: 기프팅 캠페인 채널이 `Instagram & TikTok`이면, 인플루언서가 두 채널 모두 게시물을 올리고 **각각 승인**받아야 완료 배지가 붙습니다.

### OT 발송 체크박스
같은 화면에 **OT 발송** 체크박스도 있습니다:
- 기프팅/방문형 캠페인의 **승인된 신청 건만** 활성
- 체크하면 발송 날짜 기록, 해제 시 확인 모달
- 리뷰어 캠페인은 "—" 표시 (OT 개념 없음 — 자비 구매 후 영수증 제출만)

## 10-9. 자주 묻는 질문

### Q1. 인플루언서가 결과물을 제출하면 알림이 오나요?
**별도 알림은 없습니다.** 정기적으로 결과물 관리 화면에 들어가서 검수대기 건을 확인해주세요.

### Q2. 반려 후 인플루언서가 재제출하면 어떻게 되나요?
자동으로 **검수대기(pending)** 상태로 돌아옵니다. 결과물 관리 목록에 다시 나타나므로 재검수하면 됩니다.

### Q3. "다른 관리자가 이미 처리했습니다" 메시지가 자주 떠요.
**같은 결과물을 2명이 동시에 검수**하면 발생합니다. 캠페인별로 담당자를 나누거나, 한 명씩 처리하세요.

### Q4. 게시물 URL을 클릭했는데 페이지가 없어요.
인플루언서가 게시물을 **삭제하거나 비공개로 전환**한 경우입니다. "게시물 삭제/비공개" 템플릿으로 반려 처리하세요.

### Q5. 제출 이미지가 너무 작아서 안 보여요.
모달에서 이미지를 **클릭하면 원본 크기**로 열립니다. 그래도 불선명하면 "이미지 불선명" 템플릿으로 반려하세요.

### Q7. 인플루언서가 "리스트에 추가했는데 관리자 화면에 안 보여요"라고 합니다.
초안(draft) 상태라 그렇습니다. 인플루언서에게 **"관리자에게 제출"** 버튼을 눌러야 검수대기(pending)로 전환되어 관리자 화면에 보인다고 안내해주세요.

### Q8. 방문형 캠페인인데 이미지만 제출된 경우 승인해도 되나요?
**방문형은 이미지 + URL 둘 다 필요**합니다. 둘 다 제출됐는지 확인 후 승인하세요. URL이 없으면 "URL 제출 요청" 사유로 반려 권장.

### Q6. 캠페인 진행현황의 "완료" 배지가 안 뜨는데요?
채널 매칭이 `&`(AND)인 캠페인은 **모든 선택 채널에서 각각 결과물이 승인**되어야 완료입니다. 어떤 채널의 결과물이 빠졌는지 확인해주세요.

---

## 10-10. 운영 주의사항

- **검수대기 건은 매일 확인** — 기본 필터가 검수대기이니 화면 열기만 하면 됨
- **PR 태그 확인 필수** — 일본 스테스 마케팅 규제 위반 방지 (#PR, #広告, #プロモーション 중 1개)
- 반려 시 **사유를 구체적으로** — 인플루언서가 무엇을 고쳐야 하는지 명확히
- 되돌리기는 **알림 전달 전에** — 이미 완료 안내를 보낸 후 되돌리면 혼선
- 동시 처리 충돌이 잦으면 **캠페인별 담당 분담** 권장

|---|---|---|
| 목록이 비어있음 | 필터가 좁음 (검수대기인데 건이 없음) | 상태 필터를 "전체"로 변경 |
| "다른 관리자가 이미 처리했습니다" | 동시 처리 충돌 | 목록 새로고침 후 최신 상태 확인 |

| 반려 사유가 인플루언서에게 안 보임 | 인플루언서가 새로고침 안 함 | 인플루언서에게 새로고침 안내 |

# 11. 알림 시스템

<!-- NOTION:SYNCED 2026-04-16 -->

> 위치: 인플루언서 앱 기능. **관리자 UI에도 알림 벨 표시됨** (관리자 계정이 인플루언서로도 가입할 수 있기 때문 — 2026-04-16 수정).
> 배포일: 2026-04-15 (Stage 6) / 관리자 표시 지원 (2026-04-16) / 30초 폴링 추가 (2026-04-16)

## 11-1. 개념 한눈에

결과물 검수 처리 시 **인플루언서에게 자동으로 알림이 발송**되는 시스템입니다.

관리자가 별도로 조작할 필요 없이, **승인/반려/되돌리기 버튼만 누르면 DB 트리거가 자동으로 알림을 생성**합니다.

### 관리자가 알아야 하는 이유
- "인플루언서가 왜 반려 사실을 알고 있지?" → 자동 알림으로 이미 전달됨
- "승인 후 되돌렸는데 안내해야 하나?" → 시스템이 자동 알림 발송
- 알림은 **삭제 불가**이므로 신중하게 처리

---

## 11-2. 알림 종류 (3가지)

| 종류 | 언제 발송 | 아이콘 색상 | 본문 |
|---|---|---|---|
| **결과물 반려** | 검수대기/승인 → 반려 시 | 빨강 | 캠페인명 + 반려 사유(관리자가 입력한 일본어 문구) |
| **심사 결과 변경** | 승인 → 검수대기/반려 (되돌리기) | 주황 | 캠페인명 + "관리자에 의해 결과가 변경되었습니다. 마이페이지에서 확인" |
| **결과물 승인** | (예약 - 현재 미사용) | 초록 | — |

> 💡 **승인 알림은 아직 비활성**입니다. 당첨 안내는 현재 LINE/메일로 수동 전달하는 방식이 유지됩니다.

---

## 11-3. 인플루언서 화면에서의 표시

- **햄버거 메뉴 → 通知**에 미읽음 개수 배지 (최대 `9+`)
- 클릭 시 **슬라이드업 모달**이 아래에서 올라옴 (최대 30건, 최신순)
- 미읽음 알림은 연한 주황색 배경
- 알림 클릭 → 해당 결과물의 활동관리 페이지로 이동 + 읽음 처리
- 하단 **"すべて既読"** 버튼으로 전체 읽음 처리
- **배지 자동 갱신**: 30초마다 폴링 + 탭 포커스 복귀 시 즉시 갱신 (2026-04-16 추가)

### 관리자 계정도 알림을 봅니다 (2026-04-16 수정)
관리자 계정도 **인플루언서로 응모할 수 있으므로**, 관리자 UI에도 알림 벨이 표시됩니다. 본인이 응모한 캠페인의 결과물 알림만 **본인에게만** 보입니다 (RLS로 타인 알림 차단).

## 11-4. 관리자 액션과의 연동

| 관리자 액션 | 자동 발생하는 일 |
|---|---|
| 결과물 **반려** 버튼 | 인플루언서에게 "반려" 알림 발송 + 활동관리 빨간 배너 표시 |
| 결과물 **승인** 버튼 | 알림 **없음** (LINE/메일로 수동 전달 필요) |
| 결과물 **되돌리기** (승인→pending) | "심사 결과 변경" 알림 발송 |
| 결과물 **되돌리기** (반려→pending) | "심사 결과 변경" 알림 발송 |
| 인플루언서 **재제출** | 해당 결과물의 **미읽음 반려 알림이 자동 읽음 처리됨** (UI 깨끗하게) |

## 11-5. 자주 묻는 질문

### Q1. 관리자에게도 알림이 오나요?
**관리자 계정이 인플루언서로도 응모했다면 옵니다.** 2026-04-16부터 관리자 UI에도 알림 벨이 표시됩니다. 단, RLS로 **본인 알림만** 보이므로 타 인플루언서 알림은 보이지 않습니다. 관리자 전용 알림(신청 심사 필요 등)은 현재 없습니다.

### Q2. 승인 시 알림이 안 가면 인플루언서는 어떻게 알죠?
- 마이페이지 **"응모이력"**에서 상태 `승인`으로 표시됨
- 운영팀이 **LINE 또는 메일로 당첨 안내 수동 전달** (현재 방식)
- 추후 Stage에서 승인 알림도 자동화 예정

### Q3. 잘못 반려 처리한 경우, 알림 취소할 수 있나요?
**알림 자체는 취소 불가.** 하지만 되돌리기를 하면 "심사 결과 변경" 알림이 자동 추가되어 인플루언서가 혼선 없이 파악 가능합니다.

### Q4. 인플루언서가 알림을 못 받았다고 합니다.
- 인플루언서 앱을 **새로고침**하도록 안내 (배지 지연 가능)
- 햄버거 메뉴 → 通知에서 직접 확인 요청
- 정말 미발송이면 개발팀 문의 (DB 트리거 확인)

### Q5. 알림 내용을 관리자가 볼 수 있나요?
현재 UI로는 **볼 수 없음**. DB 레벨에서는 관리자가 조회 가능하지만 별도 메뉴는 없음. 필요 시 개발팀에 요청.

---

## 11-6. 운영 주의사항

- **반려 사유 = 알림 본문**: 관리자가 입력한 사유(또는 템플릿의 일본어 문구)가 인플루언서 알림으로 그대로 전달됩니다. **정중하고 명확하게 작성** 필수.
- **되돌리기는 신중히**: 한 번 되돌리면 "심사 결과 변경" 알림이 발송되므로, 인플루언서가 혼란스러울 수 있습니다.
- **승인 알림은 수동 전달 유지**: 승인만 자동 알림이 없으므로, LINE/메일 안내를 빠뜨리지 않도록 주의.
- **알림 삭제 불가**: 실수로 잘못된 사유를 입력해도 알림은 영구 보관됩니다 (되돌리기로 보완만 가능).

## 11-7. 문제 발생 시

| 증상 | 원인 | 해결 |
|---|---|---|
