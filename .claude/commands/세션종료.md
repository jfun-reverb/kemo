# 세션종료

REVERB JP에서 작업을 마치고 dev 브랜치로 PR을 생성하는 단축키. 전시장(worktree)이든 메인 폴더든 **feature 브랜치이기만 하면** 동작한다(보호 브랜치 `dev`/`main`에서만 안내 후 멈춤). dev 브랜치로 작품을 옮기는 「감독 호출」.

## 비유

전시장에서 가구 조립이 끝났습니다. 감독님(PR 리뷰)에게 "본 거실로 옮겨도 될까요?" 묻기 위해 PR을 생성합니다. 머지 자체는 사용자가 직접 GitHub에서 확인 후 진행 (안전 장치).

## 입력

`/세션종료 [PR 제목]`

- PR 제목은 선택. 없으면 현재 브랜치명에서 자동 생성.

## 메인 Claude 가 수행할 절차

### Step 0 — 위치·브랜치 감지 후 분기 (막지 않고 알맞게 동작)

```bash
pwd
git rev-parse --show-toplevel
git branch --show-current
git worktree list
```

**핵심: "막는다"가 기본이 아니다. 어디서 호출하든 상황에 맞는 PR 흐름을 고른다. 위험한 경우(보호 브랜치)만 멈춘다.** 아래 표로 분기:

| 상황 | 현재 폴더 | 현재 브랜치 | 동작 |
|---|---|---|---|
| **A. 전시장 흐름** | 메인 폴더 아님(worktree) | `feature/*` | 기존 전체 흐름 (Step 1~3 + Step 3.5 메인 복귀) |
| **B. 메인 폴더 흐름** | 메인 폴더 | `feature/*` | 안전 가드 확인 후 PR (Step 1~3, **Step 3.5 생략** — 메인 폴더라 복귀 불필요) |
| **C. 보호 브랜치** | 무관 | `dev` / `main` | PR 자동화 안 함 — 아래 안내 분기 |
| **D. 임의 폴더 + feature** | 메인도 worktree도 아님 | `feature/*` | B에 준해 진행하되 현재 위치 한 줄 안내 |

#### B(메인 폴더 + feature) 안전 가드 — 필수

메인 폴더에서 호출되면 거부하지 말고 진행하되, **commit 전에 반드시** 보여주고 확인받는다 (실험·임시 브랜치 오발행 방지):
- `git status --short` 변경 파일 목록 + 대상 표기: `head=현재 feature 브랜치 → base=dev`
- `AskUserQuestion`: "이 변경들을 dev로 올리는 PR을 만들까요? / 아니요(중단)"
- ⚠️ **커밋하지 않기로 한 실험 브랜치**(예: `feature/ios-app` 처럼 미커밋이 쌓인 장기 실험)면 사용자가 여기서 중단할 수 있게 한다. 확인 없이 자동 commit 절대 금지.

#### C(보호 브랜치 = dev/main) 안내 분기 — 막되 다음 행동 제시

- "보호 브랜치(`{브랜치명}`)에서는 `/세션종료`로 PR을 자동 생성하지 않습니다."
- 이어서 무엇인지 `AskUserQuestion` 으로 묻기:
  - "거버넌스 문서(규칙·커맨드·메모리 등) 변경 → dev 직접 커밋이 맞습니다 (이 커맨드 불필요, `.claude/rules/git.md` 예외)"
  - "코드 변경 → 새 feature 브랜치로 옮겨 PR을 만들까요?" (선택 시 `git switch -c feature/{슬러그}` 후 B 흐름)

### Step 1 — 변경 사항 확인

```bash
git status --short
git diff --stat
```

#### 1a. 변경 없음 + 미푸시 커밋 없음
"변경 사항이 없습니다. 작업한 것이 없으면 그냥 worktree를 정리해도 됩니다." 안내 후 종료.

#### 1b. 미커밋 변경 있음
사용자에게 commit 메시지 묻기 (`AskUserQuestion`로 옵션 또는 직접 텍스트):
- 옵션 1: "이번 변경 한 줄 요약을 적어주세요" 직접 입력
- 옵션 2: 자동 제안 (변경 파일 패턴으로 추측, 예: `feat(admin): {SLUG} 작업`)

빌드 필요 여부 자동 점검:
- `dev/` 폴더 안 파일이 변경되었으면 `cd dev && bash build.sh` 자동 실행 (`.claude/rules/build.md` 패턴) → 루트 `index.html`/`admin/index.html` 재빌드
- 빌드 산출물도 `git status`에 잡히면 함께 add

reverb-reviewer 에이전트 자동 호출 → 검수 후 commit:
```bash
git add <변경된 파일들>
git commit -m "{사용자 입력 메시지}"
```

#### 1c. 미푸시 커밋만 있음
바로 push 단계로.

### Step 2 — push

```bash
git push -u origin {현재 브랜치}
```

처음 push면 `-u`로 upstream 설정. 이미 설정됐으면 그냥 push.

### Step 3 — PR 생성

PR 제목·본문 자동 작성:
- 제목: 사용자 인자로 받았으면 그대로, 없으면 브랜치명·최근 커밋에서 추출
- 본문: 최근 커밋 메시지 + "## Test plan" 체크리스트 (`gh pr create` HEREDOC 패턴)
- base: `dev` (운영 직접 PR은 보호. dev 머지 후 별도 main PR로)
- head: 현재 브랜치

```bash
gh pr create \
  --base dev \
  --head feature/{SLUG} \
  --title "{제목}" \
  --body "..."
```

생성 결과 PR URL을 사용자에게 안내.

### Step 3.5 — 메인 폴더로 복귀 (A 분기 전용 — EnterWorktree 진입 세션 정리)

**B(메인 폴더 흐름)에서는 이 단계를 건너뛴다 — 이미 메인 폴더이므로 복귀 불필요.** A(전시장 흐름)일 때만 수행한다.

PR 생성 후, 이 세션이 `/새세션`의 `EnterWorktree` 진입으로 전시장에 들어와 있던 경우 메인 폴더로 되돌린다:
- `ExitWorktree` 도구를 `action: "keep"` 으로 호출 (전시장 폴더·브랜치는 디스크에 그대로 두고 세션만 메인 폴더로 복귀 — PR 머지 전이라 삭제하면 안 됨)
- 이 세션이 EnterWorktree로 진입한 세션이 **아니면**(예: 새 터미널에서 직접 `cd` 로 들어온 경우) 이 호출은 **아무 일도 하지 않는다(no-op)** — 그대로 두면 된다.
- 복귀 후에는 같은 터미널에서 곧바로 `/새세션 다음작업` 으로 이어갈 수 있다.

### Step 4 — 사용자 안내 (한국어 + 다음 단계, 이모지 사용 안 함)

```
[감독 호출 완료] PR이 만들어졌습니다.

PR 주소: {URL}

(이 세션이 /새세션으로 진입했던 경우, 지금 메인 폴더로 돌아왔습니다.
 같은 터미널에서 바로 /새세션 다음작업 으로 이어갈 수 있습니다.)

다음 단계:
  1. 위 주소에서 PR 내용 확인
  2. 문제 없으면 「Merge pull request」 버튼 클릭 (직접)
  3. 머지가 끝나면 메인 폴더에서 전시장 철거:

     cd ~/Documents/projects/reverb-jp
     git worktree remove ~/Documents/projects/reverb-jp-{SLUG}
     git branch -D feature/{SLUG}
     git pull origin dev

[주의] 운영서버(main) 배포는 별도 단계입니다.
       dev에서 검증 끝나면 dev → main PR을 따로 만들어야 합니다.
```

### Step 5 — 운영 배포 확인 (선택)

dev에 머지된 후 운영 배포 여부는 `AskUserQuestion`으로 묻기:
- "지금 운영 배포까지 진행 (dev → main PR + 머지)"
- "개발서버 검증 후 결정 (추천)"
- "보류"

명시 지시가 있을 때만 자동 진행. 그 외에는 사용자 안내만.

## 실패·예외 처리

- `gh` 미설치/미인증: "GitHub CLI(`gh`)가 필요합니다. 설치 또는 인증 후 재시도하세요." 안내
- 빌드 실패: 에러 그대로 전달 + 사용자가 코드 수정 후 다시 `/세션종료`
- reviewer 차단 이슈 발견: 자동 commit 안 함. 사용자에게 정정 요청
- push 거부 (force needed): 절대 force 안 함. 사용자에게 상황 설명 + 수동 대응 요청

## 사용 시점

- ✅ 전시장(worktree) 안에서 작업이 끝났을 때 (A 흐름)
- ✅ **메인 폴더에서 feature 브랜치로 작업을 끝냈을 때** (B 흐름 — 변경 확인 후 PR. 더 이상 막지 않음)
- ✅ 중간 PR을 만들고 추가 검토받고 싶을 때 (작업 계속)
- ⚠️ 보호 브랜치(`dev`/`main`)에서는 PR 자동 생성 안 함 — 거버넌스 문서는 직접 커밋, 코드는 feature 브랜치로 이동 (C 흐름이 안내)
- ⚠️ 커밋 안 하기로 한 실험 브랜치(예: `feature/ios-app`)면 B 흐름의 확인 단계에서 중단할 것
