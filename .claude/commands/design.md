## 1단계: 요구사항 분석

**무엇을 하려는지 정리:**
- 기능의 목적과 해결하려는 문제
- 대상 사용자: 인플루언서? 관리자? 둘 다?
- 기존 기능과의 관계: 신규 추가 / 기존 수정 / 기능 삭제

**질문을 통해 요구사항 구체화:**
- 애매한 부분은 반드시 사용자에게 질문
- 기능 범위(scope) 확인: MVP로 최소한 구현할 것 vs 나중에 추가할 것
- 우선순위: 핵심 기능 vs nice-to-have 구분

---

## 2단계: 영향도 분석

**현재 프로젝트 상태와의 충돌/영향 체크:**

### 기술 영향도
- DB 스키마: 기존 테이블(campaigns, influencers, applications, admins) 수정 필요 여부
- 새 테이블/컬럼 추가 시: RLS 정책, 인덱스, 트리거 영향
- localStorage 폴백(DEMO_MODE) 대응 필요 여부
- build.sh 수정 필요 여부 (새 CSS/JS 파일 추가 시)

### UI/UX 영향도
- 인플루언서 앱: 모바일 480px 안에서 구현 가능한지
- 관리자 앱: PC 레이아웃에 어떻게 배치할지
- 네비게이션: 햄버거 메뉴/사이드바에 새 항목 추가 필요한지
- 기존 페이지/화면 흐름 변경 여부

### 데이터 영향도
- 기존 데이터 마이그레이션 필요 여부
- 캠페인-신청-인플루언서 관계에 미치는 영향
- 데이터 삭제 시 연쇄 삭제(cascade) 고려

### 비즈니스 영향도
- 일본 시장 특성 고려 (일본어 UI, 일본식 주소/이름/은행 체계)
- 인플루언서 경험: 신청 과정이 복잡해지지 않는지
- 관리자 경험: 운영 워크플로우에 맞는지

---

## 3단계: 리서치 및 벤치마크

**필요 시 조사:**
- 유사 플랫폼(체험단, 인플루언서 마케팅)의 구현 방식
- 일본 시장의 UX 관행 및 사용자 기대
- 기술적 제약 확인: vanilla JS + Supabase로 구현 가능한 범위
- Supabase 기능 활용: Auth, Storage, Realtime, Edge Functions 등

---

## 4단계: 설계 제안

**구체적인 설계안 제시:**

### 화면 설계
- 어떤 페이지/화면이 필요한지
- 각 화면의 주요 요소와 인터랙션
- 화면 흐름(유저 플로우) 정리
- 일본어 UI 텍스트 초안 (상태명, 버튼명, 안내문구 등)

### DB 설계
- 테이블/컬럼 변경 사항
- RLS 정책 설계
- 마이그레이션 SQL 초안

### 파일 구조
- 수정할 파일 목록 (dev/ 경로)
- 새로 추가할 파일 (있다면)
- build.sh 변경 필요 여부

### 구현 단계
- 작업을 단계별로 나누기 (각 단계가 독립적으로 동작하도록)
- 1단계: DB 스키마 → 2단계: 백엔드 함수 → 3단계: UI 구현 → 4단계: 빌드/테스트

---

## 5단계: 리스크 및 대안

**잠재적 문제와 대안 제시:**
- 기술적 리스크: 복잡도, 성능, 호환성
- 사용자 리스크: UX 혼란, 학습 비용
- 데이터 리스크: 마이그레이션 실패, 기존 데이터 손상
- 대안: 더 간단한 접근법, 단계적 출시 방안

---

## 6단계: 확인 및 합의

**설계안을 정리하여 사용자에게 제시:**
- 구현 범위 요약 (무엇을 하고, 무엇을 하지 않을지)
- 예상 수정 파일 목록
- DB 변경사항 요약
- 주요 결정사항 정리

**사용자 승인 후 다음 단계 안내:**
- `/feature <기능명>` 또는 `/admin-feature <기능명>`으로 구현 시작
- `/migration <설명>`으로 DB 변경 먼저 적용
- 단계별로 `/build` + 확인 반복
