# /서비스점검 — 서비스 전체 건강도 점검 (Service Health Audit)

> 기능이 많아지는 서비스가 "전체가 잘 작동하는가 / 불필요·충돌 코드는 없는가 / 정책·문서와 맞는가"를 주기적으로 점검하는 플레이북. 고문 세션이 "에이전트 팀"을 점검하듯, 이 명령은 "서비스 자체"를 점검한다.
>
> 사양서: `docs/specs/2026-06-02-service-health-audit.md`

## 모드

- 인자 없음 = **빠른 점검** — 차원 2(죽은·중복 코드) + 5(문서 정합성)만. 10~20줄 리포트. 월 1회 권장.
- `심층` = **풀스윕** — 전체 차원(1~9). 시간이 걸리더라도 빠짐없이.
- 특정 차원만: `/서비스점검 4,6` 처럼 차원 번호 지정 가능.

### 풀스윕 정기 실행 트리거 (필수 — "있는데 안 돌려서 놓침" 방지, 2026-06-23)

플레이북이 있어도 **실행을 강제하는 트리거가 없으면 안 돌려서 놓친다**(@cosme 채널 중복이 차원 2-A 예시에까지 적혀 있었는데도 못 잡은 게 그 증거). 아래 시점엔 고문 세션이 풀스윕을 **반드시** 돌린다:
- **베타 각 단계 출시 직전** (1차 인플 핵심 / 2차 브랜드 포털 / 3차 iOS 앱 — `project_beta_launch` 참조)
- **분기 1회** (직전 풀스윕일 `docs/health/` 최신 파일로 경과 판단)
- **주요 기능 3개 이상 머지가 누적**됐거나 DB 스키마·기준 데이터가 바뀐 직후
- ⚠️ **단일 자원 조율**: 차원 1·9는 브라우저 1개(Playwright)·운영 DB를 쓴다 → 다른 세션이 안 쓰는 걸 확인 + 단일 세션에서. 동시 실행 금지(`.claude/rules/multi-session.md`).

### 차원 2 정합성 미러 점검 (3개 하위 — 2026-06-02 추가)

전수조사(죽은 코드·잔존 참조)로는 안 잡히는 유형. 각 부분은 정상인데 데이터 drift·타이밍에서 깨진다.

- **2-A 하드코딩 미러 ↔ DB 정합**: 코드에 DB 데이터를 하드코딩한 상수(`CHANNEL_LABEL_FALLBACK` 류 라벨·상태·카테고리 미러)가 `lookup_values` 등 원천과 동기화돼 있나. 원천에 있는데 미러에 없는 키 = drift (예: 마이그레이션 157 LIPS·@cosme 추가 시 폴백 상수 누락)
- **2-B 핫픽스 횡단 재발**: "보장"·"폴백"·"임시"·"진입 전 1회" 류 주석/가드 grep → 한 곳만 패치한 패턴이 다른 진입 경로에 남아 있나
- **2-C 캐시 의존 동기 함수**: 캐시 로드 전제 동기 함수(`getChannelLabel` 등)가 캐시 미로드 경로에서 호출되나
- 상세·근거: 사양서 `docs/specs/2026-06-02-service-health-audit.md` §4-1-2

### 차원 4 기준 데이터 내부 의미 중복 점검 (2026-06-23 추가)

차원 2-A는 "코드 미러 ↔ DB"를 보지만, **DB 안에서 같은 항목이 두 행으로 들어간 중복**은 못 잡는다(@cosme 사고의 빈틈). 차원 4에서 별도로 본다.

- **무엇**: `lookup_values` 등 기준 데이터에 표기·대소문자·전각/반각만 다른 의미 중복 행이 있나(예: `@Cosme`(code `channel-96r9y3`) vs `@cosme`(code `cosme`)). 식별자가 달라 유니크 제약에 안 걸리고 화면 선택지에 나란히 노출됨.
- **방법**: 운영·개발 양 DB에서 `kind` 별로 `SELECT code, name_ko, name_ja, sort_order, active FROM lookup_values WHERE kind='…' ORDER BY sort_order;` 조회 → 같은 표시명/유사 표기 묶음 육안 점검. 시드(`supabase/seed/lookup_values.sql`)와 후속 마이그레이션을 함께 grep해 "시드에 있는데 마이그레이션이 또 추가"한 패턴 확인.
- ⚠️ **코드만 봐선 안 잡힌다** — 운영 중 관리자가 화면에서 직접 추가한 행은 코드에 없으므로 **운영 DB 실데이터 조회 필수**.
- 예방 규칙: `.claude/rules/supabase.md` 「기준 데이터 추가 시 중복 확인」.

### 차원 9 UI·UX 정합성 점검 (2026-06-23 추가)

코드·DB가 정상이어도 **화면 실물**에서 깨지는 유형(중복 선택지·빈 화면·끊긴 진입). `reverb-qa-tester`(Playwright)로 실제 화면을 띄워 본다. 기준은 `.claude/rules/planning.md` 「UX 의심 체크리스트」를 재사용하되, 점검은 아래를 훑는다:

- **중복·모순 노출**: 같은 선택지가 두 번 뜨나(@cosme), 라벨이 화면마다 다른가(브랜드명 한·일 혼재 류), 권한별로 보여선 안 될 버튼이 보이나(campaign_manager가 못 누를 CUD 버튼).
- **빈 상태·로딩·에러**: 데이터 0건 화면이 텅 비지 않나(안내 문구), 깜빡임(`visibility:hidden` cloak), 에러 문구가 이해되는 일본어/한국어인가(`friendlyErrorJa`).
- **진입·이탈 경로**: 화면에 들어가는 길/나오는 길이 끊기지 않나(유일 진입점 제거 여부), 모바일 뒤로가기·해시 복원 동작.
- **인플 친화도·접근성**: 인플 안내가 초등학생 눈높이·번호 단계인가, 키보드 Tab·색 대비(WCAG AA)·`alt`·아이콘 `translate="no"`.
- 점검 대상 화면: 캠페인 등록/편집 폼, 응모 플로우, 관리자 목록 페인, 마이페이지 — 변경이 잦은 곳 우선.

### 차원 8 점검 방식 (자동 + 사람 혼합)

코드·마이그레이션은 PR로 추적되나 **대시보드 수동 설정은 repo에 반영 안 됨** — REVERB 최다 사고 영역(배포 큐·메일 한도·cron 미등록).

- **자동 확인** (명령이 직접): pg_cron job 가동 여부(`cron.job`), Edge Function 배포(`supabase functions list`), Vercel 배포 상태(`gh` deployments), DNS SPF/DKIM/DMARC(`dig`), 하드코딩 URL/key(grep), 마이그레이션 양 서버 적용 여부(SQL)
- **사람 체크리스트** (대시보드 확인 요청): Brevo 쿼터 잔량(20,000/월), Auth Confirm email 토글(운영 ON·개발 OFF), Rate limit(운영 100/h), Storage 버킷·정책, Vercel 도메인·환경변수
- 수시 트리거: cron·Edge Function·도메인·SMTP·DNS 변경 작업 시 동반

## 실행 절차

### Step 0: 사전 상태 파악
- `git status --short` — untracked/미커밋, 다른 세션 작업 영역 식별
- `git log --oneline --since=<지난 점검일>` — 점검 대상 기간 변경
- `ls supabase/migrations/ | tail -3` — 실제 최신 마이그레이션 번호
- **다른 세션이 작업 중인 영역(미커밋·진행 중 PR)은 "진행 중"으로 분류하고 오탐하지 않는다.**

### Step 1: 차원별 점검 실행
- 모드에 맞는 차원만. 가능한 차원은 **병렬**로(Explore 여러 개 동시).
- 각 차원은 사용 도구를 호출하되, **신규 에이전트 만들지 않고** 기존 5종 + Explore + 기존 슬래시 명령을 오케스트레이션.

### Step 2: ⚠️ 검증 후 등재 (필수 — 시범 점검 교훈)
- 서브에이전트(Explore 등) 발견을 **그대로 리포트에 넣지 않는다.** 핵심 발견은 메인 세션이 직접 grep/Read로 1차 검증.
- 2026-06-02 시범 점검에서 Explore가 "CLAUDE.md PR3 stale" 오탐 → 직접 확인으로 걸러낸 사례 있음.
- 검증 결과 오탐이면 리포트 "오탐 기록" 섹션에 남겨 서브에이전트 신뢰도를 추적.

### Step 3: 건강도 리포트 작성
- 파일: `docs/health/YYYY-MM-DD-service-health.md` (점검마다 누적)
- 필수 항목:
  - 모드 / 점검한 차원 / **점검 안 한 차원 명시** (silent skip 금지)
  - 차원별 발견 (없으면 "✅ 이상 없음") + 확실성(확실/추정) 표시
  - ⚠️ 오탐 기록 (있으면)
  - **발견 → 백로그 연결** (메모리 슬러그 또는 사양서 경로) — 조치 누락 방지
  - 이전 리포트 대비 반복 발견 추세

### Step 4: 발견 → 조치 고리
- 이 명령은 **진단 + 백로그 등록까지만.** 실제 수리는 개발 세션이 별도 진행.
- 발견 항목은 반드시 ① 메모리 등록 ② 사양서 승격 ③ 개발 세션 즉시 인계 중 하나로 연결.

## 원칙
- **과탐 금지**: 확실한 것 위주. 추정은 "추정"으로 명시.
- **silent skip 금지**: 점검 안 한 차원을 리포트에 명시.
- **비용 의식**: 매번 7차원 풀스윕은 과함. 차원별 주기 차등(배포전=회귀, 월1=죽은코드·문서, 분기1=전체).
- **운영 실데이터 권고**: 성능·비승인 분기 등 일부 차원은 개발서버 데이터가 적어 안 드러남 → 운영 확인 권고 명시.
