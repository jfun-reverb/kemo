# 배포진단 (Vercel/GitHub Deployment Diagnosis)

> Vercel 배포가 예상대로 안 됐을 때 진단 순서를 표준화한 플레이북.
> "왜 안 됐는지 체계적으로 파고들고 싶을 때" 사용. 추측 배포 금지.

## 트리거 상황 예시
- main에 push 했는데 운영 사이트에 반영 안 됨
- Deploy Hook 호출했는데 빌드 안 시작
- PR merge 했는데 Production Deployment 없음
- 운영과 개발 배포 상태가 기대와 다름

## 진단 원칙
1. **추측 배포 금지**: `git push`로 "되겠지"하지 말고 근본 원인부터 찾기
2. **우선순위는 좁히는 순서**: GitHub → Vercel → 쿼터 → 내부 큐
3. **결정적 증거 수집**: 사용자에게 스크린샷 요청 경로 명확히

---

## Step 0: 증상 정확히 분류

먼저 아래 질문에 답변 확보:
- 어느 브랜치 push가 막혔는가? (main = Production / dev = Preview)
- **Preview 배포는 되는가, Production 배포만 안 되는가?** (이게 Vercel 내부 stuck 감별 핵심)
- 언제부터 안 됐는가? (시간대 특정)
- 두 프로젝트(kemo, reverb-sales) 모두 영향? (경합 vs 개별 문제)

## Step 1: 현재 main HEAD와 Vercel Production SHA 비교

```bash
# main 최신 SHA
git log origin/main -1 --oneline

# Vercel Production이 실제로 배포한 SHA (curl로 HTML fetch)
curl -sIL https://www.globalreverb.com/admin/ | grep -i "age\|last-modified"
curl -sL https://www.globalreverb.com/admin/ | grep -c "<키워드>"
```

→ 다르면 Production 배포 막힘 확정

## Step 2: Vercel Deployments 전수 확인

Vercel 프로젝트 → **Deployments** 탭

- 상단 필터 **`Status 5/6`를 `6/6`으로 확장** (Canceled 포함 전수 확인)
- **All Branches / All Authors / All Environments** 필터 모두 해제
- 문제 commit SHA가 어디에도 없다 = Vercel이 **이벤트를 받지조차 못함**

## Step 3: 경합 이론 검증 (양 프로젝트 비교)

같은 GitHub repo에 연결된 다른 Vercel 프로젝트(예: reverb-sales)의 Deployments도 확인:
- **한쪽만 멈춤** → 해당 프로젝트 Git 연결 문제
- **양쪽 다 멈춤** → 공통 레이어(GitHub App 또는 Vercel 계정 차원) 문제
- **한쪽 Building인데 다른 쪽 Queued** → (Pro 전환 2026-04-22 이전) Hobby plan 동시 빌드 1개 한도 경합. **Pro 전환 후에는 동시 빌드 복수 가능**하므로 같은 증상 발생 시 **다른 원인**(브랜치 protection·CI 동시성·webhook 누락) 우선 의심

## Step 4: GitHub Integration 상태 확인

### 경로 A: GitHub App (현재 Vercel 기본 방식)
```
https://github.com/settings/installations
→ Vercel Configure
→ Repository access: 해당 repo 있는지
→ Permissions: Read+Write, repository hooks 포함 확인
```

### 경로 B: 레포 Webhooks
```
https://github.com/{org}/{repo}/settings/hooks
```
→ 비어있어도 정상 (GitHub App 방식은 여기에 안 나옴). OAuth 방식이면 여기에 등록

## Step 5: Vercel 쿼터 / 계정 상태

```
https://vercel.com/{team}/~/usage
```

- **Pro 플랜** (2026-04-22 업그레이드 — Team 기준):
  - **Fast Data Transfer**: 1 TB/월
  - **Edge Requests**: 10M/월
  - **Function Duration**: 1000 GB-Hrs/월
  - **일일 배포 한도**: 6,000개 (Hobby 100의 60배)
  - **동시 빌드**: 복수 병렬 (Hobby 1개 제한 해소)
- (참고) Hobby 이전 쿼터: Fast Data Transfer 100 GB/월, Edge Requests 1M/월, Function Duration 100 GB-Hrs/월, 일일 배포 100개

쿼터 초과면 대시보드 상단에 경고 배너 뜸. 사용률 1-10%면 문제 아님.

## Step 6: Vercel 프로젝트 설정 확인

### Settings → Git
- Connected Git Repository: `Connected` 상태 + 올바른 repo
- Production Branch: main (기본값, GitHub default branch)
- deployment_status / repository_dispatch / Commit Status Events: **모두 Enabled**

### Settings → General
- Pause Deployments 없음
- Last updated 시각 확인 (예상과 맞는지)

### Settings → Build and Deployment
- Framework Preset: Other (REVERB JP 기준)
- Root Directory: `./`
- Ignored Build Step: Automatic (수동 스크립트 걸려있지 않은지)

## Step 7: "Preview는 되는데 Production만 stuck" = 내부 큐 lock

**진단 결론**: Vercel 내부 Production deployment queue가 stuck.
사용자가 고칠 수 있는 3가지:

### 해결책 (권장 순서)

1. **Disconnect → Reconnect (가장 빠름)**
   - Vercel Settings → Git → Disconnect 확인
   - 30초 대기 (내부 큐 정리)
   - 같은 페이지 GitHub 버튼 → `{org}/{repo}` 재연결
   - **⚠️ Deploy Hooks는 삭제됨** (재연결 후 재생성 필요하면)

2. **새 empty commit으로 webhook 테스트**
   ```bash
   git checkout main && git pull --rebase
   git commit --allow-empty -m "chore: verify post-reconnect webhook"
   git push origin main
   ```
   - 이미 push된 stuck commit은 **소급 배포 안 됨** — 반드시 새 push 필요
   - empty commit이 main tip이 되면 이전 모든 미배포 변경사항이 함께 반영

3. **Vercel CLI로 강제 배포 (webhook 완전 우회)**
   ```bash
   npx vercel --prod
   ```

## 근본 원인별 대응표

| 진단 결과 | 대응 |
|----------|------|
| Vercel Deployments에 해당 SHA 전무 + Preview 정상 | Disconnect/Reconnect + empty commit |
| Deployments에 Failed/Canceled | 빌드 로그 확인 |
| Deployments에 Queued 장기간 | Hobby 동시 빌드 경합 — 다른 프로젝트 빌드 끝 대기 |
| GitHub App 권한 빠짐 | App 재설치 |
| Production Branch가 main이 아님 | Settings → Git에서 수정 |
| 쿼터 초과 | Pro plan 업그레이드 검토 |
