# 문서 추적 규칙 — 기획 초안 vs 구현 결과 (영구)

> 기획/설계 세션이 초안을 작성하고, 개발 세션이 실제 구현한 뒤 **무엇이 달라졌는지**를 문서에 남기는 규칙. 다음 기획 세션이 현황을 파악할 수 있도록 하기 위함.

---

## 문서별 역할 분리

| 문서 | 역할 | 작성 주체 |
|---|---|---|
| `docs/specs/YYYY-MM-DD-feature-name.md` | 기획 초안 (무엇을 만들 것인가) + **구현 결과** (어떻게 실제로 만들었나, 무엇이 달라졌나) | 기획 세션이 초안 → **개발 세션이 구현 결과 추가** |
| `docs/FEATURE_SPEC.md` | 현재 구현된 기능의 최신 상태 (레퍼런스) | **개발 세션이 기능 커밋과 동시에 업데이트** |
| `CLAUDE.md` | DB 스키마·아키텍처·URL 등 기술 레퍼런스 | **개발 세션이 변경 시 동시에 업데이트** |
| PR description | 변경 요약 + 요청 외 추가 변경 내역 | 개발 세션 |

---

## 개발 세션 의무 (커밋 전 필수)

### 1. 기능 구현 커밋 = 코드 + 문서 한 세트
- 신규 기능·DB 변경·동작 변경이 있는 커밋에 `FEATURE_SPEC.md` + `CLAUDE.md` 업데이트를 **동일 커밋**에 포함
- 나중에 몰아서 `docs:` 커밋으로 처리하는 것 금지 (마이그레이션 117·118 누락 사례 재발 방지)

### 2. 기획 초안 사양서가 있는 작업 → 구현 결과 섹션 추가 (의무)
`docs/specs/` 안에 해당 일감의 초안 파일이 있으면, **구현 완료 후** 하단에 아래 섹션을 추가:

```markdown
## 구현 결과

**구현일:** YYYY-MM-DD
**관련 커밋:** (커밋 해시 또는 PR 링크)

### 초안 대비 변경 사항
- 추가된 것: (기획 초안에 없었지만 구현 중 추가된 것)
- 빠진 것: (기획 초안에 있었지만 구현에서 제외된 것 + 이유)
- 달라진 것: (방향은 같지만 구현 방식·범위가 바뀐 것)

### 구현 중 기술 결정 사항
- (초안에 없던 마이그레이션·RLS·DB 구조 결정 등)
```

### 3. 기획 초안 없는 작업도 중요 변경은 기록
- 버그 수정이라도 DB 구조·동작 흐름이 달라지면 `FEATURE_SPEC.md` 또는 `CLAUDE.md`에 반영
- 단순 CSS·라벨 변경은 생략 가능

---

## 기획 세션 의무

- 초안 작성 시 `docs/specs/YYYY-MM-DD-feature-name.md` 파일 생성 (없으면 개발 세션이 구현 결과를 어디에 남길지 모름)
- 초안에 "구현 결과" 섹션은 비워두거나 `(개발 세션이 채울 것)` 플레이스홀더만 남기기
- 다음 기획 세션 시작 전 `docs/specs/` 내 최근 파일의 "구현 결과" 섹션 먼저 확인 (현황 파악)

---

## PR description 필수 형식

```markdown
## 변경 요약
- (구현한 것 1~3줄)

## 요청 외 추가 변경
- (없으면 "없음"으로 명시)

## 관련 사양서
- docs/specs/YYYY-MM-DD-feature-name.md (있는 경우)
```

---

**Why:** 기획 세션은 큰 줄기 초안만 작성하고 개발 세션에 넘기는 구조라, 개발 세션이 구현 결과를 기록하지 않으면 다음 기획 세션이 현황을 모른 채 설계를 시작하게 됨. 마이그레이션 117·118처럼 신규 DB 구조가 사양서 없이 쌓이면 누락·회귀 위험이 높아짐 (2026-05-13 지적).
