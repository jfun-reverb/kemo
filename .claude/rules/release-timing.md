# 시행일 운영 규칙 — 배포 타이밍 거버넌스 (영구)

> 약관·정책에 **시행일**(부칙 공고일 ≠ 시행일)이 걸린 기능을 출시할 때, "코드를 언제 배포하고, 효력은 언제 발생하며, 시행 전까지 코드를 어떻게 관리하나"를 일관되게 판단하기 위한 영구 규칙. 2026-06-02 사용자 질문에서 도출. 사양서: `docs/specs/2026-06-02-release-timing-governance.md`.

---

## 핵심 한 줄

**"문서는 미리 안 됨 / 코드는 미리 됨 / 집행은 시행일."**

---

## 1. 3층 모델 (가장 중요 — 충돌 방지)

시행일이 걸린 기능은 **세 가지를 분리**해서 판단한다. 하나로 묶으면 기존 규칙(`policy.md` "미구현 기능 선제 반영 금지")과 모순돼 보인다.

| 층 | 무엇 | 시점 | 근거 |
|---|---|---|---|
| **1. 약관 문서** | 약관·개인정보처리방침에 문구를 쓰는 것 | 미리 쓰지 않음 (실제 기능 배포에 맞춤) | `.claude/rules/policy.md` |
| **2. 코드 배포** | 코드가 서버에 올라가 동작 가능 상태 | **미리 가능** | 본 규칙 |
| **3. 효력(집행)** | 새 조항을 근거로 회원에게 권리·의무를 실제 행사 | **시행일부터** (+동의 절차 있으면 동의 후) | `docs/specs/2026-05-28-policy-versioning.md` |

→ "코드는 미리 가능"(2층)은 "약관 문서 미리 반영 금지"(1층)와 충돌하지 않는다. 층이 다르다.

## 2. 기능 종류 판정 (항상 1단계)

| 종류 | 정의 | 시행 전 동작 가부 |
|---|---|---|
| 유리한 변경 | 회원에게 이득·보호 (데이터 자동 삭제, 편의 기능 등) | 미리 가동 무방 (문서 문구 일치 위해 시행일 권장) |
| 불리한 변경 | 새 의무·제재·환수·불이익 | 코드는 미리, **집행은 시행일 이후 발생한 사유에만** (소급 금지) |
| 새 개인정보 수집 | 수집 항목·처리 목적 추가 | 시행 + 동의 완료 전 **운영 실회원 대상 수집 금지** |
| **판정 애매** | 유리/불리가 갈림 (예: 새 알림=편의지만 메일 빈도 증가=불편) | **불리로 간주** + `/약관확인` + 사용자 확인 (escape hatch) |

## 3. 자동 시행 ON — 시각 판정 위치

사람이 당일 손으로 켜지 않고 코드가 날짜를 보고 자동 전환할 수 있다. 단 **시각을 어디서 판정하느냐**가 신뢰도를 가른다.

| 대상 | 판정 위치 | 비고 |
|---|---|---|
| UI 노출·문구 전환 | 클라 상수 `new Date() >= EFFECTIVE` | 조작해도 피해 작음 → 허용 |
| **법적 집행(환수·제재·데이터 처리)** | **서버측** (원격 호출 함수·행 단위 보안 정책에서 `now() >= '...+09'`) | 클라 시각 신뢰 금지 (조작 시 시행 전 집행 뚫림) |
| 일괄 상태 전환 | pg_cron 예약 (마이그레이션 166 선례) | 서버 시각 → 신뢰 |

- **타임존 명시 의무**: 시각 상수·DB 비교에 항상 `+09:00` 오프셋. "6/9"가 어느 시간대인지 누락 금지.
- **자동 ON 동반 의무**: kill switch(비상 정지) + 시행 당일 관리자 통지. 무인 진행 중 직전 버그가 그대로 시행되는 것 방지.

## 4. REVERB 기본 전략 — 브랜치 우선

| 방식 | 언제 |
|---|---|
| **브랜치 분리 (1순위 기본값)** | 대부분. dev에 보류 → 시행일에 dev→main 머지. 운영 코드베이스에 잠든 코드를 안 남김. 메모리 `project_prod_hotfix_rebuild.md` 패턴 |
| 인코드 플래그 (예외) | 무인 자동 전환이 꼭 필요하거나, 시행일에 사람이 머지를 못 누르는 경우만 |

- 빌드가 단순 이어붙이기(concat)·전역 스코프 1개라, 인코드 플래그는 잠든 함수가 전역에 상존해 충돌·중복 정의 위험이 더 크다.
- 보류 기능이 2개 이상 같은 영역을 건드리면 플래그 중첩이 위험 → 반드시 브랜치 분리.

## 5. 잠든 코드(시행 대기 코드) 관리

OFF 상태라도 코드는 코드베이스에 살아있으므로, 다른 작업이 공유 지점을 건드리면 잠든 코드도 영향받는다.

- **개발서버는 ON 상태로 검증** — OFF만 테스트하면 회귀가 시행일에 폭발한다.
- **공유 지점 변경 시 OFF 경로 점검** — 공통 함수(`storage.js`)·DB 테이블/트리거·DOM id·i18n 키를 다른 작업이 건드리면 잠든 코드 호출부를 grep. (reverb-reviewer 의 stale 참조 점검 역할)
- **DB는 플래그로 못 가린다** — 마이그레이션 산출물(테이블·컬럼·트리거)은 OFF여도 운영 DB에 존재. DB 변경은 플래그와 무관하게 항상 영향 범위 점검.
- **시행 후 플래그 청소** — 플래그·OFF 경로는 임시. 시행 직후 분기를 제거해 drift 누적 방지.

## 6. 베타 운영 가부

| 베타 대상 | 시행 전 가능? |
|---|---|
| 단순 UI·유리한 기능 | 자유 |
| 새 개인정보 수집 / 불리한 의무 | ① 베타 참여자 별도 명시 동의(opt-in) / ② 테스트·사내 계정만 / ③ 개발서버만 — 셋 중 하나로 한정 |

운영 실회원 대상으로 동의 없이 새 조항 근거 동작을 시행 전 가동 = 위반 소지.

## 관련 규칙·문서

- `.claude/rules/policy.md` — 약관 문서 관점(미구현 선제 반영 금지·통지 기간)
- `docs/specs/2026-05-28-policy-versioning.md` — 효력 동기화·변경 통지·버전관리 (상위 그림)
- `.claude/rules/git.md` — dev→main 배포 워크플로
- `.claude/rules/multi-session.md` — 브랜치·worktree 운영
- 메모리 `project_prod_hotfix_rebuild.md` — 운영 핫픽스 시 소스 diff apply + 재빌드

**Why:** 시행일이 걸린 기능마다 "언제 배포·언제 효력·시행 전 코드 관리"를 매번 새로 판단하면 일관성이 깨지고, "코드는 미리 가능"이 "약관 미리 반영 금지"와 모순돼 보여 혼란이 생긴다. 3층 분리로 충돌을 없애고, 유리/불리 판정·시각 판정 위치·브랜치 우선 전략을 고정해 반복 판단을 제거한다 (2026-06-02 사용자 질문에서 도출).
