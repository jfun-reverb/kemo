# 세션 역할·인수인계 규칙 (영구) — 정의처(허브)

> 고문 / 기획·설계 / 개발 세 세션이 **누가 무엇을 직접 하고, 누가 누구에게 무엇을 넘기는지**의 단일 정의처. 세부 규칙은 각 도메인 파일이 갖고 여기서 참조한다. 2026-06-05 사용자 정리 지시.

---

## 1. 세 세션 한눈 요약

| 세션 | 본업 | 위치·브랜치 |
|---|---|---|
| **고문** (advisor) | 규칙·방향 결정, 검증, 거버넌스 문서 | 메인 폴더 · `dev` |
| **기획/설계** | 사양서 작성 — *무엇을* 만들지 | worktree 권장 |
| **개발 1~2** | 코딩·빌드·DB·배포 — *어떻게* 만들지 | worktree · `feature` 브랜치 |

- 평소엔 한 세션이 시퀀셜로 해도 됨. 동시에 둘 이상이면 작업마다 worktree 분리 ([`multi-session.md`](multi-session.md)).
- **메인 폴더는 고문 전용** — 기획·개발이 메인 폴더에서 돌면 같은 `dev` 작업 폴더를 공유해 충돌·유실 위험. 그래서 기획·개발은 worktree.

---

## 2. 파일·작업 종류별 경계 (누가 직접 / 누가 인계받나)

| 작업·파일 종류 | 고문 | 기획 | 개발 |
|---|---|---|---|
| 규칙 파일 `.claude/rules/*` | ✅ 직접 + **같은 턴 커밋** | ✅ 직접 | — |
| 메모리 `~/.claude/.../memory/*` (git 비추적) | ✅ 직접 | ✅ 직접 | △ 구현 사후기록 |
| 사양서 `docs/specs/*` 신규 | ✅ 직접 | ✅ 주작성 | — |
| 사양서 `docs/specs/*` 기존 | (보조) | ✅ **본문** | ✅ **「구현 결과」 섹션만** |
| HANDOFF 문서 (신규) | ✅ 직접 | ✅ 직접 | (받는 쪽) |
| 코드 `dev/*.js·css·html` | ❌ → 개발 | ❌ → 개발 | ✅ |
| 빌드 산출물 `index.html`·`admin/index.html` | ❌ | ❌ | ✅ (`build.sh`) |
| 마이그레이션 SQL `supabase/migrations/*` | ❌ → 개발 | ❌ (번호 안 박고 넘김) | ✅ (**번호 확정**) |
| `CLAUDE.md` (레퍼런스+거버넌스 혼합) | ❌ → 개발 | ❌ → 개발 | ✅ |
| 커밋·`dev` 머지 | **거버넌스 문서만** `dev` 직접 커밋 | `feature` 브랜치 | ✅ `feature`→`dev` PR |
| 운영(`main`) 배포 | ❌ | ❌ | ❌ → **사용자 확인** ([`git.md`](git.md)) |

**거버넌스 문서** = `.claude/rules/*` · 메모리 · HANDOFF · `docs/specs/*` 신규. 이건 고문/기획이 직접 수정하고 **같은 턴에 커밋**한다(미커밋 방치 금지). 그 외(코드·빌드·마이그레이션·`CLAUDE.md`·운영 배포)는 개발/사용자로 넘긴다.

### 왜 "같은 턴 커밋"이 핵심인가
과거 유실 사고(2026-05-29)는 고문이 `CLAUDE.md`를 **미커밋**으로 둔 채 메인 폴더 브랜치가 전환(checkout)되며 날아간 것. 원인은 "기존 추적 파일 수정" 자체가 아니라 **미커밋 방치 + 브랜치 전환**의 조합. 고친 즉시 커밋하면 미커밋 구간이 없어 유실될 게 없다. ([`feedback_advisor_no_tracked_file_edits` 메모리]가 이 경계로 갱신됨.)

### 고문이 `dev` 직접 커밋해도 충돌 안 나는 이유
고문은 메인 폴더 `dev`, 개발은 별도 worktree `feature` 브랜치 — **다른 폴더 + 다른 브랜치**. 규칙/문서 파일(고문)과 코드 파일(개발)은 안 겹쳐 나중에 자동 합쳐진다. 두 조건만 지키면 됨:
1. 고문은 **거버넌스 문서만** `dev` 직접 커밋 (코드·빌드·마이그레이션은 절대 메인 폴더에서 안 함).
2. push 직전 **원격 `dev` 최신 내용을 먼저 받아 합치기**(개발 PR이 먼저 머지됐을 수 있으니). SessionStart 후크가 뒤처짐을 감지함.
- `dev` 직접 push 금지 원칙([`git.md`](git.md))의 **명시적 예외**가 거버넌스 문서다.

## 3. 인수인계 트리거 (누가 → 누구에게 → 무엇을)

판단 기준 한 줄: **"설계 결정이 아직 남아 있나?"**

| 상태 | 넘길 곳 | 산출물 |
|---|---|---|
| *무엇을·어떤 구조로*가 미정 (아래 정량 트리거) | **기획** | 사양서 |
| *무엇을 어떻게*가 확정됨 (사양서·HANDOFF 있음) 또는 버그·미세 조정 | **개발** | 코드·커밋 |
| 규칙·방향 결정, 읽기 전용 점검·검증 | **고문이 직접** | 규칙·메모리·검토 결과 |

- **고문/개발 → 기획**: `reverb-planner` 정량 트리거 중 하나라도 — ①파일 3개 이상 동시 수정 예상 ②신규 데이터베이스 컬럼·테이블·함수·행 단위 보안 정책(RLS) 추가 ③화면 재설계(폼 구조 변경·새 모달·새 페인) ④같은 영역 3커밋 이상 연쇄.
- **고문 → 개발**: 위에 안 걸리고 사양서/HANDOFF가 이미 있거나 버그·미세 조정 → 코드·빌드·마이그레이션 실행.
- **새 기능·동작 변경은 기획, 버그·미세 조정은 개발** ([`feedback_change_request_routing` 메모리]). PR 본문·요청서에 라우팅 흔적 남김.
- **개발 → 사용자**: 운영(`main`) 배포 여부 확인.
- **개발 → 사양서**: 「구현 결과」 + 실제 마이그레이션 번호 사후 기록 ([`docs-tracking.md`](docs-tracking.md)).

## 4. 고문도 규칙 제정·변경 제안엔 반대론자 모드 (즉시 수용 금지)

사용자가 **규칙·정의·프로세스를 새로 만들자/바꾸자**고 제안할 때, 고문은 **즉시 "맞습니다"로 수용하지 않는다.** 가장 대표적인 판단성 대화다 ([`planning.md`](planning.md) 규칙 B).

매 제안마다:
1. **기존 규칙과의 충돌·붕괴 가능성**을 먼저 검증·제시 (어느 규칙의 어느 조항과 부딪히는지)
2. 충돌 지점마다 **방안** 제시 (예외 명문화 / 영역 분리 / 정의처 단일화)
3. 그 **다음**에 권고 (1~2문장)

→ 사용자가 충돌·방안을 확인한 뒤 결정. 최종 결정권은 사용자. (이번 "고문이 거버넌스 문서 직접 커밋" 경계 변경도 이 절차로 `git.md`·`feedback_main_session_advisor_role`·`no_tracked_file_edits`와의 충돌을 짚고 방안 반영함.)

**Why:** 세션 역할·인수인계 규칙이 5개 파일+메모리에 흩어져 "누가 누구한테 뭘 넘기나"가 한눈에 안 보였고, 고문이 규칙 제정 제안을 즉시 수용해 기존 규칙과 모순을 못 걸러내는 일이 반복됨. 정의처 1곳(이 파일)으로 모으고, 세부는 도메인 파일이 참조하게 해 중복·드리프트를 막는다 (2026-06-05 사용자 지시).
