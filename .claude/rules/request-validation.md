# 수정 요청 구조 적합성 검토 규칙 (영구)

> 사용자가 개발 세션에 수정을 요청했을 때, 개발 세션이 **무조건 실행하지 않고** 먼저 현재 구조와의 적합성을 검토하도록 하는 규칙. 2026-05-21 사용자 명시 지시.

## 세션별 역할 분담

| 세션 | 이 규칙에서의 역할 |
|---|---|
| **고문 세션** | 규칙·정의를 정해 개발/기획 세션의 방향을 잡아준다. 개발에 직접 관여하지 않고, 매 요청을 직접 검토하지도 않는다 (메타 레벨). |
| **개발 세션** | 사용자 수정 요청을 **무조건 실행 금지**. 구조 영향 있는 요청은 현재 구조와의 적합성을 먼저 검토하고, 안 맞거나 애매하면 코딩 전에 되묻는다. |
| **기획/설계 세션** | 개발 세션이 판단하기 애매한 구조 결정(설계 분기 큼)을 위임받아 판단한다. |

## 개발 세션 행동 규칙

### 적용 대상 — 구조 영향 있는 요청만
- 데이터베이스 구조 변경 (컬럼/테이블/원격 호출 함수(RPC)/행 단위 보안 정책(RLS))
- 화면 흐름 변경 (페이지 이동·라우팅·진입 경로)
- 기능 동작 변경 (기존 로직의 결과·조건이 달라짐)
- 여러 파일을 동시에 손대야 하는 변경

### 행동 순서
1. 요청을 받으면 **바로 코딩하지 않는다.** 먼저 현재 코드에서 그 부분이 어떻게 되어 있는지 파악한다.
2. 요청이 현재 구조와 맞지 않거나 부작용이 우려되면, 코딩 전에 사용자에게 되묻는다:
   - 형식 예시: 「현재 ◯◯는 ~이렇게 동작합니다. 요청하신 변경은 여기와 맞지 않아 보입니다(이유). 어떤 의도이신가요?」
   - 선택지가 생기면 `AskUserQuestion` 사용 (`.claude/rules/interaction.md` 규칙 적용)
3. 적합성 판단이 개발 세션 수준에서 애매하면(설계 분기가 크면) → `reverb-planner` 호출 또는 기획/설계 세션으로 라우팅.
4. 사용자가 의도를 설명하고 그대로 진행을 원하면 **따른다** (최종 결정권은 사용자). 단, 우려했던 점은 PR 본문·사양서에 기록.

### 적용 제외 — 바로 진행
- 단순 라벨/색상/오탈자/문구 수정
- 1개 파일 내 사소한 스타일 조정

## 관련 규칙
- `.claude/rules/interaction.md` 「변경 프로세스」 — 개선 요청 시 계획 먼저 → 영향 체크 → 애매한 부분 질문
- 추가 수정 요청 라우팅(새 기능→기획, 버그→개발): 메모리 `feedback_change_request_routing.md`
- 고문 세션 역할 경계: 메모리 `feedback_main_session_advisor_role.md`

**Why:** 사용자는 비개발자라 요청이 현재 구조와 충돌하는지 스스로 판단하기 어렵다. 기획 단계는 질문-답변으로 길을 찾지만, 개발 단계에서 요청을 무조건 실행하면 구조가 어긋난 채 쌓인다. 개발 세션이 「이 요청이 여기 맞나」를 먼저 검증하고 되묻는 게이트를 둬서, 사용자가 의도를 다시 점검할 기회를 만든다 (2026-05-21 사용자 명시 지시).
